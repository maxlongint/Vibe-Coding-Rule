# 更新需求变化

## 概览

这个技能用于处理「已有需求」的补充、范围变化、验收调整和追踪记录同步。核心规则是：需求正文更新为当前有效口径，需求变化记录保留历史变化事实，避免直接覆盖历史。

**工具无关：** 记录载体可以是 OpenSpec change（若已启用）、issue、PR、任务卡或外部需求系统。未启用 OpenSpec 时，按 AGENTS.md「需求治理与影响分析（自洽）」在等价载体中更新正文并追加变化记录。

## 触发边界

只在用户明确点名本技能，或明确要求「按需求变化同步技能处理」时使用。

不要在这些场景自动触发：

- 普通需求讨论。
- 规则问答。
- 临时想法或探索性设想。
- 尚未确认要更新已有需求记录的聊天内容。
- 新需求从零创建；这类任务应先按项目需求治理规则判断是否创建新的追踪记录（OpenSpec change **或** 新 issue/PR 等）。

## 处理原则

- 先确认这是已有需求的变化，而不是全新需求、缺陷修复或普通讨论。
- 需求正文只保留当前有效口径，让后续实现者不用在旧口径和新口径之间猜测。
- 需求文档正文默认使用简体中文；代码标识、API 名称、字段名、命令和既有英文模板可保留英文。
- 变化记录必须追加，不要覆盖或删除历史变化。
- 记录变化时保留时间、来源、原口径、新口径、影响范围、验收变化和剩余风险。
- 如果变化会影响行为、接口、数据、权限、流程或验收标准，先同步需求记录，再改实现。
- 如果无法确认变化归属或有效口径，先向用户确认，不要猜测改写。

## 标准流程

1. 确认变化对象。
   - 找到已有 OpenSpec change、issue、PR、外部需求记录、任务记录或用户明确指定的需求文档。
   - 确认该需求当前状态是否允许继续变更。
   - 若原需求已完成、废弃或被替代，先判断是恢复原需求还是创建新需求/新 issue。

2. 判断变化类型。
   - 补充说明：不改变目标，只补清约束、边界或解释。
   - 范围变化：新增、移除或替换能力范围。
   - 验收调整：改变通过条件、场景、反向断言或验证方式。
   - 方案调整：改变实现方案、数据流、权限规则、发布或回滚策略。
   - 记录同步：把用户确认、外部 issue 或评审结论同步回主记录。

3. 更新当前有效口径。
   - **OpenSpec 项目：** 更新 `proposal.md`、`design.md`、`tasks.md`、`spec.md`。
   - **非 OpenSpec 项目：** 更新 issue/PR 正文、外部系统描述或项目指定的规格文档。
   - 当前正文应表达最新目标、边界、验收标准和剩余风险。
   - 删除或改写过期口径时，确保变化记录已经保留原口径和变更原因。

4. 追加需求变化记录。
   - 在 OpenSpec change、issue 评论、PR 描述、任务记录、交付说明或项目指定位置**追加**记录。
   - 记录至少包含：

```text
时间：
来源：
变化类型：
原口径：
新口径：
影响范围：
验收变化：
实现影响：
验证调整：
剩余风险：
```

5. 同步任务和验收。
   - 更新任务状态、待办项和验证命令。
   - 增加或调整正向场景、反向场景和回归范围。
   - 如果变化影响模块、数据流、权限或状态流转，重新做影响分析（CodeGraph **若可用**则优先，否则 IDE/人工并记录风险）。

6. 再进入实现。
   - 只有在当前有效口径、变化记录、任务和验收标准同步后，才修改代码。
   - 如果用户只是要求同步记录，不要顺手实现代码。

## OpenSpec 同步规则（仅已启用 OpenSpec 时）

使用 OpenSpec 时，优先按文件职责更新：

| 文件 | 更新内容 |
| --- | --- |
| `proposal.md` | 目标、非目标、范围、用户影响、风险、当前状态 |
| `design.md` | 方案取舍、数据流、权限、状态流转、发布和回滚 |
| `tasks.md` | 任务拆分、状态、验证命令、剩余风险 |
| `spec.md` | 外部可观察行为、验收场景、正向和反向断言 |

`spec.md` 不要写成内部实现清单；应描述用户、系统或外部接口可观察到的行为。

## 非 OpenSpec 项目的等价动作

| OpenSpec 文件 | 等价落点（示例） |
| --- | --- |
| `proposal.md` | issue/PR 描述中的目标与非目标 |
| `design.md` | issue 评论、PR 设计说明或 `docs/` 设计段落 |
| `tasks.md` | issue 任务列表、PR checklist 或项目管理工具 |
| `spec.md` | 验收标准段落、测试用例说明或 API 契约文档 |
| 变化记录 | issue 时间线评论、PR 变更说明区块（追加，不覆盖历史） |

## 常见错误

| 错误 | 修正 |
| --- | --- |
| 直接覆盖需求正文，历史变化消失 | 当前正文更新为最新口径，同时追加变化记录 |
| 把临时想法当成需求变化 | 先确认来源和有效性 |
| 只改 proposal，不改 spec 和 tasks | 根据影响同步设计、任务和验收 |
| 验收标准变化后仍沿用旧测试 | 更新测试和验证命令 |
| 未启用 OpenSpec 却强行创建 `openspec/` | 在 issue/PR/外部系统中同步 |
| 需求正文继续使用英文或中英混杂导致口径不清 | 默认改为简体中文，必要的代码、接口、字段和命令英文除外 |
| 原需求已完成后继续追加任务 | 先判断恢复原记录还是新建 issue/change |
| 用户没有点名技能就自动套用 | 不自动触发，按 AGENTS.md 需求治理处理 |

## 完成检查

- 已确认用户明确点名本技能或明确要求使用需求变化同步技能。
- 已确认变化属于已有需求。
- 当前有效口径已更新（OpenSpec **或** issue/PR/外部记录）。
- 需求文档正文已使用简体中文，必要的代码标识、接口名、字段名和命令除外。
- 变化记录已追加，且包含时间、来源、原口径、新口径、影响范围、验收变化和剩余风险。
- 任务、设计和验收已按影响同步。
- 验证范围和剩余风险已说明。
