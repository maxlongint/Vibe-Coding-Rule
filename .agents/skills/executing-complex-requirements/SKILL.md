# 执行复杂需求

## 概览

这个技能用于把复杂业务需求从意图推进到有证据的交付，并避免丢失可追溯性。核心顺序是：先记录需求，再理解代码影响，再确认方案和计划，然后按测试驱动实现，最后用验证和评审收口。

## 判断入口

改文件前先判断任务类型：

| 信号 | 归类 | 必要动作 |
| --- | --- | --- |
| 改变行为、API 契约、数据结构、权限、状态流、用户路径或验收标准 | 真实需求 | 实现前创建或更新 OpenSpec，或使用已批准的外部记录 |
| 触及认证、授权、支付、审计、数据一致性、发布或回滚 | 重大需求 | 使用 OpenSpec；须完成影响分析（CodeGraph 优先）；计划、TDD、评审和完成前验证 |
| 修复实现缺陷，但不改变已发布业务规则 | 缺陷修复 | 使用系统化调试和 TDD；只有业务规则或验收标准变化时才创建 OpenSpec |
| 只改文案、局部呈现、内部测试或文档 | 轻量调整 | 在最终说明、任务记录或 PR 描述中记录范围和验证 |

如果项目没有 OpenSpec，而当前任务属于真实需求，应先询问用户是初始化 OpenSpec，还是使用外部或轻量记录。若任务紧急且范围小，只能先用轻量追踪记录推进，并说明剩余风险。

## 必需流程

1. 读取项目规则。
   - 读取 `AGENTS.md`。
   - 存在 `docs/规范/` 和 `.agents/skills/` 时，先扫描入口和相关文件名。
   - 确认提交、发布、部署、编码和验证规则。

2. 澄清需求。
   - **REQUIRED SUB-SKILL:** 使用 `superpowers:brainstorming` 澄清目标、非目标、约束、验收标准、风险和发布边界。
   - 只询问会阻塞安全实现的决策。
   - 使用默认假设继续时，必须显式记录假设。

3. 创建或更新追踪记录。
   - 中等和重大需求优先使用 `openspec/changes/<change-name>/`。
   - 使用 OpenSpec 时，通常包含 `proposal.md`、`design.md`、`tasks.md` 和 `specs/<domain>/spec.md`。
   - 使用外部系统时，在任务记录、PR 描述、提交正文或交付说明中保留链接、编号或用户确认来源。
   - 需求文档正文默认使用简体中文编写，包括 proposal、design、tasks、spec、变化记录和外部需求记录中的目标、边界、验收、风险和结论。代码标识、API 名称、字段名、命令和既有英文模板可保留英文。

4. 分析代码影响（涉及代码改动时必须完成）。
   - CodeGraph 可用时优先使用：
     - `codegraph_explore` 理解功能区域。
     - `codegraph_callers` 和 `codegraph_callees` 分析关键函数。
     - 共享模块改动或重构前使用 `codegraph_impact`。
   - CodeGraph 不可用时，使用 `rg`、目录检查和测试索引，并记录限制和剩余风险。
   - 将影响结论写入 `design.md`、`tasks.md` 或交付说明。

5. 确认方案并更新追踪记录。
   - 对中等和重大需求，先比较可选方案，再确认推荐方案。
   - 记录采用和不采用方案的原因、影响范围、安全点、发布方式、回滚方式和剩余风险。
   - 在进入实现计划前，同步更新 `proposal.md`、`design.md`、`tasks.md` 和相关 `spec.md`。
   - 不要把关键方案取舍只留在聊天记录里。

6. 编写实现计划。
   - **REQUIRED SUB-SKILL:** 使用 `superpowers:writing-plans` 拆分多步骤实现。
   - 每个任务都应包含目标文件、测试、验证命令、完成标准、发布或回滚注意点。
   - 不要把高风险行为变化藏进宽泛重构里。

7. 先测试后实现。
   - **REQUIRED SUB-SKILL:** 使用 `superpowers:test-driven-development` 处理功能、缺陷修复和行为变化。
   - 先让目标测试因预期原因失败，再修改生产代码。
   - 复用现有架构和真实业务链路；不要在生产路径用 mock、旧字段、本地缓存、默认对象或猜测值伪造成功。

8. 需求变化先改记录，再改代码。
   - 先判断变化是小变化、中等变化还是重大变化。
   - 实现变化前，先更新 OpenSpec、外部记录、任务和测试。
   - 当模块、数据流、权限模型或验收标准变化时，重新做影响分析。

9. 完成前验证。
   - **REQUIRED SUB-SKILL:** 使用 `superpowers:verification-before-completion`。
   - 从项目文档、`package.json` scripts 或规范中选择验证命令。
   - Node 项目必须先检查 `.node-version`；存在时按项目规则通过 `fnm` 使用对应版本。
   - 汇报实际运行的命令、结果、跳过项、跳过原因和剩余风险。

10. 评审和发布安全收口。
    - **REQUIRED SUB-SKILL:** 重大需求或安全敏感变更使用 `superpowers:requesting-code-review`。
    - 评审材料应覆盖追踪记录、CodeGraph 影响、修改文件、测试、安全点、发布、回滚和开放风险。
    - 未经用户明确要求，不要执行 commit、push、merge、deploy、release。

11. 完成或归档需求。
    - 在 OpenSpec 或外部记录中标记状态：已完成、已暂停、已废弃或已替代。
    - 记录日期、最终验收结果、未解决风险和后续方向。
    - 完成后出现新范围时，先判断是恢复同一 change，还是创建新 change。

## OpenSpec 最小记录

中等或重大需求至少要让这些信息可追溯：

```text
change path:
source:
status:
goals:
non-goals:
acceptance criteria:
impact analysis:
solution decision:
tasks and verification:
rollout:
rollback:
residual risks:
```

`spec.md` 应描述外部可观察行为和验收场景，不应只描述内部文件或实现细节。

## 通用首响示例

遇到复杂需求时，第一轮回复应先确认治理和影响分析，不要直接开始写实现：

```text
这是一个会影响行为、契约或验收标准的复杂需求。我会先确认需求追踪记录和验收标准，再用 CodeGraph 分析受影响的业务模块、关键调用链、数据流、权限或状态流转。实现代码会等 OpenSpec 或已批准的外部记录存在，并完成方案确认后再开始。
```

如果用户给的是认证、支付、审批、数据迁移、权限模型等具体领域，把上面示例中的“业务模块、关键调用链、数据流、权限或状态流转”替换成该领域真实受影响的模块，不要套用某个固定案例。

## 常见错误

| 错误 | 修正 |
| --- | --- |
| 真实需求还没有记录就开始写代码 | 先创建或更新 OpenSpec，或使用已批准的外部记录 |
| 把某个案例的模块清单套到所有需求 | 按当前项目和当前需求重新做影响分析 |
| 跨模块工作把 CodeGraph 当成可选项 | 使用 CodeGraph，或记录人工分析方式和限制 |
| 影响分析后直接写计划，跳过方案确认 | 先比较方案、记录取舍，并同步设计、任务和规格 |
| 只写内部文件和函数，不写外部可观察行为 | 用场景和验收标准描述规格 |
| 需求文档正文使用英文导致团队理解成本升高 | 默认使用简体中文，只保留必要的代码、接口、字段和命令英文 |
| 用 mock 或缓存数据宣称生产链路成功 | mock 只用于测试或演示，生产链路必须单独验证 |
| 需求变化直接被代码吸收 | 先更新记录、任务、测试和计划，再实现变化 |
| 没有新鲜证据就说验证通过 | 实际运行命令，或说明未运行原因和风险 |
| 默认提交或发布 | 等用户明确授权 |

## 完成检查清单

- 已识别需求类型和治理模式。
- 已检查 `AGENTS.md` 和相关项目规则。
- 需要时已更新 OpenSpec、外部记录或轻量追踪记录。
- 需求文档正文已使用简体中文，必要的代码标识、接口名、字段名和命令除外。
- 已完成 CodeGraph 影响分析，或记录人工分析限制。
- 已比较方案，并把取舍同步到追踪记录。
- 实现计划包含代码、测试、验证、发布、回滚和剩余风险。
- 行为变化使用 TDD。
- 需求变化先更新记录，再进入实现。
- 已运行并汇报验证命令。
- 重大或敏感变更已准备评审材料。
- 未经授权没有 commit、push、merge、release 或 deploy。
