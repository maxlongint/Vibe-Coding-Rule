# 规则干条

> **范围**：仅本规范仓维护者速查，**不**复制到业务项目，**不**参与接入/升级。细则与例外以 `AGENTS.md`、`docs/` 与工作流原文为准；冲突时按「用户已确认需求 → 唯一需求载体 → 项目规范 → 工具/skill 默认」处理。

## 一、工具与规范载体

1. 默认假定 **OpenSpec、CodeGraph、Superpowers** 均已安装并可调用；**需要使用**某项却发现未安装、未初始化或当前会话不可用时，立即中断并明确告知是哪一个工具，不得伪装可用、静默降级或擅自安装。不依赖该工具的任务不受此阻塞。
2. **`AGENTS.md` 未经用户明确要求不得修改**；只保留 AI 每次 Chat 都需要知道、在读取索引前必须生效，以及用于判断需求授权和按需路由的规则。
3. 项目长期知识与条件化流程放 `docs/`，以 `docs/README.md` 为索引并按需读取；不得把 AI 每次 Chat 都需要知道的规则只写在 `docs/`。新增、移动、改名或删除规范时必须同步维护索引。
4. OpenSpec 只由用户明确确认的新增需求或需求变更触发。新增需求须在口径确认后由用户主动授权创建，并由 AI 生成 ID、检查冲突；需求变更或已有需求须确认继续原 change / 新建关联 change，新建分支同样由创建请求授权生成 ID 和检查冲突。完成对应授权后，同一需求后续工作只用该 **OpenSpec change** 作为唯一载体；issue/PR/任务卡等只作输入来源，不得替代载体或维护平行口径。
5. 项目级 skill 放 `.agents/skills/`，先维护 `.agents/README.md` 索引；个人全局 skill 不属于项目规范。
6. **用户提供内容不得擅自落盘**：用户提供的经验、素材、偏好或讨论结论，除非用户明确要求保存并指定或确认归属，否则不得写入 `docs/`、`AGENTS.md`、`.agents/`、OpenSpec 或其他仓库文件；归属不明确时必须先询问。
7. 需要项目特有知识或 `AGENTS.md` 明确触发时：先读 `docs/README.md`，再只加载直接相关文档；不得在每个 Chat 中默认加载 `docs/`，不得猜测缺失内容或为扩上下文批量读取。
8. Superpowers 只作方法，不提供创建、选定或更新需求载体的用户授权。用户确认进入 OpenSpec 流程后，change 优先于 skill 默认落盘与提交流程；不得另建 `docs/superpowers/` 等平行文档，不得因 skill 默认自动提交；用户要求跳过某类方法 skill 时从其要求。

## 二、六条底线（始终生效）

9. **不伪造成功**：真实失败必须显式暴露，不用默认值、空对象、mock、旧链路或跳过检查掩盖。隔离 mock 仅可用于已标明边界的测试、演示或约定的 UI 阶段 1，不得冒充真实业务依赖或掩盖失败。
10. **完成前必须验证**：必须说明实际结果；未完成必要验证时不得声称完成或通过，并必须写明原因、未覆盖项和剩余风险。
11. **改行为前先记录**：用户确认新增或变更需求后，改变外部行为、接口、数据、权限、流程或验收前，必须先更新用户确认的唯一需求载体；未更新前不得实施。
12. **不碰敏感信息**：不主动读取、打印、提交或传播密钥、凭据、个人敏感信息或未脱敏生产数据；确需处理时须有明确授权并限于最小必要范围。
13. **不擅自制造外部副作用**：未经用户明确要求，不 commit / push / merge / release / deploy / 归档需求记录，不删除、迁移、写入生产数据或执行不可逆、付费、对外操作。
14. **拿不准就问**：AI 判断内容可能构成新增需求或需求变更时，必须先说明依据和影响，请求用户确认是否作为正式需求处理；需求意图确认前不得查找 OpenSpec 候选或进入 OpenSpec 流程。

## 三、需求与跨 Chat

15. 用户及已确认记录决定需求事实；源码、测试、版本历史、行为与代码分析只证明代码事实，不能单独定义需求或验收。
16. 用户确认需求意图后才能继续对应流程。新增需求先澄清并确认目标、范围和验收口径；用户主动要求创建时，AI 生成 ID 并检查现有 change，发现同名或可能承载该需求的 change 就停止，否则直接创建，无需单独确认 ID。需求变更或已有需求即使只有一个候选，也必须展示状态、关系与建议，由用户确认继续原 change / 新建关联 change；新建分支确认新口径并收到用户主动创建请求后，AI 生成 ID、检查冲突，有冲突时停止，无冲突则创建，无需单独确认 ID。
17. 跨 Chat 或上下文压缩后，不依赖对话记忆自动继续；用户明确 `change-id` 后才重新读取对应 OpenSpec change（proposal / specs / design / tasks；变更还要读变化记录）。未明确时只读查找并展示候选，取得确认后才能恢复。
18. 用户未确认需求意图，新增需求或新建关联 change 未取得创建授权并通过冲突检查，或继续现有 change 时未确认载体前，不创建或更新 change、不实现、不形成最终设计与计划；已确认结论须先写入当前唯一载体后再切换 Chat。
19. 需求变更：先定位原记录 → 判断变化等级 → 正文改为当前口径 → **追加**变化记录（不覆盖历史）→ 再改代码；已归档 change 原则上新开 change 承接。

## 四、实施边界

20. 修改前区分「目标变更」与「非目标不变式」；未要求改变的行为、接口、数据、权限、错误处理、性能、布局、调用方式与兼容性一律视为不变式。
21. 只改实现目标所必需的内容；不得靠放宽校验、吞错、默认成功、绕过权限、扩大默认权限、削弱类型或破坏兼容性制造局部效果。
22. 必须改非目标不变式时，先说明原因、影响与验证方式，等用户确认后再实施。
23. 保护用户已有工作：不得擅自覆盖、回退、删除或格式化已有改动；无法安全合并时停止并说明。
24. 用最少代码解决问题；不为一次性或假想需求加抽象与配置；不顺手重构或清理无关死代码。
25. 引入依赖、服务、技术栈或绕过既有依赖前，说明原因、影响和验证成本。
26. 源码、配置和文档默认 UTF-8 无 BOM；中文项目新增内容默认简体中文，写入后确认无乱码。

## 五、正确性与安全

27. 生产与真实业务链路不得用 mock、本地缓存、seed、旧字段或旧链路冒充真实依赖；测试/演示/本地模拟必须隔离并标明边界。
28. 只有业务上确实无数据时才能返回空；不可读、不可解析、契约缺失、接口失败或权限失败都必须当错误处理。
29. 面向用户的错误信息不得泄露内部敏感细节；疑似泄露时停止扩散，只说明风险和位置，不复制敏感内容。
30. 不把 `.env`、凭据、连接串或令牌写入源码、文档、示例、测试快照、日志或提交说明；不伪造权限、凭据或安全校验结果。
31. 可能重试、跨系统或影响真实业务的关键写操作须幂等（或等效防重），确认写入结果后再继续依赖该结果的流程。

## 六、验证与交付

32. 验证覆盖本次改动及直接影响；结构/交互/校验/服务/权限/数据契约/外部集成须扩到直接调用方与关键反向路径。
33. 不得通过跳过、删除或弱化测试，缩小必要验证范围，或只改断言/期望值来制造通过结果。
34. 完成前重新运行能证明结论的验证并读完整结果；不得把部分验证外推为整体通过。
35. 完成后说明：实际改变、已核对的直接相关不变式、验证证据、未验证项和剩余风险。
36. 用户要求提交/发布前，先汇总变更、关联需求、验证结果和剩余风险；验证不完整或高风险必须明确披露。
37. 提交信息无约定时默认简体中文，标题描述实际结果，避免空泛措辞，并按需注明需求记录。

## 七、代码设计（取舍）

38. 先保证行为正确、错误显式、契约清楚，再追求简单直接。
39. **KISS**：能局部、用现有模式解决的，不新建平行架构或多余抽象。
40. **YAGNI**：不为假想需求提前做扩展点、配置层、通用框架。
41. **DRY**：合并的是同一知识/业务规则，不是「长得像」；语义不同可保留独立实现。
42. 关注点分离、高内聚低耦合；命名与默认行为符合最小惊讶，默认值不得掩盖失败。

## 八、工具协作顺序

43. AI 识别潜在新增/变更需求 → 说明依据与影响 → 用户确认需求意图 → 新增需求澄清口径后由用户主动授权创建、AI 生成 ID 并检查冲突；需求变更或已有需求只读展示候选并由用户确认继续原 change / 新建关联 change，新建分支同样由创建请求授权生成 ID 和检查冲突 → 再执行写入及后续工具流程（OpenSpec 承载需求；CodeGraph 按需查代码事实；Superpowers 提供方法）→ 工具输出与源码/测试/命令结果核对 → 回填该 change。细则见 `docs/规范/工具协作.md`「Superpowers 与载体优先级」。
44. CodeGraph 不定义需求、不要求每步点名；OpenSpec 不根据源码猜需求；工具已安装不等于已获提交/发布/写生产授权。
45. 斜杠命令等只按当前 AI 工具实际提供的形式使用；需要用户重启才能加载新能力时说明后停止，不得假装已加载。

## 九、UI 两阶段（仅适用时）

46. 仅在明确采用规定标题「阶段 1：UI、隔离 mock 与页面跳转；阶段 2：真实接口、数据、权限、状态流转与错误处理」或已有对应重组标记时，启用 `docs/规范/工具协作.md` §5。
47. 阶段 1 禁止接入真实接口、真实数据、权限判断或业务状态流转；须人工 UI 验收通过后再做阶段 2。
48. UI 权威来源必须可读取、非占位符；不得擅自把仓库目录、旧稿、缓存或历史附件加入权威集合。
49. 首次落盘任务只依据已确认文字需求；再读 UI；读 UI 前须固化基线/快照，不得覆盖已有有效基线。
50. 重组后必须通过机械核对（checkbox、编号、映射、原子约束、职责）方可声明无遗漏或开始实现；任一检查失败则停止。
