# 需求变更工作流

> **范围**：仅本规范仓工作流教程，**不**随 Vibe Coding Rule 接入复制到业务项目。使用时在本仓库查阅或打开对应页面。默认使用 OpenSpec 主流程。强制底线见 [`AGENTS.md`](./AGENTS.md)，工具默认假定与协作规则见 [`docs/规范/工具协作.md`](./docs/规范/工具协作.md)。各步骤可以在同一个或不同的 AI Chat 中完成；操作已有 change 时，须在提示词中提供明确的 change-id。从零新增需求见 [`新增需求工作流.md`](./新增需求工作流.md)。
>
> **核心原则：不急着新建——先定位原记录、判断变化等级、把正文更新为当前口径、追加变化记录（不覆盖历史），再改代码。**

本流程只在用户明确确认要把内容作为正式需求变更处理后使用。AI 识别到潜在需求变更时必须先说明依据和影响并请求需求意图确认；确认前不得查找 OpenSpec 候选或自动进入本流程。用户确认需求意图后，才能执行下方只读原记录与候选扫描；具体载体确认前不得创建、选定或更新 change，也不得形成最终设计、计划或开始实现。

开始前按默认假定检查 OpenSpec 是否可用（本流程以 OpenSpec change 为唯一载体，故先检 OpenSpec）。若发现未安装、未初始化或无法调用，立即中断并明确告知是 OpenSpec，说明已完成项与阻塞点；不自动安装，也不改用 issue/PR/任务卡或其他轻量载体继续。后续步骤若需要 CodeGraph 或 Superpowers 却不可用，同样中断并指明该工具；不因「流程开头未预检」而视为可缺省。

先只读扫描相关的进行中及历史 change，核对原始需求来源、当前口径、任务状态和验收，并始终列出全部候选的 `change-id`、状态、关系和建议。即使只有一个候选，也必须等待用户明确确认是更新该 change 还是新建关联 change；没有候选时也必须说明并等待用户决定。确认具体载体前不创建、选定或更新 change，也不修改代码。

需求变更走六个步骤：

```text
定位原记录 → 澄清并确认新口径 →〔需要时确认 ID 并新建关联 change〕→ 更新 change 并追加变化记录 → 实现 → 验证 → 归档
```

先由 AI 根据变更描述只读定位原记录和候选 `change-id`，由用户明确确认继续原 change 还是新建关联 change 的方向。若用户确认需新建 change 承接变更，必须先完成第 2 步并取得用户对目标、范围和验收口径的明确确认；此后 AI 才能根据已确认口径提出简短、明确的小写英文 ID，使用短横线分隔，检查是否与现有 change 重名，并说明与原 change 的关系。本步只提议，不创建；用户明确确认该准确 ID 后，AI 才能创建 change。执行后续步骤时，将已确认的 ID 填入提示词中的 `<change-id>` 占位符，确保切换 AI Chat 后仍能定位同一个 change。

**变更涉及 UI 设计图时**采用条件式两阶段实现：主流程不变，在第 3 步更新 `design` 与 `tasks.md` 时按两阶段拆分，第 4 步分两轮执行（先 UI + mock，**人工验收 UI** 后再接真实逻辑）。

## 1. 定位原记录

来了一个变更，先找到它对应的原 change 或外部需求记录，**别上来就新建**。确认这真的是「已有需求的变化」，而不是全新需求、缺陷修复或随口讨论。

在 AI Chat 中发送：

```text
我收到一项变更：<变更描述>。

请先了解与这项变更直接相关的项目现状，定位对应的原 OpenSpec change 或外部需求记录，并核对原始需求来源。

请说明：原记录位置、change-id、当前有效口径、change 当前状态（进行中 / 已完成未归档 / 已归档），以及应继续用原 change 还是新建 change 承接。如果原始来源不可读、OpenSpec 未初始化或无法可靠定位，请明确说明原因，不要猜测归属。先不要创建或更新 change，也不要改代码。
```

判断口诀：

-   **继续用原 change**——目标没变、还是同一项能力，只是细化或调整。例如「密码最短长度从 8 改到 12」，仍是同一条密码策略。
-   **新建 change**——目标变成了另一件事、影响完全不同的模块、或需要独立发布和回滚。例如原本只做「密码策略」，现在要追加「登录失败锁定账号」，这是另一项能力。已归档的 change 原则上新开 change 承接，并注明与旧 change 的关系。

再判断变化等级：

| 等级         | 典型例子                                     | 处理力度                     |
| ------------ | -------------------------------------------- | ---------------------------- |
| **小变化**   | 文案微调、不影响流程的展示调整               | 可简化流程，第 2 步可跳过    |
| **中等变化** | 校验规则调整、新增一种 OAuth 绑定方式        | 走澄清 + 影响分析            |
| **重大变化** | 认证协议更换、权限模型调整、默认登录策略变化 | 完整走澄清 + 影响分析 + 回归 |

注意：看着像「小变化」，但只要它影响了操作路径、权限可见性、状态流转、验收标准、用户对结果的预期、数据提交意图或合规含义，就按中等/重大处理。

本步只确认原记录、变化等级以及继续原 change / 新建关联 change 的方向，不提出或创建新 `change-id`。即使用户已确认需要新建关联 change，也必须先完成第 2 步，明确并确认目标、范围和验收口径。

## 2. 澄清新口径

中等/重大变化必做，小变化可简化。若需要新建关联 change，无论变化等级如何，都不得跳过目标、范围和验收口径的明确确认。让 AI 把「原来是怎样、现在要变成怎样、差异在哪、影响哪些范围、验收怎么跟着变」理清楚。这一步只澄清，不更新 change，也不实现。

在 AI Chat 中发送：

```text
请基于原需求记录澄清本次变更的新口径。

原 Change ID：<用户已确认的原 change-id>

请先读取并对照原始需求来源和原 change 的当前内容，再确认：原口径 → 新口径的差异、目标、范围、非目标、验收标准的变化和待决问题。Superpowers 只作澄清等方法；本步不要按其默认路径另建计划/设计文档，不要创建、选定或更新 change，也不要实现。等待我明确确认目标、范围和验收口径后，再决定继续原 change 还是按已确认方向新建关联 change。
```

与 AI 对话，直到你明确确认目标、范围和验收口径。

若用户确认继续使用原 change，可直接进入下一步。若用户已确认需要**新建关联 change**，此时才发送下面的提示词确认具体 ID；本步不得创建 change：

```text
本次需求变更的目标、范围和验收口径均已确认，并确认需要新建关联 change。请只提出拟使用的 OpenSpec change-id。

原 Change ID：<用户已确认的原 change-id>
变更说明：<本次变更已确认的目标、范围和验收口径>

请只读检查现有 change，提出一个简短、明确、使用小写英文和短横线分隔且不与现有 change 重名的 change-id，说明命名依据以及与原 change 的关系。先不要创建、选定或更新任何 change，等待我明确确认具体 ID。
```

用户明确确认新 `change-id` 后，再发送：

```text
请根据本次变更及已定位的原记录，新建一个 OpenSpec change，并写明它与原 change 的关系。

原 Change ID：<用户已确认的原 change-id>
新 Change ID：<用户已明确确认的新 change-id>

如果本次变更来源或本消息同时包含新的 UI 设计图，本步只根据已确认的文字新口径创建初始任务，不读取或解释新设计图，也不加入设计衍生任务；请列出暂未读取的 UI 输入。若文字新口径与设计图无法可靠分离，请停止并说明。先形成原始 tasks，设计图在第 3 步的 UI 补充提示词中处理。

请先读取并对照本次变更与原记录，确认上面的新 change-id 与用户确认结果完全一致且不与现有 change 重名；不一致、仍是占位符或发生重名时停止并说明，不得改用其他 ID。确认无误后创建 change，起草 proposal、specs、design 和 tasks，使新口径、影响范围、验收标准、方案、任务以及与原 change 的关系保持一致。

Superpowers 只作方法；确认后的口径、设计与计划只写入本 change，禁止写入 `docs/superpowers/` 或其他平行文档，禁止自动提交。如果原记录不可读、OpenSpec 未初始化或创建失败，请明确说明原因，不要伪装完成。成功后返回最终使用的 change-id，并说明创建了哪些内容；先不要实现。
```

确认归属、状态、继续/新建决定以及具体 `change-id` 均与用户确认结果一致后，再进入下一步。

## 3. 更新 change 并追加变化记录

让 AI 把受影响的文档改成**当前有效口径**，并**追加**一条变化记录——保留原口径，**绝不覆盖历史**。按变化类型决定改哪些文件：

| 变化类型 | 要改的文件                           |
| -------- | ------------------------------------ |
| 补充说明 | `proposal`                           |
| 范围变化 | `proposal` + `specs`                 |
| 验收调整 | `specs` + `tasks`                    |
| 方案调整 | `design` + `tasks`（可能含 `specs`） |

在 AI Chat 中发送：

```text
请将本次变更所归属的 OpenSpec change 正文更新为当前口径，并追加一条变化记录（不覆盖历史）。

Change ID：<change-id>

如果本次变更同时包含新的 UI 设计图，本步只根据已确认的文字新口径更新正文、验收与当前 tasks，不读取或解释新设计图，也不加入设计衍生任务；请列出暂未读取的 UI 输入。若文字新口径与设计图无法可靠分离，请停止并说明。更新后的当前 tasks 是本轮读取 UI 前的任务状态；后续按 `docs/规范/工具协作.md` 固化基线或快照，不得覆盖已有基线。

中等/重大变化请重新分析直接影响的模块、调用方、数据流和测试面，并把结论、证据与未覆盖范围写进 `design` / `tasks`。

Superpowers 只作方法；确认后的口径、设计、计划与影响分析只写入本 change，禁止写入 `docs/superpowers/` 或其他平行文档，禁止自动提交。

变化记录须包含：时间、来源、变化类型、原口径、新口径、影响范围、验收变化、实现影响、验证调整、剩余风险。必须写入本 change（change 目录的变化记录区或 `proposal.md` 末尾「变化记录」章节）；issue/PR 等可作来源引用或同步摘要，不得单独充当唯一口径。

同步 `tasks.md` 与验收标准——不能拿旧测试覆盖新验收。先不要实现。
```

**变更涉及 UI 设计图时**，再发送一条补充提示词：

```text
本次变更涉及 UI 设计图，请补充并更新本次变更所归属的 OpenSpec change 的 design 与 tasks.md。

Change ID：<change-id>
UI 设计图来源：<实际本地路径 / Figma·蓝湖等链接 / 附件说明及作用 / 沿用 design.md 已记录的来源集合>

上面的字段必须替换为实际且可读取的来源，不能保留占位符。先不要读取 UI 来源；先读取该 change 的 proposal、specs、design、当前 tasks 和变化记录，并按项目 `docs/规范/工具协作.md` 完成读取前的基线、快照与旧结构修复门禁。门禁未通过时停止。

门禁通过后，读取 UI 来源并严格按该规范更新 design 与 tasks.md，形成两个阶段、任务映射、未决问题和核对记录。完成后返回核对结果，先不要实现。

```

检查正文、变化记录、任务拆分与验收标准是否准确。有 UI 设计图时，确认 AI 已读取你提供的设计图位置，且 `tasks.md` 已按两阶段拆分。确认无误后，再进入实现。

## 4. 实现

正文、变化记录与任务全部同步后，再让 AI 按 change 中的任务清单完成开发，并同步任务状态。实现途中又冒出新变化，回到第 2 步重新判断，**别直接把变化吸进代码**。

**无 UI 设计图**，在 AI Chat 中发送：

```text
请实现本次变更所归属的 OpenSpec change。

Change ID：<change-id>

请先读取并对照该 change 的 proposal、specs、design、tasks 和变化记录，再按 tasks 逐项实现，不要改变未要求调整的行为、接口、数据、权限和流程。

完成后更新任务状态，并说明实际改动、已经执行的检查、尚未验证的内容和剩余风险。不要归档。
```

**有 UI 设计图**——分两轮执行；阶段 1 由 AI 实现，**UI 由你人工验收**，通过后再发阶段 2。

第一轮，在 AI Chat 中发送：

```text
请只实现本次变更所归属的 OpenSpec change 中「阶段 1：UI、隔离 mock 与页面跳转」的任务，不要开始阶段 2。

Change ID：<change-id>

请先根据 change-id 重新读取并逐项对照该 change 的 proposal、specs、design、tasks、变化记录，以及 tasks 中的“任务重组基线”和“任务重组映射”；以当前落盘内容为准，不依赖之前的对话或摘要。

执行前按 `docs/规范/工具协作.md` 的后续触发条件完成全部核对；不符合时按其修复规则处理，证据不足时停止并说明，不要开始实现。

核对通过后，逐项实现规范核对后的阶段 1 全部未完成 checkbox；不得删除、合并或改写阶段 2 及其他未完成任务。

禁止接入真实接口、真实数据、权限判断或业务状态流转；mock 须与真实链路隔离（如独立目录、环境开关或占位 service），边界写进 design 或代码。

完成后只更新实际完成的阶段 1 任务状态，列出完成项、未完成项和仍待阶段 2 处理的全部任务，并说明如何本地预览。不要归档，不要开始阶段 2。
```

你在本地对照设计图验收 UI 与页面跳转。不通过则让 AI 继续修阶段 1，通过后再回复 **「UI 验收通过」**，发送第二轮：

```text
UI 验收通过。请实现本次变更所归属的 OpenSpec change 中「阶段 2：真实接口、数据、权限、状态流转与错误处理」的任务。

Change ID：<change-id>

请先根据 change-id 重新读取并逐项对照该 change 的 proposal、specs、design、tasks、变化记录，以及 tasks 中的“任务重组基线”和“任务重组映射”；以当前落盘内容为准，不依赖之前的对话或摘要。

执行前按 `docs/规范/工具协作.md` 的后续触发条件完成全部核对；不符合时按其修复规则处理，证据不足时停止并说明，不要开始实现。

核对通过后，逐项实现规范核对后的阶段 2 全部未完成实现 checkbox；不要只按阶段标题概括实现。

完成后根据“任务重组映射”检查整个 tasks.md，不得遗留未实现或未说明的需求任务。按计划应在下一步执行的验证类任务可以保持“待验证”，但必须完整列出，且此时不得宣称整个 change 已完成或可归档。更新实际完成的任务状态，并说明完整任务对照结果、实际改动、已经执行的检查、待验证项和尚未覆盖的风险；存在实现遗漏时不得宣称实现完成。不要归档。
```

检查实际改动是否符合更新后的 proposal 和 specs。所有任务完成后，进入验证。

## 5. 验证

让 AI 核对实现是否满足 change，并提供真实的测试、构建和检查结果。验证未通过时，先修复并重新验证，不要进入归档。

在 AI Chat 中发送：

```text
请验证本次变更所归属的 OpenSpec change。

Change ID：<change-id>

请根据 change-id 重新读取并逐项对照更新后的 proposal、specs、design、tasks、变化记录，以及 tasks 中的“任务重组基线”和“任务重组映射”；以当前落盘内容为准，不依赖之前的对话或摘要。

按顺序验证：
1. 按 `docs/规范/工具协作.md` 的后续触发条件判断是否执行完整的设计源、基线、映射、checkbox 和职责核对；任一检查失败时不得继续实现或声明无遗漏。
2. 建立“当前需求与验收标准 → tasks 任务编号 → 实现与验证证据”的逐项对应；采用两阶段重组时同时纳入基线和映射，确认两个阶段的实现任务均已处理；否则确认普通 tasks 中的实现任务均已处理。
3. 执行并更新标为“待验证”的任务，运行与本次改动及其直接影响相关的测试、构建或检查。

列出实际执行的检查、结果、遗漏项、未覆盖项和剩余风险；存在适用检查不一致、分类冲突、任务遗漏、仍无结论的待验证项或检查失败时，不要宣称完成，也不要归档。
```

只有验证通过、未覆盖项和剩余风险都已确认，才能进入归档。

## 6. 归档

验收完成后，让 AI 同步规格并归档 change，为本次需求变更收尾。

在 AI Chat 中发送：

```text
请归档本次变更所归属的 OpenSpec change。

Change ID：<change-id>

归档前确认验证结果有效，并按项目约定同步需要更新的规格。若验证无效、同步失败或无法归档，请明确说明原因，不要伪装完成；成功后告诉我归档位置和规格更新结果。
```

确认 AI 返回了真实的归档位置和规格更新结果，需求变更流程结束。
