# 需求变更工作流

> **范围**：仅本规范仓工作流教程，**不**随 Vibe Coding Rule 接入复制到业务项目。使用时在本仓库查阅或打开对应页面。默认使用 OpenSpec 主流程。强制底线见 [`AGENTS.md`](./AGENTS.md)，工具默认假定与协作规则见 [`docs/规范/工具协作.md`](./docs/规范/工具协作.md)。各步骤可以在同一个或不同的 AI Chat 中完成；操作已有 change 时，须在提示词中提供明确的 change-id。从零新增需求见 [`新增需求工作流.md`](./新增需求工作流.md)。
>
> **核心原则：不急着新建——先定位原记录、判断变化等级、把正文更新为当前口径、追加变化记录（不覆盖历史），再改代码。**

本流程只在用户明确确认要把内容作为正式需求变更处理后使用。AI 识别到潜在需求变更时必须先说明依据和影响并请求需求意图确认；确认前不得查找 OpenSpec 候选或自动进入本流程。用户确认需求意图后，才能执行下方只读原记录与候选扫描；继续现有 change 时确认载体前不得写入，新建关联 change 时确认新口径并由用户主动要求创建前不得写入，也不得形成最终设计、计划或开始实现。

开始前按默认假定检查 OpenSpec 是否可用（本流程以 OpenSpec change 为唯一载体，故先检 OpenSpec）。若发现未安装、未初始化或无法调用，立即中断并明确告知是 OpenSpec，说明已完成项与阻塞点；不自动安装，也不改用 issue/PR/任务卡或其他轻量载体继续。后续步骤若需要 CodeGraph 或 Superpowers 却不可用，同样中断并指明该工具；不因「流程开头未预检」而视为可缺省。

先只读扫描相关的进行中及历史 change，核对原始需求来源、当前口径、任务状态和验收，并始终列出全部候选的 `change-id`、状态、关系和建议。即使只有一个候选，也必须等待用户明确确认是更新该 change 还是新建关联 change；没有候选时也必须说明并等待用户决定。确认承接方向前不创建、选定或更新 change，也不修改代码。

需求变更走六个步骤：

```text
定位原记录 → 澄清并确认新口径 →〔需要时新建关联 change〕→ 更新 change 并追加变化记录 → 实现 → 验证 → 归档
```

先由 AI 根据变更描述只读定位原记录和候选 `change-id`，由用户明确确认继续原 change 还是新建关联 change 的方向。若用户确认需新建 change 承接变更，必须先完成第 2 步并取得用户对目标、范围和验收口径的明确确认；此后由用户主动要求创建，AI 根据已确认口径生成简短、明确的小写英文 ID，使用短横线分隔，检查现有 change 并说明与原 change 的关系。发现同名或可能承载该变更的 change 时停止，不得改用其他 ID 或创建；没有冲突时直接创建，无需用户单独确认 ID。执行后续步骤时，将 AI 返回的最终 ID 填入提示词中的 `<change-id>` 占位符，确保切换 AI Chat 后仍能定位同一个 change。

**变更涉及 UI 设计图时**采用条件式两阶段实现：主流程不变，在第 3 步更新 `design` 与 `tasks.md` 时按两阶段拆分，第 4 步分两轮执行（先 UI + mock，**人工验收 UI** 后再接真实逻辑）。

## 1. 定位原记录

来了一个变更，先找到它对应的原 change 或外部需求记录，**别上来就新建**。确认这真的是「已有需求的变化」，而不是全新需求、恢复既有口径的缺陷修复或随口讨论。缺陷修复不按技术类别（样式、布局、交互、接口、数据或业务逻辑）自动当成需求变更；有已确认口径证明违背既有行为时属恢复；若将重新定义目标、范围、可观察行为、接口、数据、权限、流程或验收，才按需求变更处理；证据不足或冲突时先说明并取得确认。

在 AI Chat 中发送：

```text
我收到一项变更：<变更描述>。

请先了解与本次变更直接相关的项目现状，定位对应的原 OpenSpec change 或外部需求记录，并核对原始需求来源。

列出全部候选，逐项说明原记录位置、change-id、当前有效口径、change 状态（进行中 / 已完成未归档 / 已归档）、与本次变更的关系及承接建议。即使只有一个候选也等待我确认；没有候选时明确说明。

本步仅定位原记录、判断变化等级和承接方向，不要自动选定承接载体。
```

判断口诀：

-   **继续用原 change**——目标没变、还是同一项能力，只是细化或调整。例如「密码最短长度从 8 改到 12」，仍是同一条密码策略。
-   **新建 change**——目标变成了另一件事、影响完全不同的模块、或需要独立发布和回滚。例如原本只做「密码策略」，现在要追加「登录失败锁定账号」，这是另一项能力。已归档的 change 原则上新开 change 承接，并注明与旧 change 的关系。

再判断变化等级：

| 等级         | 典型例子                                     | 处理力度                     |
| ------------ | -------------------------------------------- | ---------------------------- |
| **小变化**   | 文案微调、不影响流程的展示调整               | 可简化流程，第 2 步可跳过    |
| **中等变化** | 校验规则调整、新增一种 OAuth 绑定方式        | 走澄清 + 影响分析            |
| **重大变化** | 认证协议更换、权限模型调整、默认登录策略变化 | 完整走澄清 + 影响分析 + 回归 |

注意：看着像「小变化」，但只要它影响了操作路径、权限可见性、状态流转、验收标准、用户对结果的预期、数据提交意图或合规含义，就按中等/重大处理。

本步只确认原记录、变化等级以及继续原 change / 新建关联 change 的方向，不提出或创建新 `change-id`。即使用户已确认需要新建关联 change，也必须先完成第 2 步，明确并确认目标、范围和验收口径。

## 2. 澄清新口径

中等/重大变化必做，小变化可简化。若需要新建关联 change，无论变化等级如何，都不得跳过目标、范围和验收口径的明确确认。让 AI 把「原来是怎样、现在要变成怎样、差异在哪、影响哪些范围、验收怎么跟着变」理清楚。这一步只澄清，不更新 change，也不实现。

在 AI Chat 中发送：

```text
请基于原需求记录澄清本次变更的新口径。

原 Change ID：<用户已确认的原 change-id>

请使用 Superpowers 的 `brainstorming` 方法，先读取并对照原始需求来源和原 change 的当前内容，再逐步提问，帮我澄清原口径 → 新口径的差异、目标、范围、非目标、验收标准变化和待决问题。

本步仅澄清新口径；待我确认目标、范围和验收口径后，再决定继续原 change 或按已确认方向新建关联 change。
```

与 AI 对话，直到你明确确认目标、范围和验收口径。

若用户确认继续使用原 change，可直接进入下一步。若用户已确认需要**新建关联 change**，在目标、范围和验收口径确认后发送：

```text
请根据本次变更和已定位的原记录新建 OpenSpec change，并写明它与原 change 的关系。

原 Change ID：<用户已确认的原 change-id>
变更说明：<本次变更已确认的目标、范围和验收口径>

如果本次变更来源或本消息包含新的 UI 设计图，本步仅按已确认的文字新口径创建初始任务，不读取或解释新设计图，也不加入设计衍生任务；列出暂未读取的 UI 输入。若文字新口径与设计图无法可靠分离，请停止并说明。先形成原始 tasks，设计图留到第 3 步的 UI 补充提示词处理。

请读取并对照本次变更与原记录，根据已确认的新口径生成简短、明确、使用小写英文和短横线分隔的 change-id，说明与原 change 的关系，并只读检查现有 change。

如果存在同名 change，或发现已有 change 可能承载本次变更，请停止并列出冲突项、状态和关系；不要改用其他 ID，也不要创建 change。

确认无冲突后，使用生成的 change-id 创建 change，起草相互一致的 proposal、specs、design 和 tasks，并写明与原 change 的关系。

成功后返回最终 change-id 和创建内容；先不要实现。
```

确认归属、状态、继续/新建决定、冲突检查结果和最终 `change-id` 均准确后，再进入下一步。

## 3. 更新 change 并追加变化记录

让 AI 把受影响的文档改成**当前有效口径**，并**追加**一条变化记录——保留原口径，**绝不覆盖历史**。按变化类型决定改哪些文件：

| 变化类型 | 要改的文件                           |
| -------- | ------------------------------------ |
| 补充说明 | `proposal`                           |
| 范围变化 | `proposal` + `specs`                 |
| 验收调整 | `specs` + `tasks`                    |
| 方案调整 | `design` + `tasks`（可能含 `specs`） |

在 AI Chat 中发送：

```text
请将本次变更所归属的 OpenSpec change 正文更新为当前口径，并追加一条变化记录（不覆盖历史）。

Change ID：<change-id>

如果本次变更包含新的 UI 设计图，本步仅按已确认的文字新口径更新正文、验收和当前 tasks，不读取或解释新设计图，也不加入设计衍生任务；列出暂未读取的 UI 输入。若文字新口径与设计图无法可靠分离，请停止并说明。更新后的 tasks 是本轮读取 UI 前的任务状态；后续先固化基线或快照，不得覆盖已有基线。

中等/重大变化请重新分析直接影响的模块、调用方、数据流和测试面，并把结论、证据与未覆盖范围写进 `design` / `tasks`。

将变化记录追加到本 change，包含：时间、来源、变化类型、原口径、新口径、影响范围、验收变化、实现影响、验证调整、剩余风险；issue/PR 等可作来源引用或同步摘要。

同步 tasks 与验收标准——不能拿旧测试覆盖新验收。先不要实现。
```

**变更涉及 UI 设计图时**，再发送一条补充提示词：

```text
本次变更涉及 UI 设计图，请补充更新其所属 OpenSpec change 的 design 和 tasks。

Change ID：<change-id>
UI 设计图来源：<实际本地路径 / Figma·蓝湖等链接 / 附件说明及作用 / 沿用当前 change 已记录的来源集合>

以上字段必须替换为实际且可读取的来源，不得保留占位符。先不要读取 UI 来源；先读取该 change 的 proposal、specs、design、当前 tasks 和变化记录，完成设计源、基线、快照与旧结构修复门禁；未通过则停止。

门禁通过后，读取 UI 来源并更新 design 和 tasks，形成两个阶段、任务映射、未决问题及核对记录。完成后返回核对结果，先不要实现。

```

检查正文、变化记录、任务拆分与验收标准是否准确。有 UI 设计图时，确认 AI 已读取你提供的设计图位置，且 `tasks.md` 已按两阶段拆分。确认无误后，再进入实现。

## 4. 实现

正文、变化记录与任务全部同步后，再让 AI 按 change 中的任务清单完成开发，并同步任务状态。实现途中又冒出新变化，回到第 2 步重新判断，**别直接把变化吸进代码**。

**无 UI 设计图**，在 AI Chat 中发送：

```text
请实现本次变更所归属的 OpenSpec change。

Change ID：<change-id>

请先读取并对照该 change 的 proposal、specs、design、tasks 和变化记录，再按 tasks 逐项实现。

完成后更新任务状态，并说明实际改动、已经执行的检查、尚未验证的内容和剩余风险。不要归档。
```

**有 UI 设计图**——分两轮执行；阶段 1 由 AI 实现，**UI 由你人工验收**，通过后再发阶段 2。

第一轮，在 AI Chat 中发送：

```text
请只实现本次变更所归属的 OpenSpec change 中「阶段 1：UI、隔离 mock 与页面跳转」的任务，不要开始阶段 2。

Change ID：<change-id>

请先读取该 change 的 proposal、specs、design、tasks 和变化记录，完成设计源、基线、映射、checkbox 和职责核对。

核对通过后，逐项实现阶段 1 的全部未完成任务，不得删除、合并或改写阶段 2 及其他未完成任务。

遵守 UI 两阶段实现的阶段边界和 mock 隔离要求。

完成后仅更新实际完成的阶段 1 任务状态，列出完成项、未完成项及仍待阶段 2 处理的全部任务，并说明如何本地预览。不要归档或开始阶段 2。
```

你在本地对照设计图验收 UI 与页面跳转。不通过则让 AI 继续修阶段 1，通过后再回复 **「UI 验收通过」**，发送第二轮：

```text
UI 验收通过。请实现本次变更所归属的 OpenSpec change 中「阶段 2：真实接口、数据、权限、状态流转与错误处理」的任务。

Change ID：<change-id>

请先读取该 change 的 proposal、specs、design、tasks 和变化记录，完成设计源、基线、映射、checkbox 和职责核对。

核对通过后，逐项实现阶段 2 的全部未完成实现任务。

完成后根据“任务重组映射”检查整个 tasks，不得遗留未实现或未说明的需求任务。计划在下一步执行的验证类任务可保持“待验证”，但须完整列出，且此时不得宣称整个 change 已完成或可归档。更新实际完成的任务状态，说明完整任务对照结果、实际改动、已执行的检查、待验证项和未覆盖风险；存在实现遗漏时不得宣称实现完成。不要归档。
```

检查实际改动是否符合更新后的 proposal 和 specs。所有任务完成后，进入验证。

## 5. 验证

让 AI 核对实现是否满足 change，并提供真实的测试、构建和检查结果。验证未通过时，先修复并重新验证，不要进入归档。

在 AI Chat 中发送：

```text
请验证本次变更所归属的 OpenSpec change。

Change ID：<change-id>

请读取并逐项对照该 change 更新后的 proposal、specs、design、tasks 和变化记录。

按顺序验证：
1. 若存在 UI 重组标记，完成设计源、基线、映射、checkbox 和职责核对。
2. 建立“当前需求与验收标准 → tasks 任务编号 → 实现与验证证据”的逐项对应，确认全部实现任务均已处理。
3. 执行并更新标为“待验证”的任务，运行与本次改动及其直接影响相关的测试、构建或检查。

列出实际执行的检查及其结果、遗漏项、未覆盖项和剩余风险；若存在适用检查不一致、分类冲突、任务遗漏、仍无结论的待验证项或检查失败，不要宣称完成或归档。
```

只有验证通过、未覆盖项和剩余风险都已确认，才能进入归档。

## 6. 归档

验收完成后，让 AI 同步规格并归档 change，为本次需求变更收尾。

在 AI Chat 中发送：

```text
请归档本次变更所归属的 OpenSpec change。

Change ID：<change-id>

请按项目规范确认验证结果有效，同步需要更新的规格并归档。成功后告诉我归档位置和规格更新结果。
```

确认 AI 返回了真实的归档位置和规格更新结果，需求变更流程结束。
