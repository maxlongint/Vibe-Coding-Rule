# AGENTS.md

> Vibe Coding Rule 版本：v1.1.4
> 上游仓库：https://github.com/maxlongint/Vibe-Coding-Rule
> 完整规范仓库更新记录：CHANGELOG.md（仅复制本文件时无需该文件）

面向 AI 辅助开发的协作规范。**复制本文件到项目根目录即可使用**，不依赖任何特定工具或插件，适用于前端、后端、全栈、脚本等项目。

- **项目规范放哪**：需要新增长期有效的技术栈、框架、命令、UI/API、业务约定等项目规范时，再创建 `docs/`，并用 `docs/README.md` 作为索引；只使用本文件且没有补充项目规范时，无需预先创建 `docs/`。项目规范不写进 `AGENTS.md`，也不拆到各 AI 工具自有配置里。进行中的需求在用户明确选择 OpenSpec 时放 `openspec/changes/`，否则使用当前任务或用户指定的 issue/PR 等载体。
- **自定义 skill 放哪**：需要新增项目级 skill 时，再创建 `.agents/skills/<skill-name>/`，入口文件为 `SKILL.md`（可按需附带脚本或参考文档），并用 `.agents/README.md` 作为所有 skill 技能包的目录索引；新增或移除 skill 时同步更新索引。不使用项目级 skill 时，无需预先创建 `.agents/`。项目级 skill 随仓库共享给团队，不写进 `docs/` 或 `AGENTS.md`；个人全局 skill 放用户目录，不作为项目规范。
- **别动本文件**：**AI 未经用户明确要求，不得擅自修改 `AGENTS.md`**——不编辑、不覆盖、不顺手改写，也不把项目规则合并进来。

## 速查：六条底线

1. **不伪造成功**——真实失败要显式暴露，不用默认值、空对象、mock 或"兜底成功"掩盖。
2. **完成前必须验证**——并说明结果；无法验证就说原因和剩余风险，不得把未验证说成已通过。
3. **改行为前先记录**——改变外部可观察行为/接口/数据/权限/流程/验收前，先有可追溯记录再改代码。
4. **不碰敏感信息**——不读取、打印、提交或传播密钥、凭据与未脱敏数据。
5. **不擅自提交发布**——未经用户明确要求，不 commit / push / merge / deploy。
6. **拿不准就问**——无法判断时停下来确认，不要猜。

以下为展开，冲突时按文末「优先级」处理。

## 1. 工程与正确性

- 优先保证业务正确性、清晰边界、可测试性；不为"看起来能跑"掩盖真实问题。
- 遵循项目既有架构、目录、命名与调用方式；不随意引入新服务、新技术栈或绕过依赖（确有必要时先说明原因、影响和验证成本）。
- 生产与真实业务链路不得用 mock、本地缓存、seed 或旧链路冒充真实依赖；测试、演示、本地开发可用模拟数据，但要隔离环境并标明边界。
- 区分"无数据"与"出错"：业务上确实无数据可返回空；字段不可读、不可解析或契约缺失属于错误，必须显式失败。
- 接口失败、权限失败、字段缺失、契约不完整必须显式反馈，不得用默认值、猜测值、空对象或旧字段掩盖。
- 面向用户的错误信息不泄露敏感内部细节。
- 关键写操作要幂等、给出进行中状态、等写入结果确认后再进入下一步，不提前宣称成功。
- 呈现层问题先查数据、状态与逻辑链路，再调外观。
- 在以上边界内，改动规模、实现路径与抽象程度遵循 §7。

## 2. 安全

- 不主动读取、打印、提交或传播密钥、token、密码、会话凭据、个人敏感信息或未脱敏生产数据，除非用户明确要求且符合权限边界。
- 不把 `.env`、凭据文件、连接串、访问令牌写入文档、示例、测试快照、日志或提交说明。
- 展示日志、报错、配置或命令输出前先判断是否含敏感信息，含则脱敏并说明范围。
- 发现疑似泄露要停止扩散、提示风险与受影响位置，不复制到其他文件或上下文。
- 不伪造权限、凭据或安全校验结果来宣称流程成功。

## 3. 验证

- 验证范围覆盖本次改动及其直接影响；纯文案、局部呈现可做最小验证。
- 涉及模块结构、交互流程、输入校验、服务调用、权限、数据契约或外部集成时，扩大到直接调用方与相关业务测试。
- 验证方式匹配项目技术栈：单元/集成测试、类型检查、lint、构建、手工验收等；优先用项目 manifest（`package.json`、`pyproject.toml`、`Makefile` 等）或 README 中的脚本。
- 能自动化的覆盖正向和关键反向断言；无法覆盖时说明原因和剩余风险。
- 若项目固定了运行时或工具链版本（如 `.node-version`、`.python-version`、`.ruby-version`、`.tool-versions`、`rust-toolchain` 等），执行相关命令前按项目约定的版本管理工具切换到对应版本。

## 4. 需求可追溯

### 留痕与载体

- 当改动会改变外部可观察行为、接口契约、数据结构、权限规则、状态流转、验收标准或用户操作路径时，**先有可追溯记录，再改代码**。
- **记录载体**：当前任务本身可作为轻量载体；用户已指定 issue、任务卡、PR 描述、变更说明、设计文档或项目约定载体时，按其选择。只有项目约定或用户要求另建、更新载体但未指定时才询问，不得擅自创建。
- 同一需求的需求、规格与任务保持**单一事实来源**；辅助工具产出的澄清、设计和计划回填到该来源，不另存平行口径。
- 轻量记录至少写清三项：**来源、目标与变更范围；验收及验证方式；未验证项与剩余风险**。
- 涉及代码改动时先做影响分析，覆盖直接影响模块、调用方、数据流和测试面；有代码分析工具时优先使用，否则人工阅读源码与测试并记录未覆盖范围。
- 只读分析、复现和影响判断无需先建或更新需求载体。

### 需求性质决策

AI 只判断是否需要询问，需求事实由用户决定。用户不必使用固定术语，但须明确选择或等价表达下表中的需求性质；仅实现目标描述清楚，不等于需求性质已经确认。已有需求记录、用户本轮明确口径和用户此前已确认口径属于**需求证据**；源码、测试、版本历史和现有行为只属于**代码事实**。

按下表从上到下判断，命中第一项即停止。表中的「直接继续」只表示不再询问需求性质，不免除留痕、影响分析、验证或权限要求。

| 顺序 | 首个命中的条件 | 处理 |
| --- | --- | --- |
| 1 | 仅做只读分析、复现或影响判断 | 直接继续 |
| 2 | 用户已明确选择或等价表达需求性质，且不存在尚未解决的需求证据冲突 | 按用户表达继续，不重复询问 |
| 3 | 用户称其为全新需求、需求变更、缺陷修复或仅调整实现，但与需求证据冲突且尚未正面解决 | 暂停实现，中性询问 |
| 4 | 性质未确认，且需求证据或明确的「从 A 改为 B」表明可能修改既有需求口径、验收标准、接口契约、用户操作路径、权限规则或状态流转 | 暂停实现，中性询问 |
| 5 | 性质未确认，但可明确判定为独立新增能力、有需求证据支持且无冲突的既定行为恢复、保持原需求仅调整实现，或不触及上一项范围的局部文案、展示和内部调整 | 直接继续，不额外询问 |
| 6 | 性质未确认、不属于前项可直接处理情形，且只有代码或现有行为 A 与请求结果 B 的差异 | 暂停实现，中性询问；代码差异可以触发澄清，但不能证明需求变更 |
| 7 | 其他信息不足、答复含糊或边界无法判断 | 暂停实现，只询问尚未解决的维度 |

- **性质明确的例子**：用户说明「这是新需求」「确认按需求变更处理」「这是缺陷，预期应为……」「保持原需求，只调整实现」，或表达同等明确的含义。明确确认需求变更及目标口径，视为已经正面处理「旧口径 → 新口径」差异。
- **中性询问**：分别说明需求证据（如有）、代码事实、请求目标、影响范围、风险和未知项，再请用户选择按「全新需求」「确认需求变更」「恢复既定行为的缺陷修复」「保持原需求、仅调整实现」或其他方式处理，不得预设答案。
- **收到答复后**：答复明确且解决已指出冲突时，以其作为需求事实，先更新唯一载体再实现；答复已明确性质但未解决已指出冲突时，只追问冲突本身；其他答复仍含糊时，也只追问未决点。用户已明确确认「从旧口径改为新口径」后，不得仅因旧记录尚未更新而重复询问，应先同步记录。

### OpenSpec 门禁

- 只有用户明确要求使用 OpenSpec、直接调用 `/opsx:*`，或明确要求操作某个 OpenSpec change 时，才可进入对应流程。
- 仓库已安装 OpenSpec、存在 `openspec/` 目录、改动需要留痕、影响外部行为、AI 或辅助 skill 建议使用，以及「新增」「修改」「修复」「调整」等普通开发指令，均不构成授权。
- 未经显式触发，不得自行创建、更新、应用或归档 change；使用当前任务或用户指定的其他载体，不得因为需要留痕而转入 OpenSpec。
- 用户已显式选择 OpenSpec 但当前环境不可用时，必须说明真实失败，并询问安装 OpenSpec 或改用其他载体；不得伪装成已完成 OpenSpec 流程。

## 5. 注释与编码

- 注释解释意图、边界、约束和非显然决策，不复述代码；禁止"获取用户""调用接口"这类空注释。
- 公共接口、复杂业务规则、权限判断、状态流转、外部集成、非显然副作用要说明用途与关键输入输出。
- 临时方案、技术债务用 `TODO:` 或 `FIXME:` 标注背景与影响范围。
- 改逻辑时同步更新相关注释，不留过期注释误导后续维护者。
- 源码、配置、文档统一 UTF-8，不带 BOM；不把乱码或错误转码内容写回仓库；中文项目注释与文档默认简体中文。
- 新增中文内容后读取目标文件确认显示正常。

## 6. 提交与发布

- 未经用户明确要求，不得 commit、push、merge、release、deploy 或任何改变仓库历史、远端或线上状态的操作。
- 用户要求提交/发布前，先汇总：当前变更、关联需求、验证结果、剩余风险；有未完成验证或高风险改动必须写明，不得用笼统表述掩盖。
- 提交信息默认简体中文（项目有英文/Conventional Commits 等约定则从项目）；标题说明实际变更，避免"修改""更新""fix"等空泛词；与需求相关时注明记录路径或编号。

## 7. 实施与改动

动手改代码时的**范围与复杂度**控制；与 §1 工程边界配合。需求澄清、任务拆解、测试优先节奏、change 生命周期等执行方法，可在启用相应工具后由 OpenSpec（仅在用户明确选择时）、Superpowers 或项目 skill 辅助，**不在此重复工具级流程**。未启用时直接按本文件执行，无需安装工具或创建补充文档。

### 简洁

- 用最少代码解决当前问题；不为一次性需求建抽象层或「以后可能用到」的可配置项。
- 若改动会让资深维护者觉得过度设计，先简化再交付。
- 仍须满足 §1 的正确性、边界与可测试性；简洁不能牺牲显式错误处理与契约清晰。

### 目标变更与非目标不变式

- 修改代码前，必须区分**目标变更**与**非目标不变式**：目标变更是用户明确要求改变的行为、输出、样式、接口、数据、权限、流程或验收结果；非目标不变式是用户未要求改变的既有行为、接口契约、数据结构、权限规则、错误处理、性能特征、布局尺寸、调用方式、兼容性等。
- 实现时只能改变目标变更所必需的内容，不得通过破坏、绕开或改变非目标不变式来间接制造「完成效果」。
- 禁止为了达成局部目标而擅自改变未授权行为，例如：为了修复字段显示问题而改变接口返回结构；为了让测试通过而放宽校验、删除断言、吞掉错误或修改测试期望；为了处理异常而返回空对象、默认值或假成功；为了修复权限问题而绕过权限判断或扩大默认权限；为了修复局部样式问题而改变组件尺寸、布局结构或相邻元素行为；为了消除类型错误而改成 `any`、移除类型约束或跳过关键分支；为了兼容当前输入而破坏已有合法输入、历史数据或调用方兼容性。
- 如果确实必须改变非目标不变式，必须先说明要改变什么、为什么无法避免、会影响哪些调用方/用户路径/数据/验收标准、如何验证，并等用户确认后再实施。

### 精准改动

- 只改与当前任务直接相关的代码；不顺手改相邻逻辑、注释、格式或「顺便重构」。
- 匹配项目既有风格与调用方式，即使你更偏好另一种写法。
- 因本次修改产生的无用导入、变量、分支要删；改动前已存在的死代码只指出，不擅自删除。
- 完成后说明本次实际改变了什么、哪些相关行为保持不变、如何验证目标变更生效，以及如何确认非目标行为没有被误改；无法验证的要列出原因和剩余风险。

## 优先级（冲突时）

```text
1. 不可覆盖：平台/法律/权限安全约束、提交发布权限、需求可追溯底线、验证底线
2. 用户本轮明确指令
3. 项目 AGENTS.md 与 `docs/` 下的项目自有规则
4. 通用默认习惯
```

- **需求事实**（要做什么）：用户本轮确认 → 可追溯记录。不由代码现状决定。
- **代码事实**（代码是什么样）：实际源码与测试输出 → 代码分析工具或人工阅读（记录剩余风险）。不能覆盖需求验收标准。
- 仍无法判断时，停下来向用户确认，不要猜。
