# AGENTS.md

> Vibe Coding Rule 版本：v1.1.4
> 上游仓库：https://github.com/maxlongint/Vibe-Coding-Rule
> 完整规范仓库更新记录：CHANGELOG.md（仅复制本文件时无需该文件）

面向 AI 辅助开发的通用协作规范。复制本文件到项目根目录即可独立使用，不依赖特定工具或插件。

- `docs/` 存在时必须以 `docs/README.md` 作为项目规范目录索引；需要补充长期有效的技术栈、框架、架构、命令、UI/API 或业务约定时，再新增对应文档并同步索引，没有补充规范时无需创建该目录。项目规范不写入 `AGENTS.md`，也不拆到各 AI 工具配置；进行中需求使用本任务或用户指定载体。
- 添加自建或第三方项目级 skill 前，必须先创建 `.agents/` 并以 `.agents/README.md` 作为目录索引和安装入口，再创建 `.agents/skills/<skill-name>/SKILL.md` 并同步索引；没有项目级 skill 时无需创建实际 skill 包。项目级 skill 随仓库共享，不写入 `docs/` 或 `AGENTS.md`；个人全局 skill 不作为项目规范。
- AI 未经用户明确要求不得修改本文件，也不得把项目规则或工具说明合并进来。

## 速查：六条底线

1. **不伪造成功**：真实失败必须显式暴露，不用默认值、空对象、mock 或旧链路掩盖。
2. **完成前必须验证**：说明实际结果；无法验证时写明原因、未覆盖项和剩余风险。
3. **改行为前先记录**：改变外部行为、接口、数据、权限、流程或验收前，先建立可追溯记录。
4. **不碰敏感信息**：不主动读取、打印、提交或传播密钥、凭据、个人敏感信息或未脱敏生产数据；仅在用户明确要求且权限允许时处理。
5. **不擅自提交发布**：未经用户明确要求，不 commit、push、merge、release 或 deploy。
6. **拿不准就问**：遇到需求证据冲突，或请求可能改变既有需求、验收、接口、权限、流程且边界不清时，说明事实与风险后再确认，不靠猜测推进。

## 1. 正确性与安全

- 优先保证业务正确性、边界清晰和可测试性；遵循项目既有架构、目录、命名与调用方式。引入新服务、技术栈或绕过依赖前，说明原因、影响和验证成本。
- 生产和真实业务链路不得用 mock、本地缓存、seed 或旧链路冒充真实依赖；测试、演示和本地开发的模拟数据必须隔离并标明边界。
- 只有业务上确实无数据时才能返回空；不可读、不可解析、字段或契约缺失、接口或权限失败都属于错误，不得用默认值、猜测值、空对象或旧字段掩盖。
- 面向用户的错误信息不得泄露内部敏感细节；呈现层问题先检查数据、状态和逻辑链路，再调整外观。
- 不把 `.env`、凭据、连接串或令牌写入源码、文档、示例、测试快照、日志或提交说明。
- 展示日志、报错、配置或命令输出前先检查敏感信息；含敏感内容时脱敏并说明范围。发现疑似泄露时停止扩散、提示风险和位置，不复制到其他文件或上下文。
- 不伪造权限、凭据或安全校验结果。关键写操作应幂等，给出进行中状态，并在确认写入结果后再进入下一步。

## 2. 验证

- 验证覆盖本次改动及直接影响；涉及结构、交互、输入校验、服务、权限、数据契约或外部集成时，扩展到直接调用方和相关业务测试。纯文案或局部呈现可做最小验证。
- 使用与项目技术栈匹配的单元/集成测试、类型检查、lint、构建或手工验收，优先采用 manifest、README 或项目文档中的既有命令。
- 项目固定运行时或工具链版本时，先按其版本管理约定切换；能自动化的验证覆盖正向和关键反向断言。
- 完成前说明实际采用的验证方式和结果；失败、无法执行或未覆盖的部分必须显式说明，不得把未验证说成已通过。

## 3. 需求可追溯

- 改变外部可观察行为、接口契约、数据结构、权限、状态流转、验收标准或用户路径前，先更新可追溯记录，再改代码。
- 当前任务可作为轻量载体；用户已指定 issue、PR、任务卡、变更说明、设计文档或项目约定载体时按其选择。只有项目或用户要求另建载体但未指定位置时才询问，不擅自创建。
- 同一需求的目标、规格和任务保持**单一事实来源**；工具产生且会形成需求口径、设计或计划的成果回填该来源，不另存平行口径。只读分析可保留在当前答复中。
- 轻量记录至少写清：来源、目标与范围；验收及验证方式；未验证项与剩余风险。
- 代码改动前分析直接影响模块、调用方、数据流和测试面；有适用的代码分析工具时优先使用，否则人工阅读源码和测试并记录未覆盖范围。只读分析、复现和影响判断无需先建或更新载体。

### 需求性质判断

- 需求事实由用户和已确认的需求记录决定；实现目标描述清楚不等于需求性质已经确认。源码、测试、版本历史和现有行为只属于代码事实，不能单独证明需求口径。
- 只读分析；用户已明确性质且无未解决冲突；边界明确的独立新增、有需求证据支持且无冲突的既定行为恢复、仅调整实现或不改变既有口径的局部文案/展示/内部调整，可以直接继续。
- 若请求可能改变既有需求、验收、接口、用户路径、权限或状态，性质尚未确认；或用户口径与需求证据冲突；或信息不足、边界无法判断，应暂停实现并中性询问。
- 只有代码或现有行为 A 与请求结果 B 的差异时，不据此认定缺陷或需求变更；无法归入直接继续情形时，先确认按新需求、需求变更、既定行为缺陷修复、仅调整实现或其他方式处理。
- 中性询问应分别说明需求证据、代码事实、请求目标、影响、风险和未知项，不预设答案。答复明确且解决冲突时，更新唯一载体后再实现；答复已明确性质但未解决冲突时，只追问冲突；答复仍含糊时，只追问未决点。不得因旧记录尚未同步而重复询问已解决的问题。

## 4. 实施边界

- 修改前区分**目标变更**与**非目标不变式**。目标变更是用户明确要求改变的结果；未要求改变的行为、接口、数据、权限、错误处理、性能、布局、调用方式和兼容性均视为不变式。
- 只改变实现目标所必需的内容，不通过放宽校验、删除断言或修改测试期望、吞错、返回默认成功、绕过权限或扩大默认权限、削弱类型约束、破坏兼容性或改变无关布局来制造局部效果。
- 确实必须改变非目标不变式时，先说明原因、受影响调用方/用户路径/数据/验收和验证方式，等待用户确认后再实施。
- 用最少代码解决当前问题，不为一次性或假想需求增加抽象与配置；简洁不得牺牲正确性、显式错误处理和契约清晰。
- 只修改当前任务相关内容，不顺手重构、格式化或清理相邻代码。删除本次产生的无用导入、变量和分支；既有死代码只指出，不擅自删除。
- 注释解释意图、边界和非显然副作用，不复述代码；公共接口、复杂规则、权限、状态流转和外部集成说明关键输入输出。临时方案或技术债务用 `TODO:` / `FIXME:` 标明背景和影响，逻辑变化时同步更新注释。
- 遵循项目既有编码；不得把乱码或错误转码写回。源码、配置和文档默认 UTF-8 无 BOM，中文项目新增内容默认简体中文，写入后读取确认显示正常。
- 完成后说明实际改变、保持不变的相关行为、验证证据、未验证项和剩余风险。

## 5. 提交、发布与外部副作用

- 未经用户明确要求，不得 commit、push、merge、release、deploy 或执行其他改变仓库历史、远端和线上状态的操作。
- 用户要求提交或发布前，先汇总当前变更、关联需求、验证结果和剩余风险；未完成验证或存在高风险时必须明确披露。
- 提交信息遵循项目约定；无约定时默认使用简体中文，标题描述实际变更，避免“修改”“更新”“fix”等空泛措辞，并按需注明需求记录。

## 优先级

“六条底线”是核心规则的速查名称；不可覆盖范围以第 1 项列举为准。以下顺序仅用于处理规则冲突，各条中已经写明的用户授权条件仍按原文执行。

```text
1. 不可覆盖：平台/法律/权限安全约束、提交发布权限、需求可追溯与验证底线
2. 用户本轮明确指令
3. 项目 AGENTS.md、已确认的需求载体与 docs/ 下项目规则
4. 工具、skill 和通用默认行为
```

- **需求事实**：用户本轮确认 → 唯一需求载体；不由代码现状决定。
- **代码事实**：实际源码、测试和命令结果 → 代码分析工具或人工阅读；不能覆盖需求与验收。
- 仍无法判断时，停止实现并向用户确认，不要猜。
