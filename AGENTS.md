# AGENTS.md

本文是跨项目复用的 AI 协作章程：**强制底线与工程规则写在本文件**，不依赖 OpenSpec、CodeGraph、Superpowers 或任何特定 AI 插件即可单独使用。适用于前端、后端、全栈、脚本、移动端及基础设施类项目；技术栈差异通过项目 `README.md` 与 `docs/规范/` 补充，不写入本文件。

需求治理、可选工具链分工等细则见 `docs/规范/`（**Minimal 档位可不复制该目录**，见文末「规则包档位」）。

### 需求治理与影响分析（自洽）

以下内容为 **Minimal 档位及未复制 `docs/规范/` 时的完整执行依据**；不预设 OpenSpec、CodeGraph、Superpowers 或任何特定工具。

**何时必须有可追溯需求记录：** 本次修改会改变外部可观察行为、接口契约（含 REST/GraphQL/RPC、前后端数据契约、事件消息格式）、数据结构、权限规则、状态流转、验收标准或用户操作路径。

**可接受的记录载体（任选其一，须能追溯来源与验收）：** 需求管理系统或 issue、任务卡、PR 描述、变更说明、交付说明、设计/规格文档、或项目约定的轻量记录。未声明载体时，向用户确认；无法及时确认且范围较小时，可先用 PR 描述或任务记录承载，并说明剩余风险。

**轻量记录最少字段：** 变更范围、验证方式、未验证项与剩余风险（模板见 [轻量记录模板.md](./docs/规范/模板/轻量记录模板.md)，Minimal 档位可复制或按此字段手写）。

**影响分析（涉及代码改动时）：** 先判断直接影响模块、调用方、数据流与测试面；有代码分析工具（如 CodeGraph、IDE 引用、语言服务器）时优先使用；否则通过阅读源码、测试与项目文档人工分析，并在交付说明中记录未覆盖范围与剩余风险。

## 1. 工程守则与验证要求

> **本节是验证规则的权威来源。** 其他文档仅引用，不重复展开。

工程实现必须优先保证业务正确性、清晰边界、可测试性和长期维护成本。不得为了让程序“看起来能跑”而掩盖真实问题。

**实现规则：**

- 文件和模块必须保持聚焦，优先遵循现有架构、目录边界、命名方式和调用模式。
- 不得为了降低实现成本、缩短代码路径或绕过依赖，牺牲业务正确性、清晰分层、可测试性或长期维护成本。
- 不在未记录原因、影响范围和验证成本的情况下，引入额外服务、第三方后端、复杂基础设施或新技术栈。
- 涉及跨模块能力时，必须先明确模块边界和数据流；涉及权限、状态流转、数据一致性、审计留痕或外部集成时，不得只在调用方拼接局部操作来伪造业务成功。
- 在存在对应业务场景时，生产业务链路、真实验收链路和已接入真实逻辑的数据链路中，不得用 mock、本地缓存、seed 数据或旧链路替代真实依赖并宣称真实链路成功；测试、演示和本地开发场景可以使用模拟数据，但必须隔离环境并显式标明边界。
- 字段在业务上允许可选且有合理默认值时，该默认值应在契约文档或代码注释中显式说明，并在非生产环境可验证。
- 「无数据」与「读取出错」必须区分：业务上确实无数据时可返回空结果；字段不可读取、不可解析或缺少契约要求的必要字段时属于错误，必须显式失败。
- 接口失败、权限失败、字段缺失、契约不完整等真实失败，必须通过合适渠道显式反馈；不得用默认值、猜测值、空对象、旧字段或“兜底成功”掩盖，也不得在缺少契约时猜测业务状态。
- 面向用户的错误信息应避免泄露敏感内部细节；客户端、本地封装层或调用方不得为了避免报错而掩盖服务端、权限或数据契约错误。
- 在存在对应业务场景时，面向用户或外部系统的关键写操作必须防止重复提交（幂等）；处理期间应给出清晰进行中状态；操作完成后进入下一流程时，须正确等待写入结果并处理失败，不得提前宣称成功。
- 在存在对应业务场景时，外部资源、上传产物、临时访问 URI 等必须有明确的权限、有效期和消费方式约定；不得把仅供存储或内部标识使用的字段直接当作可对外使用的访问地址。
- 呈现层问题（若有）应先检查数据、状态与逻辑链路，再调整外观或表现细节。

**敏感信息与安全规则：**

- 不得主动读取、打印、提交或传播密钥、token、私钥、密码、会话凭据、个人敏感信息或未脱敏的生产数据，除非用户明确要求且符合权限边界和安全目的。
- 不得把 `.env`、凭据文件、云服务密钥、数据库连接串或访问令牌写入文档、示例、测试快照、日志或提交说明。
- 需要展示日志、报错、配置或命令输出时，应先判断是否包含敏感信息；如包含，应脱敏后再引用，并说明脱敏范围。
- 发现疑似泄露的敏感信息时，应停止扩散，提示风险、受影响位置和建议处理方式，不得为了继续任务而复制到其他文件或对话上下文中。
- 不得伪造权限、凭据或安全校验结果来宣称流程成功；涉及安全策略变化时，必须记录影响范围和验证方式。

**验证规则：**

- 验证范围必须覆盖本次改动及其直接影响。
- 纯文案、局部呈现或单处静态文档改动，可以做最小验证。
- 涉及模块结构、用户交互流程、输入校验、服务调用、权限、数据契约或外部集成时，必须扩大到直接调用方和相关业务测试。
- 验证方式应匹配项目技术栈与 manifest 中的脚本，例如：单元/集成测试、API 或契约测试、组件或 E2E 测试、类型检查、lint、构建打包、本地或预览环境手工验收等；前后端改动分别覆盖对应层级的直接调用方与关键路径。
- 能自动化验证的，应同时覆盖正向和关键反向断言；无法覆盖时说明原因和剩余风险。
- 完成前必须说明验证结果；无法验证须说明原因和剩余风险，不得把未验证结果表述为已通过。
- 涉及文档改动时，应同时按第 3 节编码与乱码防护要求验证中文显示。
- 选择验证命令时，应优先从项目 manifest（如 `package.json`、`pyproject.toml`、`Makefile` 等）、`docs/README.md`、`README.md`、`AGENTS.md` 或 `docs/规范/` 确认；仍不明确时，说明采用的最小验证方式和剩余风险。
- 项目为 Node.js 且执行 Node 相关验证命令时，应先检查是否存在 `.node-version`；如存在，应通过项目约定的版本管理工具（如 fnm、nvm）使用对应版本。

## 2. 注释规范

注释用于解释意图、边界、约束和非显然决策，不用于重复代码表面含义。

- 注释语言应优先与所在文件和项目既有风格一致；中文项目默认使用简体中文，除非项目已有其他约定。
- 新增或修改包含复杂业务规则、状态流转、外部调用、错误处理、权限判断或跨模块副作用的模块，且仅靠命名和结构无法清楚表达边界时，应在文件或模块入口处保留简短说明。
- 新增或修改公共接口、跨模块调用、复杂业务规则、非显然副作用或容易误用的函数、类、模块、脚本任务、配置块时，应说明用途、关键输入输出和重要副作用。
- 业务规则、权限判断、状态流转、数据标准化、错误处理、外部集成、缓存策略、兼容策略等非显然决策点，应说明为什么这样做。
- 临时方案、技术债务、待迁移逻辑必须显式标注 `TODO:` 或 `FIXME:`，并写明背景、影响范围和后续处理方向。
- 禁止写只复述代码的空注释，例如“获取用户”“设置状态”“调用接口”。
- 修改逻辑时必须同步更新相关注释，禁止保留过期注释误导后续维护者。
- 对外暴露的接口、公共工具、共享模块和关键配置，应优先写清使用约束和失败行为。
- 复杂实现若可通过拆分函数、重命名或调整结构变清楚，应优先改结构，不要只靠长注释解释混乱代码。

适用范围：源码、配置、脚本、测试文件，以及项目约定的公共接口说明。

## 3. 编码与乱码防护

源码、配置和文档必须稳定可读，禁止把乱码、错误编码或终端异常输出写回仓库。

- 源码、配置、脚本和文档统一使用 UTF-8；新建文件默认 UTF-8，不主动使用带 BOM 的编码。
- 不得将终端乱码、复制异常的中文、错误转码内容直接写回源码或文档。
- 执行环境出现编码警告或编码错误时，必须先排查文件编码问题，不得在未确认编码的情况下直接修改内容。
- 修改已有文件时，应保持原文件换行和编码风格，不做无关格式 churn。
- 跨平台路径说明应优先使用项目相对路径；涉及特定操作系统或 Shell 时，按项目本地环境说明使用对应语法。
- 读取文件时发现中文乱码，先确认真实编码，不要直接按乱码内容修改。
- 命令输出出现乱码时，不要把该输出作为事实写入文档或代码。
- 不确定某段内容是否乱码时，必须说明风险，不得猜测修复。
- 新增中文注释或中文文档后，必须读取目标文件确认中文显示正常。
- 如果项目有格式化、lint 或静态检查命令，应按改动影响范围运行。

## 4. 提交与发布规范

> **本节是提交/发布权限的权威来源。** 其他文档仅引用，不重复展开。

**提交信息规则：**

- 默认使用简体中文；项目已有英文提交、Conventional Commits 或其他格式约定时，优先遵循项目约定。
- 标题必须说明实际变更，避免只写 `修改`、`更新`、`fix`、`优化` 等空泛描述。
- 如果提交包含多类改动，正文应列出关键变更点。
- 如果提交与需求相关，正文应注明需求记录路径或编号（如 issue、任务卡、PR、规格文档路径等；项目若使用 OpenSpec 则注明 change 路径）。
- 如果提交包含风险或未完成验证，正文必须说明剩余风险和未验证原因。
- 推荐格式与 PR 模板字段对齐，见 `.github/pull_request_template.md`（若已复制到项目）。

**AI 在执行提交前必须：** 读取 Git 状态和 diff；确认提交信息与实际变更一致；确认关联需求与验证结果；不得为未验证的结果写“已通过”。

**提交与发布权限：**

- 未经用户明确要求，AI 不得执行 commit、push、merge、release、deploy 或其他会改变仓库历史、远端状态或线上状态的操作。
- 用户要求提交、推送、合并、发布或部署前，AI 必须先汇总当前变更、关联需求、验证结果和剩余风险。
- 如果存在未完成验证、未确认需求变化或高风险改动，提交或发布说明中必须明确写出，不得用笼统表述掩盖风险。

## 5. 规则优先级

> **本节是冲突处理的权威来源。** 需求事实、代码事实、执行方法的详细展开见 `docs/规范/` 对应文档。

判断任何冲突时，先判断是否触及不可覆盖的强制规则，再按对应维度处理。

**不可覆盖的强制规则：**

- 平台或系统安全约束、法律合规要求、权限边界。
- 发布、提交、推送、合并、部署、上线等操作（见第 4 节）。
- 需求治理底线：真实业务需求或需求变化影响行为、接口、数据、权限、流程或验收标准时，必须有可追溯记录（见速查「需求治理与影响分析（自洽）」；已复制 `docs/规范/` 时另见 [需求治理与文档结构.md](./docs/规范/需求治理与文档结构.md)）。
- 验证底线（见第 1 节）。
- 未经用户明确要求，不得擅自修改 `AGENTS.md` 或长期规范文档。

**处理「该听谁的规则」时：**

```text
0. 不可覆盖的强制规则
1. 用户本轮明确指令
2. 当前项目根目录的 AGENTS.md
3. docs/规范/ 中的项目规则
4. .agents/skills/ 中的项目级技能包规则
5. 当前运行环境提供的 AI 流程能力、插件或连接器规则
6. 通用默认习惯
```

**需求事实来源（要做什么）：** 用户本轮确认 → 可追溯需求记录（issue、任务卡、PR、外部需求系统、规格文档、轻量记录等；项目若启用 OpenSpec 则包含 change/spec）。需求事实不描述代码现状。

**代码事实来源（代码是什么样）：** 实际源码与测试输出 → 代码分析工具（可用时，不特指某一产品） → 人工搜索与阅读（须记录剩余风险）。代码事实不能覆盖需求验收标准。

**执行方法优先级（怎么推进）：** 用户点名 → 项目 `.agents/skills/`（存在时）→ 运行环境提供的 AI 流程能力（含 Superpowers 等插件，若已安装）→ 通用默认习惯。可选工具链（OpenSpec、CodeGraph、Superpowers 等）的安装、分工与任务分流见 [AI协作与工具链.md](./docs/规范/AI协作与工具链.md)（**Full 档位可选**，缺失时不影响本文件强制底线）。

如果仍无法判断，应停止执行并向用户确认，不得自行猜测。

## 规则包附带文档

本节只列出**随 Vibe Coding 规则包提供、路径固定**的规范文件，供复制时对照。**项目新增或调整规则时，只改 `docs/规范/` 内文件及 `docs/规范/规范总览.md`，不修改 `AGENTS.md`。**

按所选档位复制到业务项目；**Minimal 档位未复制时**，AI 仍须遵守本文件速查与第 1–5 节。

| 附带文档（固定） | 内容 | 何时加载 |
| --- | --- | --- |
| `docs/规范/需求治理与文档结构.md` | 目录结构、治理模式、需求变化与归档 | Standard / Full |
| `docs/规范/AI协作与工具链.md` | 可选工具链（OpenSpec、CodeGraph、Superpowers 等）分工、技能包、任务分流 | Full（可选） |
| `docs/规范/模板/轻量记录模板.md` | 轻量记录字段 | 采用轻量治理时 |

**项目特有规范**（UI、API、业务约定等）不在上表维护；复制后在 `docs/规范/规范总览.md` 自行索引，AI 执行前按该索引或文件名识别相关正文。

**规则包档位（复制时可裁剪）：**

- **Minimal：** 仅本文件（第 1–5 节 + 速查）；**不复制** `docs/规范/`。
- **Standard：** 本文件 + `docs/规范/需求治理与文档结构.md` + PR 模板与 [轻量记录模板.md](./docs/规范/模板/轻量记录模板.md)（推荐）；`docs/规范/` 中需求治理文档为**必需**。
- **Full：** 本文件 + 全部 `docs/规范/` + `.agents/skills/`；另可按项目需要安装 OpenSpec、CodeGraph、Superpowers 等可选工具（见 [AI协作与工具链.md](./docs/规范/AI协作与工具链.md)）。

## 执行前扫描顺序

1. 本文件速查与第 1–5 节。
2. 若存在 `docs/规范/`，先读 `规范总览.md` 或按文件名识别与任务相关的规范正文；规则包附带文档（见上一节）按任务类型按需加载，不批量读取无关内容。
3. 若存在 `.agents/skills/`，检查索引或技能名；发现明显相关技能时再读 `SKILL.md`。
4. 有文件修改时，完成前说明实际修改的文件和验证结果；无修改时说明未修改。
