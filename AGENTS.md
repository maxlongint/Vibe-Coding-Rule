# AGENTS.md

本文定义 Vibe Coding 的文档、需求、工程实现与 AI 协作规则。推荐采用 `AGENTS.md + CodeGraph + OpenSpec + Superpowers` 组合规范项目开发：`AGENTS.md` 定规则和边界，Superpowers 定 AI 执行方法，OpenSpec 定需求、规格、任务和验收，CodeGraph 定代码事实、调用链和影响范围。创建或修改文档前，必须先判断文档类型，并按对应目录和结构落档。

## 文档内容目录

1. 文档目录与需求记录结构
2. 需求变化与归档规则
3. 技能包目录规则
4. 工程守则与验证要求
5. 注释规范
6. 编码与乱码防护
7. 提交与发布规范
8. AI 执行规则
9. 规则优先级

## 1. 文档目录与需求记录结构

Vibe Coding 推荐默认结构：

```text
docs/
  README.md
  规范/
    规范总览.md
    工作流规范.md
    需求文档规范.md
    工程协作规范.md
openspec/
  project.md
  changes/
    <change-name>/
      proposal.md
      design.md
      tasks.md
      specs/
        <domain>/
          spec.md
  specs/
    <domain>/
      spec.md
```

需求记录规则：

-   目录结构是默认建议；项目已有明确约定时，可在根目录 `AGENTS.md` 或项目规范中声明覆盖原因和适用范围，但只能覆盖目录结构、命名方式等可适配规则。
-   不为未发生的需求、未启用的规范、未确认复用价值的案例预建空目录或空文件。
-   `docs/README.md` 只做项目入口，说明项目环境、技术栈、框架、常用命令和文档入口；不承载详细规范或具体业务需求。
-   `docs/规范/` 放跨模块、跨角色、长期有效的项目级规则。只有团队明确把当前仓库作为规则模板、规范源或跨项目规则载体时，才承载跨项目通用规则。
-   `docs/案例/` 只在用户明确确认某个案例具有培训和复用价值时再建立；AI 不得主动创建案例目录。

真实业务需求是指会改变用户能力、业务流程、接口契约、权限规则、数据模型、状态流转或验收标准的变更。需求治理的底线是可追溯：必须能确认需求来源、当前状态、关键验收标准、变化记录和剩余风险。

OpenSpec 硬触发条件：只要本次修改会改变外部可观察行为、接口契约、数据结构、权限规则、状态流转、验收标准或用户操作路径，就必须先有可追溯需求记录；中等及以上真实业务需求默认使用 OpenSpec change。

项目可采用三种治理模式：

-   OpenSpec 需求治理：真实业务需求默认使用 OpenSpec change、spec 和 tasks 承载；`AGENTS.md` 只规定何时必须使用、如何验证以及如何处理冲突。
-   外部需求系统治理：使用需求管理系统、任务系统、issue 系统等；仍应在 OpenSpec change、任务记录、变更说明、PR 描述、提交说明或交付说明中记录外部链接、编号或用户确认来源。
-   轻量记录治理：适用于原型、一次性验证、小范围内部工具等；可用任务记录、任务卡、issue、变更说明、PR 描述或交付说明承载变更范围和验收标准，但仍必须可追溯。轻量记录不替代中等和重大真实业务需求的 OpenSpec 记录。

轻量记录最小模板：

```text
变更范围：
- ...

验证：
- ...

未验证/剩余风险：
- ...

是否涉及 OpenSpec：
- 否，原因：...
```

项目文档未声明治理方式、且 AI 无法从上下文确定时：

-   只读分析、复现、影响判断和临时验证可以继续。
-   一旦准备正式修改会影响行为、接口、数据、权限、流程或验收标准的内容，应先检查或创建 OpenSpec change；如果项目尚未初始化 OpenSpec，应先向用户确认是否初始化 OpenSpec 或采用外部需求系统记录。
-   用户选择外部或轻量记录方式时按选择执行。
-   用户无法及时确认、但任务小且需要继续推进时，可先用任务记录、变更说明、PR 描述、提交说明或交付说明作为轻量记录载体，并说明原因和剩余风险；后续若范围升级，应补建 OpenSpec change。
-   不应仅因项目未声明治理方式，就默认创建自定义需求目录；真实业务需求的主记录载体统一使用 OpenSpec。

目录规则：

-   OpenSpec 是中等和重大需求变化的主记录载体；不再维护自定义的完整需求目录。
-   OpenSpec 的 spec 应描述外部可观察行为和验收标准，不应只描述内部实现。
-   OpenSpec 的 tasks 应包含实现步骤、验证方式和完成状态。
-   小修、纯文案、局部样式、内部测试补充通常不需要创建 OpenSpec change，但必须在任务记录、交付说明或 PR 描述中记录变更范围和验证结果。
-   真实业务需求或需求变化影响行为、接口、数据、权限、流程或验收标准时，必须先更新 OpenSpec 或外部需求系统记录，再修改代码；探索、复现和影响判断阶段可以先只读分析或临时验证。
-   纯缺陷修复、测试补充、内部重构、依赖修补、格式调整或文案微调，通常不创建 OpenSpec change；若改变既有业务规则、已发布行为规格、验收标准或对外契约，仍按需求变化处理。

## 2. 需求变化与归档规则

需求变化必须先确认归属和记录载体，再记录变化、判断影响、更新文档或任务记录以及实现计划。

标准动作：

```text
1. 确认变化归属：原需求变化、新需求、缺陷修复、轻量调整，或外部系统已有事项。
2. 确认记录载体：OpenSpec change，或外部需求记录、任务记录、变更说明、PR 描述、交付说明等。
3. 判断变化等级：小变化、中等变化、重大变化。
4. 记录变化：OpenSpec 治理默认更新 change 下的 proposal、design、tasks 和 specs；外部或轻量治理应在对应记录中说明变更范围、取舍原因和验收变化。
5. 更新 OpenSpec change 下的 proposal.md、design.md、tasks.md、spec.md 等相关文档；若使用外部需求系统，应同步保留外部链接、编号或确认来源。
6. 必要时更新影响分析；可用 CodeGraph 时优先使用 CodeGraph 分析调用链、依赖关系和影响范围，没有工具则记录人工影响判断。
7. 更新实现计划。
8. 更新测试和验收标准。
9. 根据变化等级重新确认，可由用户确认、负责人评审，或在记录中写明 AI/开发者自检结论和剩余风险。
10. 再继续实现。
```

变化等级：

-   小变化：非关键文案、非流程性布局调整、错误提示文案等。更新任务记录和验收标准，并重新运行相关测试。
-   中等变化：例如邮箱登录从大小写敏感改为不敏感、增加第三方 OAuth 账号绑定、新增表单校验规则。需更新设计记录、相关规格、任务记录，重新做必要影响分析和相关验证。
-   重大变化：例如认证协议、登录策略、权限模型、数据模型变化。应暂停实现，更新目标、设计和规格记录，重新做影响分析、方案评审、任务拆分，并在必要时新建 OpenSpec change。

升级规则：

-   小变化若影响操作路径、权限可见性、状态流转、验收标准、用户对操作结果的预期、数据提交意图、法律或合规含义，应按中等或重大变化处理。
-   目标仍相同、只是范围或规则变化，且仍围绕同一能力时，继续使用原 OpenSpec change。
-   目标变成另一项系统能力、原需求已完成或归档、新需求影响完全不同模块，或需要独立评审、发布和回滚时，新建 OpenSpec change。

需求归档：

-   需求状态默认使用 `进行中`、`已完成`、`已暂停`、`已废弃`、`已替代`；项目可以声明等效状态，但必须表达相同含义。
-   需求完成、暂停、废弃或被替代时，必须在 OpenSpec change 或外部需求系统中标记状态、日期和原因。
-   已完成需求应记录最终验收结果；未完成或废弃需求应记录停止原因、剩余风险和后续方向。
-   是否移动到归档目录由项目约定决定；没有归档目录时保持原目录不动，通过文档状态标记归档。
-   归档触发条件是需求已完成且通过验收，或用户/负责人明确要求归档。归档前应确认不再需要追加实现任务。
-   归档操作可由用户发起，AI 也可在交付总结中建议，但需用户确认后执行。
-   归档后原则上不继续追加实现任务；如需恢复或扩展，应先判断是恢复原 OpenSpec change 还是新建 change。目标和验收仍有效且范围相近时，可恢复原 change 并追加变更说明；目标已变或影响范围显著变化时，应新建 OpenSpec change。

## 3. 技能包目录规则

如项目启用技能包，默认放在：

```text
.agents/skills/
```

目录规则：

-   项目也可在根目录 `AGENTS.md` 或规范文档中声明其他技能包目录、入口文件或命名方式；未声明时使用默认目录。
-   每个技能包使用独立目录，例如 `.agents/skills/<skill-name>/SKILL.md` 和按需的 `references/`。
-   如果 `.agents/skills/README.md` 或 `.agents/skills/index.md` 存在，表示项目维护了技能包索引；AI 检查技能包时应先读索引，再按需读具体 `SKILL.md`。
-   `SKILL.md` 是默认入口，应说明适用场景、执行规则和需要读取的参考资料；项目使用其他入口时，应在项目规则中说明。
-   `references/` 放技能包补充资料，不要在任务开始时一次性读取全部资料。

使用规则：

-   技能包发现和检查顺序按第 8 节执行；本节只定义目录、入口和读取边界。
-   凡任务可能影响代码结构、运行行为、用户体验、数据流、外部集成、测试结果或协作规范，且 `.agents/skills/` 存在时，AI 应检查是否有相关技能包。小范围、低风险修改可只检查目录索引或技能包名称；发现明显相关技能包时，再读取对应 `SKILL.md` 和必要参考资料。
-   如果 `.agents/skills/` 不存在或没有相关技能包，继续按第 8 节执行。
-   简单问答、纯解释、纯命令查询等不涉及工程决策的任务，可以不加载技能包。
-   技能包规则与当前项目 `AGENTS.md` 冲突时，以当前项目 `AGENTS.md` 为准；与用户本轮明确指令冲突时，在不违反平台安全、项目治理和权限边界的前提下，以用户本轮指令为准。
-   不要把技能包内容复制到通用规范文档中；通用规范只规定技能包目录、发现方式和使用顺序。

## 4. 工程守则与验证要求

工程实现必须优先保证业务正确性、清晰边界、可测试性和长期维护成本。不得为了让程序“看起来能跑”而掩盖真实问题。

实现规则：

-   文件和模块必须保持聚焦，优先遵循现有架构、目录边界、命名方式和调用模式。
-   不得为了降低实现成本、缩短代码路径或绕过依赖，牺牲业务正确性、清晰分层、可测试性或长期维护成本。
-   不在未记录原因、影响范围和验证成本的情况下，引入额外服务、第三方后端、复杂基础设施或新技术栈。
-   涉及跨模块能力时，必须先明确模块边界和数据流；涉及权限、状态流转、数据一致性、审计留痕或外部集成时，不得只在调用方拼接局部操作来伪造业务成功。
-   在存在对应业务场景时，生产业务链路、真实验收链路和已接入真实逻辑的数据链路中，不得用 mock、本地缓存、seed 数据或旧链路替代真实依赖并宣称真实链路成功；测试、演示和本地开发场景可以使用模拟数据，但必须隔离环境并显式标明边界。
-   字段在业务上允许可选且有合理默认值时，该默认值应在契约文档或代码注释中显式说明，并在非生产环境可验证。
-   空状态和错误状态必须区分：业务数据为空可以展示空状态；标准字段不可读取、不可解析或缺少必要展示字段时属于错误，必须显式失败。
-   接口失败、权限失败、字段缺失、契约不完整等真实失败，必须通过合适渠道显式反馈；不得用默认值、猜测值、空对象、旧字段或“兜底成功”掩盖，也不得在缺少契约时猜测业务状态。
-   用户侧错误信息应避免泄露敏感内部细节；前端、本地或调用方不得为了避免报错而掩盖服务端、权限或数据契约错误。
-   在存在对应业务场景时，用户可见的关键写操作必须防止重复提交，并在提交期间给出清晰等待状态；写入成功后跳转或进入下一流程时，应正确等待和处理写入与跳转，不得提前隐藏失败状态。
-   在存在对应业务场景时，外部资源、上传资源、临时访问地址等必须有明确的权限、有效期和展示字段约定；不得把仅供存储或内部标识使用的字段直接当作展示地址。
-   样式或交互问题应先检查结构、状态和覆盖链路，再做视觉微调。

敏感信息与安全规则：

-   不得主动读取、打印、提交或传播密钥、token、私钥、密码、会话凭据、个人敏感信息或未脱敏的生产数据，除非用户明确要求且符合权限边界和安全目的。
-   不得把 `.env`、凭据文件、云服务密钥、数据库连接串或访问令牌写入文档、示例、测试快照、日志或提交说明。
-   需要展示日志、报错、配置或命令输出时，应先判断是否包含敏感信息；如包含，应脱敏后再引用，并说明脱敏范围。
-   发现疑似泄露的敏感信息时，应停止扩散，提示风险、受影响位置和建议处理方式，不得为了继续任务而复制到其他文件或对话上下文中。
-   不得伪造权限、凭据或安全校验结果来宣称流程成功；涉及安全策略变化时，必须记录影响范围和验证方式。

验证规则：

-   验证范围必须覆盖本次改动及其直接影响。
-   纯文案、局部样式或单处静态文档改动，可以做最小验证。
-   涉及组件结构、页面行为、表单逻辑、服务调用、权限、数据契约或外部集成时，必须扩大到直接调用方和相关业务测试。
-   能自动化验证的，应同时覆盖正向和关键反向断言；无法覆盖时说明原因和剩余风险。
-   涉及文档改动时，应同时按第 6 节编码与乱码防护要求验证中文显示。
-   执行 Node 相关验证命令前，必须先检查项目中是否存在 `.node-version`；如存在，应通过 `fnm` 使用对应版本；如不存在，使用当前默认 Node 环境。
-   选择验证命令时，应优先从 `package.json` scripts、`docs/README.md`、`README.md`、`AGENTS.md` 或项目规范中确认；仍不明确时，说明采用的最小验证方式和剩余风险。

## 5. 注释规范

注释用于解释意图、边界、约束和非显然决策，不用于重复代码表面含义。

通用规则：

-   注释语言应优先与所在文件和项目既有风格一致；中文项目默认使用简体中文，除非项目已有其他约定。
-   新增或修改包含复杂业务规则、状态流转、外部调用、错误处理、权限判断或跨模块副作用的模块，且仅靠命名和结构无法清楚表达边界时，应在文件或模块入口处保留简短说明。
-   新增或修改公共接口、跨模块调用、复杂业务规则、非显然副作用或容易误用的函数、类、组件、脚本任务、配置块时，应说明用途、关键输入输出和重要副作用。
-   业务规则、权限判断、状态流转、数据标准化、错误处理、外部集成、缓存策略、兼容策略等非显然决策点，应说明为什么这样做。
-   临时方案、技术债务、待迁移逻辑必须显式标注 `TODO:` 或 `FIXME:`，并写明背景、影响范围和后续处理方向。
-   禁止写只复述代码的空注释，例如“获取用户”“设置状态”“调用接口”。
-   修改逻辑时必须同步更新相关注释，禁止保留过期注释误导后续维护者。
-   对外暴露的接口、公共工具、共享组件和关键配置，应优先写清使用约束和失败行为。
-   复杂实现若可通过拆分函数、重命名或调整结构变清楚，应优先改结构，不要只靠长注释解释混乱代码。

适用范围：源码、配置、脚本、测试文件，以及项目约定的公共接口说明。

推荐写法：

```text
// 这里按手机号匹配 SSO 用户，是因为企业身份源不保证邮箱唯一。
// 若后续接入多身份源，需要先扩展用户绑定表再调整匹配策略。
```

## 6. 编码与乱码防护

源码、配置和文档必须稳定可读，禁止把乱码、错误编码或终端异常输出写回仓库。

编码规则：

-   源码、配置、脚本和文档统一使用 UTF-8；新建文件默认 UTF-8，不主动使用带 BOM 的编码。
-   不得将终端乱码、复制异常的中文、错误转码内容直接写回源码或文档。
-   执行环境出现编码警告或编码错误时，必须先排查文件编码问题，不得在未确认编码的情况下直接修改内容。
-   修改已有文件时，应保持原文件换行和编码风格，不做无关格式 churn。
-   跨平台路径说明应优先使用项目相对路径；涉及特定操作系统或 Shell 时，按项目本地环境说明使用对应语法。
-   文档中出现中文、英文、命令、路径、代码块时，必须保持可读和可复制。

乱码防护：

-   读取文件时发现中文乱码，先确认真实编码，不要直接按乱码内容修改。
-   命令输出出现乱码时，不要把该输出作为事实写入文档或代码；如需引用，应重新设置正确编码后获取，或用人工可确认的内容转述。
-   不确定某段内容是否乱码时，必须说明风险，不得猜测修复。
-   禁止为了“看起来正常”而删除、替换或重写不理解的中文内容。

编辑和验证：

-   对中文文档做大段改写时，应保持标题层级、列表结构和代码块完整。
-   对代码文件做编辑时，只改与任务相关的内容，不因编码或格式问题重写整个文件。
-   不要把不同编码来源的片段混合粘贴到同一个文件中。
-   新增中文注释或中文文档后，必须读取目标文件确认中文显示正常。
-   大量中文或多文件改动后，检查是否出现替换字符、问号占位、异常空格或明显断句。
-   如果项目有格式化、lint 或静态检查命令，应按改动影响范围运行。

## 7. 提交与发布规范

提交信息必须让后续维护者看懂本次提交的实际内容和原因。

提交信息规则：

-   默认使用简体中文；项目已有英文提交、Conventional Commits 或其他格式约定时，优先遵循项目约定。
-   标题必须说明实际变更，避免只写 `修改`、`更新`、`fix`、`优化` 等空泛描述。
-   如果提交包含多类改动，正文应列出关键变更点。
-   如果提交与需求相关，正文应注明 OpenSpec change 路径或外部需求记录，例如 `openspec/changes/add-sso-login/`。
-   如果提交包含需求变化，正文应注明更新了哪些 OpenSpec 文件，例如 `proposal.md`、`design.md`、`tasks.md`、`spec.md`。
-   如果提交包含风险或未完成验证，正文必须说明剩余风险和未验证原因。

推荐格式：

```text
<一句话说明本次提交改了什么>

主要变更：
- ...

关联需求：
- OpenSpec change 路径、外部需求链接或编号

验证情况：
- ...

剩余风险：
- ...
```

AI 在执行提交前必须：

-   优先读取 Git 状态和 diff；如果不是 Git 仓库，应基于实际修改文件列表整理变更摘要。
-   确认提交信息与实际变更一致。
-   确认是否有关联 OpenSpec change、外部需求链接或编号。
-   确认是否需要说明验证结果或剩余风险。
-   不得为未验证的结果写“已通过”。

提交与发布权限：

-   未经用户明确要求，AI 不得执行 commit、push、merge、release、deploy 或其他会改变仓库历史、远端状态或线上状态的操作。明确要求包括“帮我提交”“commit 一下”“push 上去”等口语化表述。
-   用户要求提交、推送、合并、发布或部署前，AI 必须先汇总当前变更、关联需求、验证结果和剩余风险。
-   如果存在未完成验证、未确认需求变化或高风险改动，提交或发布说明中必须明确写出，不得用笼统表述掩盖风险。

## 8. AI 执行规则

AI 在采用本规范的项目中应遵守：

本项目推荐使用 `AGENTS.md + CodeGraph + OpenSpec + Superpowers` 作为 AI 开发协作模型：

-   `AGENTS.md` 是项目协作章程，定义规则、权限边界、文档治理、验证底线和优先级。
-   Superpowers 是 AI 执行方法，用于需求澄清、调试、TDD、计划、审查和完成前验证。
-   OpenSpec 是需求与规格治理工具，用于需求变化、设计记录、任务拆分、验收场景和归档。
-   CodeGraph 是代码理解与影响分析工具，用于探索架构、调用链、依赖关系和重构影响。

标准执行顺序：

```text
1. 读取 AGENTS.md，确认项目规则、权限边界和验证要求。
2. 判断任务类型：问答、小修、bugfix、中等需求、重大需求、重构或紧急修复。
3. 根据任务类型启用对应 Superpowers 流程。
4. 如任务涉及行为、接口、数据、权限、流程或验收标准变化，必须先检查或创建 OpenSpec change；如项目尚未初始化 OpenSpec，应先向用户确认初始化方式或外部需求记录。
5. 如任务涉及代码修改、影响判断、调用链、重构或缺陷定位，应优先使用 CodeGraph 分析现有代码。
6. 更新 OpenSpec change 后，再进入实现；若用户明确使用外部需求系统，应同步保留外部链接、编号或确认来源。
7. 完成后运行必要验证，并说明验证结果和剩余风险。
```

工具使用边界：

-   OpenSpec 定义“要做什么、为什么做、做到什么程度算完成”；不替代代码分析和测试。
-   CodeGraph 定义“代码现在真实是什么样、改动影响哪里”；不替代需求确认和业务验收。
-   Superpowers 定义“AI 如何工作”；不替代项目规则、需求记录、代码事实和验证结果。
-   当 OpenSpec、CodeGraph 和实现现状冲突时，应先判断是需求变更、代码偏离规格，还是分析范围不足；不得直接选择最省事的实现路径。

事实和方法边界：

-   需求事实优先从用户确认、OpenSpec 和外部需求系统获得。
-   代码事实优先从 CodeGraph、测试结果和实际源码获得。
-   执行方法优先使用 Superpowers；Superpowers 只规定怎么推进，不改变需求事实和代码事实。

任务分流表：

| 任务类型 | OpenSpec | CodeGraph | Superpowers | 备注 |
| --- | --- | --- | --- | --- |
| 只读分析、解释、命令查询 | 不需要 | 可选 | 可选 | 不改文件时说明未修改 |
| 小修、文案、局部样式 | 通常不需要 | 视影响范围 | verification-before-completion | 用轻量记录说明范围和验证 |
| 缺陷修复 | 视是否改变业务规则 | 必须用于定位或影响分析 | systematic-debugging、test-driven-development、verification-before-completion | 若改变验收标准或对外行为，补 OpenSpec |
| 新功能或中等需求 | 必须 | 必须 | brainstorming、writing-plans、test-driven-development、verification-before-completion | 先规格后实现 |
| 重大需求、权限、数据、接口变更 | 必须 | 必须 | brainstorming、writing-plans、test-driven-development、requesting-code-review、verification-before-completion | 需要记录方案、风险和验收 |
| 重构 | 视行为是否变化 | 必须 | writing-plans、verification-before-completion | 重构前记录影响范围；若行为变化按需求变化处理 |
| 紧急修复 | 可先轻量记录，事后补齐 | 必须覆盖故障链路 | systematic-debugging、verification-before-completion | 不得跳过验证底线 |

三件套最低动作标准：

-   使用 OpenSpec 时，至少明确 change 路径、目标、边界、验收标准、任务状态和验证方式；归档或废弃时记录状态、日期、原因和剩余风险。
-   使用 CodeGraph 时，理解模块优先用 `codegraph_explore`；查调用关系用 callers/callees；重构或跨模块修改前用 impact；结论应写入设计记录、任务记录或交付说明。CodeGraph 不可用时，说明改用人工搜索并记录剩余风险。
-   使用 Superpowers 时，按任务类型启用对应流程；出现 bug、测试失败或异常结果时优先使用 systematic-debugging；完成前必须使用 verification-before-completion 的原则核对验证证据。
-   三件套结论不一致时，先暂停实现并判断冲突来源：需求变化、规格过期、代码偏离规格、测试覆盖不足或分析范围不足；必要时更新 OpenSpec 或向用户确认。

-   未经用户明确要求修改 `AGENTS.md` 或规则文档时，不得擅自修改；只能提出建议或说明建议改动点。
-   修改 `AGENTS.md` 或长期规范时，应说明变更原因、影响范围；项目有 changelog、需求记录或规范变更记录时，应同步记录。若已有 `docs/规范/`，优先在规范总览、规范 changelog 或相关规范文档中记录；若没有专门记录文件，应在提交信息或交付说明中写明。
-   `AGENTS.md` 应保持为协作入口和强制规则索引；细节模板、长示例、FAQ 或工具专项教程应优先沉淀到 `docs/规范/`，再由 `AGENTS.md` 引用，避免主规则文件无限膨胀。
-   执行 Vibe Coding 或工程任务前，先扫描 `docs/规范/` 的目录索引或文件名，识别与当前任务相关的规范；仅加载明显相关的规范正文，不批量读取无关内容。如果 `docs/规范/` 不存在或无相关规范，可在执行说明中简短记录判断后继续，无需新建规范文件。
-   扫描规范后，再检查 `.agents/skills/` 是否有相关技能包。小范围、低风险修改可以只检查目录索引或技能包名称；发现明显相关技能包时，再读取对应 `SKILL.md` 和必要参考资料。
-   如果 `docs/规范/` 和 `.agents/skills/` 都存在相关规则，按第 9 节优先级处理，技能包规则仅作为项目规范补充。两者均不存在或均无匹配时，按当前项目根目录 `AGENTS.md` 和通用规范执行。
-   修改文档前先判断文档性质：通用规则、真实业务需求、教学案例。
-   用户明确要求沉淀或修改通用规则时，写入 `docs/规范/` 或对应规则文档。
-   用户明确要求沉淀或修改真实业务需求时，默认写入 OpenSpec change；若用户明确使用外部需求系统，应在 OpenSpec、任务记录、变更说明、PR 描述、提交说明或交付说明中保留外部链接或编号。
-   用户明确要求沉淀教学案例时，写入 `docs/案例/`；否则不得主动创建案例目录。
-   需求变化时，按第 1、2 节先更新 OpenSpec change，再更新实现计划，再改代码；若用户明确使用外部需求系统，应同步保留外部链接、编号或确认来源。探索、复现、影响判断阶段可以先只读分析或临时验证。
-   不为具体 AI 工具、插件或连接器单独建立长期规范，除非用户明确要求。
-   如有文件修改，完成前必须说明实际修改的文件和验证结果；无文件修改时说明未修改。

## 9. 规则优先级

规则优先级用于处理用户指令、项目规则、OpenSpec、CodeGraph、技能包、AI 流程能力、插件、连接器和通用习惯之间的冲突。判断冲突时，先判断是否触及不可覆盖的强制规则，再判断是否属于项目可适配规则。

不可覆盖的强制规则：

-   平台或系统安全约束、法律合规要求、权限边界不可被任何项目规则、用户指令、技能包、AI 流程能力、插件或连接器规则覆盖。
-   发布、提交、推送、合并、部署、上线等会改变仓库历史、远端状态或线上状态的操作，必须遵守第 7 节权限规则；未经用户明确要求不得执行。
-   需求治理底线不可被跳过。真实业务需求或需求变化影响行为、接口、数据、权限、流程或验收标准时，必须有可追溯记录；OpenSpec 治理按第 1、2 节更新 change、spec 和 tasks，外部或轻量治理必须在对应记录中保留外部链接、编号或用户确认来源。
-   验证底线不可被跳过。完成实现或文档修改前，必须说明验证结果；无法验证时必须说明原因和剩余风险，不得把未验证结果表述为已通过。
-   规则文档和长期规范的修改权限不可被弱化。未经用户明确要求，不得擅自修改 `AGENTS.md` 或长期规范文档；用户要求修改时，应说明变更原因、影响范围和验证情况。

可适配规则：

-   目录结构、命名方式、流程承载位置、具体文档模板、技术栈命令、工具使用方式、技能包入口等，属于可适配规则。
-   项目根目录 `AGENTS.md` 或 `docs/规范/` 可以在明确说明适用范围和覆盖原因时，覆盖通用默认做法，但不能覆盖安全、权限、发布、需求治理、验证和规则文档权限。
-   Superpowers、AI 流程能力、插件、连接器和项目内技能包只提供执行方法，不改变本文规则优先级。CodeGraph 提供代码事实和影响分析，OpenSpec 提供已确认需求事实，两者都不得绕过安全、权限、需求治理和验证底线。

在不触及不可覆盖强制规则的前提下，按以下优先级执行。若较低层级文档是在较高层级规则允许范围内声明的项目适配规则，则该适配规则可覆盖对应的通用默认做法：

```text
0. 不可覆盖的强制规则：平台安全约束、法律合规要求、权限边界，以及发布权限、需求治理、验证底线和规则文档权限
1. 用户本轮明确指令
2. 当前项目根目录的 AGENTS.md
3. OpenSpec 中已确认的需求、规格和验收标准
4. docs/规范/ 中的项目规则
5. .agents/skills/ 中的技能包和 Superpowers 流程
6. CodeGraph 代码事实和影响分析结论
7. 当前运行环境提供的 AI 流程能力、插件或连接器规则
8. 通用默认习惯
```

如果仍无法判断，应停止执行并向用户确认，不得自行猜测。
