# AGENTS.md

> Vibe Coding Rule 版本：v2.1.1

## 速查：六条底线

1. **不伪造成功**：真实失败必须显式暴露，不用默认值、空对象、mock、旧链路或跳过检查掩盖。
2. **完成前必须验证**：说明实际结果；无法验证时写明原因、未覆盖项和剩余风险。
3. **改行为前先记录**：改变外部行为、接口、数据、权限、流程或验收前，先更新唯一需求载体。
4. **不碰敏感信息**：不主动读取、打印、提交或传播密钥、凭据、个人敏感信息或未脱敏生产数据；确需处理时须有明确授权并限于最小必要范围。
5. **不擅自制造外部副作用**：未经用户明确要求，不提交、发布、删除、迁移、写入生产数据或执行其他不可逆、付费或对外操作。
6. **拿不准就问**：需求证据冲突，或请求可能改变既有需求、验收、接口、权限、流程且边界不清时，先说明事实、影响和未知项，再请求确认。

## 1. 适用范围与优先级

-   外部指令优先级遵循当前运行环境；本文件不能覆盖平台、法律、权限安全约束或用户本轮明确指令。
-   项目内部发生冲突时，按“用户已确认的需求事实 → 当前唯一需求载体 → 适用的项目规范 → 工具或 skill 默认行为”处理；仍无法判断时停止并询问。
-   用户及已确认记录决定需求事实；源码、测试、版本历史、现有行为和代码分析结果只证明代码事实，不能单独定义需求或验收。
-   多级目录存在多个 `AGENTS.md` 时，按当前运行环境的作用域规则执行，并优先遵守离目标文件最近且适用的项目规则；无法确定作用域时先确认。
-   修改本规范时，现有条文是审阅对象，不得形成阻止其被用户明确修订的循环门禁；敏感信息、权限安全、外部副作用和完成前验证约束仍然有效。

## 2. 内容归属与上下文

-   **通用协作底线 → `AGENTS.md`**：只保留跨项目适用、需要始终生效的规则；未经用户明确要求不得修改。
-   **项目规范 → `docs/`**：以 `docs/README.md` 为索引，保存长期有效、跨需求适用的项目知识；新增或调整规范时同步维护索引。
-   **进行中需求 → 唯一需求载体**：使用项目已确认或用户指定的 change、issue、PR、任务卡、文档或当前任务；同一需求不得维护平行口径。当前任务只适合无需跨 Chat 的轻量工作；需要跨 Chat 继续时，先迁移到后续 Chat 可重新读取的持久载体。
-   **项目级 skill → `.agents/`**：先建立 `.agents/README.md` 索引，再按需创建 `.agents/skills/<skill-name>/SKILL.md`；个人全局 skill 不属于项目规范。
-   项目存在 `docs/README.md` 且当前任务需要项目特有上下文时，先读取该索引，再只加载直接相关的文档。索引或文档缺失、不可读或互相冲突，只在当前任务确实依赖其内容时构成阻塞；不得猜测缺失内容，也不得为扩大上下文而批量读取。
-   具体工具的职责、调用、失败处理和限制见 [`docs/规范/工具协作.md`](docs/规范/工具协作.md)，不在本文件重复。只有用户明确要求，或当前任务已经确认采用某项工具时，该工具才视为启用；工具文档、索引、目录、配置或命令的存在都不能自行启用工具。工具规范只约束已启用工具的使用方式。非必需工具未启用不得阻塞任务，已确认必需的工具不可用时不得伪装或擅自降级。

## 3. 正确性与安全

-   优先保证业务正确性、边界清晰和可测试性；遵循项目既有架构、目录、命名与调用方式。引入依赖、服务、技术栈或绕过既有依赖前，说明原因、影响和验证成本。
-   生产和真实业务链路不得用 mock、本地缓存、seed、旧字段或旧链路冒充真实依赖；测试、演示和本地开发的模拟数据必须隔离并标明边界。
-   只有业务上确实无数据时才能返回空；不可读、不可解析、契约缺失、接口失败或权限失败都属于错误，不得用默认值、猜测值、空对象或默认成功掩盖。
-   面向用户的错误信息不得泄露内部敏感细节。展示日志、报错、配置或命令输出前先检查敏感信息；发现疑似泄露时停止扩散，只说明风险和位置，不复制敏感内容。
-   不把 `.env`、凭据、连接串或令牌写入源码、文档、示例、测试快照、日志或提交说明；不伪造权限、凭据或安全校验结果。
-   可能重试、跨系统或影响真实业务的关键写操作应采用幂等或等效防重机制，并在确认写入结果后再继续依赖该结果的流程。

## 4. 需求与实施边界

-   改变外部可观察行为、接口契约、数据结构、权限、状态流转、验收标准或用户路径前，先更新唯一需求载体；只读分析可以保留在当前答复中。
-   新需求或需求修改先核对直接相关的现有记录。若请求可能改变已有口径但归属不清，先说明需求证据、请求目标、影响和未知项，再确认更新原记录还是建立关联记录。
-   跨 Chat 或上下文压缩后需要继续的已确认结论必须先进入后续 Chat 可重新读取的唯一持久载体；原载体仅为当前任务且无法重新读取时，切换前先迁移到 issue、PR、任务卡、用户指定文档或已启用的 change。不得依赖未落盘的对话记忆继续实施。
-   修改前区分**目标变更**与**非目标不变式**。未要求改变的行为、接口、数据、权限、错误处理、性能、布局、调用方式和兼容性均视为不变式。
-   只改变实现目标所必需的内容，不通过放宽校验、吞错、返回默认成功、绕过权限、扩大默认权限、削弱类型约束或破坏兼容性制造局部效果。
-   必须改变非目标不变式时，先说明原因、影响范围和验证方式，等待用户确认后再实施。
-   保护用户已有工作：修改前先识别工作区和目标文件中的已有改动，包括与当前任务相关、尚未提交或与目标行重叠的内容；不得擅自覆盖、回退、删除或格式化。无法安全合并或隔离时停止并说明。
-   用最少代码解决当前问题，不为一次性或假想需求增加抽象与配置，不顺手重构或清理既有死代码。
-   注释解释意图、边界和非显然副作用，不复述代码；临时方案或技术债务用 `TODO:` / `FIXME:` 标明背景和影响。
-   源码、配置和文档默认 UTF-8 无 BOM；中文项目新增内容默认简体中文，写入后读取确认无乱码或错误转码。

## 5. 验证

-   验证覆盖本次改动及直接影响；涉及结构、交互、输入校验、服务、权限、数据契约或外部集成时，扩展到直接调用方和关键反向路径。纯文案或局部呈现可以做最小验证。
-   优先使用项目已有的测试、类型检查、lint、构建或手工验收命令；项目固定运行时或工具链版本时先按其约定切换。
-   不得通过跳过、删除或弱化测试，缩小必要验证范围，或仅修改断言和期望值来制造通过结果。
-   完成前重新运行能够证明结论的验证，读取完整结果。失败、无法执行或未覆盖的部分必须显式说明，不得把部分验证外推为整体通过。
-   完成后说明实际改变、已核对的直接相关不变式、验证证据、未验证项和剩余风险。

## 6. 外部副作用与交付

-   删除、覆盖、数据迁移、生产写入、权限变更、外部通信、付费操作或其他难以恢复的动作前，确认准确目标、影响范围、授权和可恢复方式；范围不清时停止。
-   未经用户明确要求，不得 commit、push、merge、release、deploy、归档需求记录或执行其他改变仓库历史、远端及线上状态的操作。
-   用户要求提交或发布前，先汇总当前变更、关联需求、验证结果和剩余风险；验证不完整或存在高风险时必须明确披露。
-   提交信息遵循项目约定；无约定时默认使用简体中文，标题描述实际结果，避免“修改”“更新”“fix”等空泛措辞，并按需注明需求记录。
