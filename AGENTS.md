# AGENTS.md

> Vibe Coding Rule 版本：v3.2.2

## 速查：六条底线

1. **不伪造成功**：真实失败必须显式暴露，不用默认值、空对象、mock、旧链路或跳过检查掩盖。隔离 mock 仅可用于已标明边界的测试、演示或约定的 UI 阶段 1，不得冒充真实业务依赖或掩盖失败。
2. **完成前必须验证**：必须说明实际结果；未完成必要验证时不得声称完成或通过，并必须写明原因、未覆盖项和剩余风险。
3. **改行为前先记录**：用户明确确认新增或变更需求后，改变外部行为、接口、数据、权限、流程或验收前，必须先更新用户确认的唯一需求载体；未更新前不得实施。
4. **不碰敏感信息**：不主动读取、打印、提交或传播密钥、凭据、个人敏感信息或未脱敏生产数据；确需处理时须有明确授权并限于最小必要范围。
5. **不擅自制造外部副作用**：未经用户明确要求，不提交、发布、删除、迁移、写入生产数据或执行其他不可逆、付费或对外操作。
6. **拿不准就问**：需求证据冲突，或请求可能改变既有需求、验收、接口、权限、流程且边界不清时，必须停止实施，说明事实、影响和未知项，并取得用户确认后才能继续。

## 1. 适用范围与优先级

-   外部指令优先级遵循当前运行环境；本文件不能覆盖平台、法律、权限安全约束或用户本轮明确指令。
-   项目内部发生冲突时，按“用户已确认的需求事实 → 当前唯一需求载体 → 适用的项目规范 → 工具或 skill 默认行为”处理；仍无法判断时停止并询问。
-   用户及已确认记录决定需求事实；源码、测试、版本历史、现有行为和代码分析结果只证明代码事实，不能单独定义需求或验收。
-   多级目录存在多个 `AGENTS.md` 时，按当前运行环境的作用域规则执行，并优先遵守离目标文件最近且适用的项目规则；无法确定作用域时先确认。
-   修改本规范时，现有条文是审阅对象，不得将其作为阻止修订的前置门禁，不得因此强制创建 OpenSpec change 或执行正在审阅的工具流程；敏感信息、权限安全、外部副作用和完成前验证约束仍然有效。

## 2. 内容归属、需求确认与上下文

-   **AI 常驻规则 → `AGENTS.md`**：保存跨项目适用、每次 Chat 都需要知道、在读取索引前必须生效，以及用于判断是否读取 `docs/` 或是否允许继续的规则；未经用户明确要求不得修改。
-   **项目规范 → `docs/`**：以 `docs/README.md` 为索引，保存长期有效、跨需求适用且仅在特定任务中需要的项目知识与详细流程。不得把 AI 每次 Chat 都需要知道的规则只写在 `docs/`。新增或调整规范时必须同步维护索引，索引未同步不得视为完成。
-   **用户已确认的进行中需求 → OpenSpec change**：OpenSpec 只由用户明确确认的新增需求或需求变更触发；同一需求后续的设计、任务、实现、验证和归档继续使用同一个 change。issue、PR、任务卡、设计图与外部文档只作输入来源，不得替代该 change，也不得维护平行口径。
-   **项目级 skill → `.agents/`**：先建立 `.agents/README.md` 索引，再按需创建 `.agents/skills/<skill-name>/SKILL.md`；个人全局 skill 不属于项目规范。
-   **用户提供内容不得擅自落盘**：用户提供的经验、素材、偏好或讨论结论，除非用户明确要求保存并指定或确认归属，否则不得写入 `docs/`、`AGENTS.md`、`.agents/`、OpenSpec 或其他仓库文件；归属不明确时必须先询问。
-   当前任务需要项目特有知识，或本文件明确要求读取项目规范时，必须先读取 `docs/README.md`，再只加载与任务直接相关的文档。索引或文档缺失、不可读或互相冲突，只在当前任务确实依赖其内容时构成阻塞；不得猜测缺失内容，也不得为扩大上下文而批量读取。

### 2.1 新需求、需求变更与用户确认

-   AI 必须区分新增需求、需求变更、已有需求的继续工作，以及普通讨论、澄清或只读分析。改变已确认的目标、范围、用户可观察行为、接口、数据、权限、状态流转、用户路径、业务流程，或改变“什么才算完成”的验收标准，属于需求变更；只把已有且已确认的口径写得更清楚而不改变语义，不属于需求变更。
-   **缺陷修复不按技术类别判断是否属于需求变更。** 有用户已确认的需求记录、验收标准或现行契约能够证明当前行为违背既有口径时，修复属于恢复既有行为，不因涉及样式、布局、交互、接口、数据或业务逻辑而自动触发 OpenSpec；若修复将重新定义目标、范围、用户可观察行为、接口、数据、权限、状态流转、流程或验收，则属于需求变更，必须按本节确认并使用 OpenSpec；证据不足或存在冲突时，必须先说明判断依据与影响并取得用户确认，不得以代码类别或当前实现推定需求。
-   AI 判断用户内容可能构成新增需求或需求变更时，必须先说明判断依据和可能影响，并请求用户明确确认是否作为正式需求处理；不得在取得该确认前查找 OpenSpec 候选或进入 OpenSpec 流程。AI 的判断不构成执行授权。
-   新增需求与需求变更采用不同的写入授权。新增需求在用户确认需求意图及目标、范围和验收口径后，由用户明确要求创建 OpenSpec change；该请求授权 AI 生成 `change-id` 并执行创建前冲突检查，无需用户另行确认 AI 生成的 ID。需求变更或已有需求的后续工作仍须由用户明确指定已有 `change-id`，或确认继续原 change / 新建关联 change 的承接方向；选择新建关联 change 时，用户在新口径确认后明确要求创建，即授权 AI 生成 ID、检查冲突并创建，无需另行确认 ID。讨论、提问、设想或倾向性表述不构成任何授权。
-   新增需求收到明确创建请求后，AI 先根据已确认口径生成简短、明确的拟用 `change-id`，再只读检查现有 change；发现同名 change，或发现已有 change 可能承载该需求时，必须停止并列出冲突项、状态和关系，不得改用其他 ID 或创建 change。没有冲突时，才能使用生成的 ID 创建 change。
-   需求变更或已有需求的后续工作在承接方向确认前，只能继续澄清并只读查找候选记录；必须展示全部候选 change、状态、与本次请求的关系及建议。即使只有一个明显匹配的 change，也必须等待用户确认，不能擅自选定；存在多个候选或证据冲突时必须停止并请求选择。选择新建关联 change 时，必须先确认新口径；用户主动要求创建后，AI 生成拟用 ID 并检查现有 change，发现同名或可能承载该变更的 change 时停止，否则直接创建。
-   新增需求或新建关联 change 未取得明确创建授权，或需求变更及已有需求未确认继续使用的现有载体前，不得创建、选定或更新 OpenSpec change，不得把讨论内容落成正式需求，也不得形成最终设计、计划或开始实施。
-   普通问答、解释、只读分析、代码理解、仓库状态检查、未确认的需求讨论，以及不改变需求口径的内部整理，默认不使用 OpenSpec。本规范自身的审阅和修订按 §1 的例外处理。

### 2.2 工具与跨 Chat 路由

-   只有用户明确确认进入新增需求、需求变更或指定已有需求的后续流程后，才按 `docs/README.md` 索引读取 [`docs/规范/工具协作.md`](docs/规范/工具协作.md) 并使用 OpenSpec；不得在每个 Chat 开始时预检或创建 change。
-   接入本规范包后，默认假定 OpenSpec、CodeGraph、Superpowers 均已成功安装。只有当前任务确实需要某项工具时才检查其可用性；需要使用却发现未安装、未初始化或不可用时，必须中断相关流程并明确指出工具和阻塞点，不得伪装可用或静默降级。不依赖该工具的任务不受阻塞。
-   skill 只提供方法，不提供创建、选定或更新需求载体的用户授权。OpenSpec 流程启用后，change 优先于 skill 的默认落盘路径和提交流程；不得创建 `docs/superpowers/` 等平行口径，不得因 skill 默认行为自动提交。
-   跨 Chat 或上下文压缩后继续已有需求时，用户必须明确当前 `change-id`，或确认 AI 只读查找后列出的候选。确认前不得自动恢复、选定或修改 change；确认后按工具协作规范重新读取该 change 的必要制品。

## 3. 正确性与安全

-   优先保证业务正确性、边界清晰和可测试性；遵循项目既有架构、目录、命名与调用方式。引入依赖、服务、技术栈或绕过既有依赖前，说明原因、影响和验证成本。
-   生产和真实业务链路不得用 mock、本地缓存、seed、旧字段或旧链路冒充真实依赖；测试、演示和本地开发的模拟数据必须隔离并标明边界。
-   只有业务上确实无数据时才能返回空；不可读、不可解析、契约缺失、接口失败或权限失败都属于错误，不得用默认值、猜测值、空对象或默认成功掩盖。
-   面向用户的错误信息不得泄露内部敏感细节。展示日志、报错、配置或命令输出前先检查敏感信息；发现疑似泄露时停止扩散，只说明风险和位置，不复制敏感内容。
-   不把 `.env`、凭据、连接串或令牌写入源码、文档、示例、测试快照、日志或提交说明；不伪造权限、凭据或安全校验结果。
-   可能重试、跨系统或影响真实业务的关键写操作必须采用幂等或等效防重机制；必须确认写入结果后才能继续依赖该结果的流程。

## 4. 需求与实施边界

-   修复与需求变更的判断适用 §2.1 的通用规则，不得以技术类别替代语义判断。
-   用户明确确认新增或变更需求后，改变外部可观察行为、接口契约、数据结构、权限、状态流转、验收标准或用户路径前，必须先更新经用户确认的唯一需求载体；未更新前不得实施。只读分析可以保留在当前答复中。
-   用户确认需求意图后可以继续澄清。新增需求须先确认目标、范围和验收口径，再由用户明确请求创建；AI 此时只读执行冲突检查，无冲突才创建。需求变更或已有需求的后续工作可以只读核对直接相关记录；继续现有 change 时须确认该载体，新建关联 change 时须确认新口径并由用户明确请求创建。上述写入授权或载体确认完成前，不得形成最终设计、计划或开始实施。
-   跨 Chat 或上下文压缩后需要继续的已确认结论，必须在用户确认当前 change 后写入或更新该 OpenSpec change，供后续 Chat 重新读取。不得依赖未落盘的对话记忆继续实施。
-   修改前必须区分**目标变更**与**非目标不变式**。未要求改变的行为、接口、数据、权限、错误处理、性能、布局、调用方式和兼容性均视为不变式。
-   只改变实现目标所必需的内容，不通过放宽校验、吞错、返回默认成功、绕过权限、扩大默认权限、削弱类型约束或破坏兼容性制造局部效果。
-   必须改变非目标不变式时，必须先说明原因、影响范围和验证方式；未经用户确认不得实施。
-   保护用户已有工作：修改前必须先识别工作区和目标文件中的已有改动，包括与当前任务相关、尚未提交或与目标行重叠的内容；不得擅自覆盖、回退、删除或格式化。无法安全合并或隔离时必须停止并说明。
-   用最少代码解决当前问题，不为一次性或假想需求增加抽象与配置，不顺手重构或清理既有死代码。
-   注释解释意图、边界和非显然副作用，不复述代码；临时方案或技术债务用 `TODO:` / `FIXME:` 标明背景和影响。
-   源码、配置和文档默认 UTF-8 无 BOM；中文项目新增内容默认简体中文，写入后读取确认无乱码或错误转码。

## 5. 验证

-   验证必须覆盖本次改动及直接影响；涉及结构、交互、输入校验、服务、权限、数据契约或外部集成时，必须扩展到直接调用方和关键反向路径。纯文案或局部呈现可以做最小验证。
-   优先使用项目已有的测试、类型检查、lint、构建或手工验收命令；项目固定运行时或工具链版本时先按其约定切换。
-   不得通过跳过、删除或弱化测试，缩小必要验证范围，或仅修改断言和期望值来制造通过结果。
-   完成前重新运行能够证明结论的验证，读取完整结果。失败、无法执行或未覆盖的部分必须显式说明，不得把部分验证外推为整体通过。
-   完成后必须说明实际改变、已核对的直接相关不变式、验证证据、未验证项和剩余风险。

## 6. 外部副作用与交付

-   删除、覆盖、数据迁移、生产写入、权限变更、外部通信、付费操作或其他难以恢复的动作前，必须确认准确目标、影响范围、授权和可恢复方式；任一项不清楚时不得执行。
-   未经用户明确要求，不得 commit、push、merge、release、deploy、归档需求记录或执行其他改变仓库历史、远端及线上状态的操作。
-   用户要求提交或发布前，必须先汇总当前变更、关联需求、验证结果和剩余风险；验证不完整或存在高风险时必须明确披露。
-   提交信息遵循项目约定；无约定时默认使用简体中文，标题描述实际结果，避免“修改”“更新”“fix”等空泛措辞，并按需注明需求记录。
