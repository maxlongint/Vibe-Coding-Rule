# 需求变更

> 这是一份面向开发人员的需求变更操作指引。仅在用户已明确选择 OpenSpec 后使用以下主流程；未选择时见文末「未启用 OpenSpec 时」。强制底线见 [`AGENTS.md`](./AGENTS.md)，工具门禁见 [`docs/规范/可选工具协作.md`](./docs/规范/可选工具协作.md)；所有操作都在同一个 AI Chat 中完成。从零开发新需求见 [`新需求开发.md`](./新需求开发.md)。
>
> **核心原则：不急着新建——先定位原记录、判断变化等级、把正文更新为当前口径、追加变化记录（不覆盖历史），再改代码。**

开始前确认项目已经完成 OpenSpec 初始化，且 change 目录结构可用。具体如何定位、创建、更新、实现、验证和归档 change，由 AI 根据当前环境与项目约定处理，提示词只需说明目标、输入、边界和期望结果。若 OpenSpec 整体未安装或未初始化，按工具协作指南暂停，请用户选择安装 / 初始化或改用轻量载体。

需求变更走六个步骤：

```text
定位原记录 → 澄清新口径 → 更新 change 并追加变化记录 → 实现 → 验证 → 归档
```

先由 AI 根据变更描述定位原记录和对应的 `change-id`。若需新建 change 承接变更，则由 AI 根据变更内容生成简短、明确的小写英文 ID，使用短横线分隔，检查是否与现有 change 重名，并在文档中注明与原 change 的关系。AI 应返回最终确定的 ID；后续步骤在同一对话中直接引用“本次变更所归属的 change”，无需用户重复填写。

**变更涉及 UI 设计图时**采用条件式两阶段实现：主流程不变，在第 3 步更新 `design` 与 `tasks.md` 时按两阶段拆分，第 4 步分两轮执行（先 UI + mock，**人工验收 UI** 后再接真实逻辑）。

## 1. 定位原记录

来了一个变更，先找到它对应的原 change 或外部需求记录，**别上来就新建**。确认这真的是「已有需求的变化」，而不是全新需求、缺陷修复或随口讨论。

在 AI Chat 中发送：

```text
我收到一项变更：<变更描述>。

请先了解与这项变更直接相关的项目现状，定位对应的原 OpenSpec change 或外部需求记录，并核对原始需求来源。

请说明：原记录位置、change-id、当前有效口径、change 当前状态（进行中 / 已完成未归档 / 已归档），以及应继续用原 change 还是新建 change 承接。如果原始来源不可读、OpenSpec 未初始化或无法可靠定位，请明确说明原因，不要猜测归属。先不要创建或更新 change，也不要改代码。
```

判断口诀：

- **继续用原 change**——目标没变、还是同一项能力，只是细化或调整。例如「密码最短长度从 8 改到 12」，仍是同一条密码策略。
- **新建 change**——目标变成了另一件事、影响完全不同的模块、或需要独立发布和回滚。例如原本只做「密码策略」，现在要追加「登录失败锁定账号」，这是另一项能力。已归档的 change 原则上新开 change 承接，并注明与旧 change 的关系。

再判断变化等级：

| 等级 | 典型例子 | 处理力度 |
| --- | --- | --- |
| **小变化** | 文案微调、不影响流程的展示调整 | 可简化流程，第 2 步可跳过 |
| **中等变化** | 校验规则调整、新增一种 OAuth 绑定方式 | 走澄清 + 影响分析 |
| **重大变化** | 认证协议更换、权限模型调整、默认登录策略变化 | 完整走澄清 + 影响分析 + 回归 |

注意：看着像「小变化」，但只要它影响了操作路径、权限可见性、状态流转、验收标准、用户对结果的预期、数据提交意图或合规含义，就按中等/重大处理。

若决定**新建 change**，先发送：

```text
请根据本次变更及已定位的原记录，新建一个 OpenSpec change，并写明它与原 change 的关系。

请先读取并对照本次变更与原记录，根据变更内容自动生成简短、明确的小写英文 change-id，使用短横线分隔，并确认不与现有 change 重名。然后创建 change，起草 proposal、specs、design 和 tasks，使新口径、影响范围、验收标准、方案、任务以及与原 change 的关系保持一致。

如果原记录不可读、OpenSpec 未初始化或创建失败，请明确说明原因，不要伪装完成。成功后返回最终使用的 change-id，并说明创建了哪些内容；先不要实现。
```

确认归属、状态与继续/新建决定无误后，再进入下一步。

## 2. 澄清新口径

中等/重大变化必做，小变化可跳过。让 AI 把「原来是怎样、现在要变成怎样、差异在哪、影响哪些范围、验收怎么跟着变」理清楚。这一步只澄清，不更新 change，也不实现。

在 AI Chat 中发送：

```text
请澄清本次变更所归属的 OpenSpec change 的新口径。

请先读取并对照原始需求来源和该 change 的当前内容，再确认：原口径 → 新口径的差异、影响范围、非目标、验收标准的变化和待决问题。先不要更新 change，也不要实现。
```

与 AI 对话，直到你确认新口径已经清楚，再进入下一步。

## 3. 更新 change 并追加变化记录

让 AI 把受影响的文档改成**当前有效口径**，并**追加**一条变化记录——保留原口径，**绝不覆盖历史**。按变化类型决定改哪些文件：

| 变化类型 | 要改的文件 |
| --- | --- |
| 补充说明 | `proposal` |
| 范围变化 | `proposal` + `specs` |
| 验收调整 | `specs` + `tasks` |
| 方案调整 | `design` + `tasks`（可能含 `specs`） |

在 AI Chat 中发送：

```text
请将本次变更所归属的 OpenSpec change 正文更新为当前口径，并追加一条变化记录（不覆盖历史）。

中等/重大变化请重新分析直接影响的模块、调用方、数据流和测试面，并把结论、证据与未覆盖范围写进 `design` / `tasks`。

变化记录须包含：时间、来源、变化类型、原口径、新口径、影响范围、验收变化、实现影响、验证调整、剩余风险。记录位置：change 目录的变化记录区、`proposal.md` 末尾「变化记录」章节，或外部需求系统 / PR 描述。

同步 `tasks.md` 与验收标准——不能拿旧测试覆盖新验收。先不要实现。
```

**变更涉及 UI 设计图时**，再发送一条补充提示词：

```text
本次变更涉及 UI 设计图，请补充并更新本次变更所归属的 OpenSpec change 的 design 与 tasks.md。

UI 设计图位置：<本地路径 / Figma·蓝湖等链接 / 附件说明>
请读取并对照设计图，在 design 中记录设计图来源、页面清单与 mock 隔离方式；在 tasks.md 中拆成两阶段，每阶段各自写清验收标准：
- 阶段 1：UI + 隔离 mock + 页面跳转
- 阶段 2：真实接口、数据、权限、状态流转和错误处理

先不要实现。
```

检查正文、变化记录、任务拆分与验收标准是否准确。有 UI 设计图时，确认 AI 已读取你提供的设计图位置，且 `tasks.md` 已按两阶段拆分。确认无误后，再进入实现。

## 4. 实现

正文、变化记录与任务全部同步后，再让 AI 按 change 中的任务清单完成开发，并同步任务状态。实现途中又冒出新变化，回到第 2 步重新判断，**别直接把变化吸进代码**。

**无 UI 设计图**，在 AI Chat 中发送：

```text
请实现本次变更所归属的 OpenSpec change。

请先读取并对照该 change 的 proposal、specs、design、tasks 和变化记录，再按 tasks 逐项实现，不要改变未要求调整的行为、接口、数据、权限和流程。

完成后更新任务状态，并说明实际改动、已经执行的检查、尚未验证的内容和剩余风险。不要归档。
```

**有 UI 设计图**——分两轮执行；阶段 1 由 AI 实现，**UI 由你人工验收**，通过后再发阶段 2。

第一轮，在 AI Chat 中发送：

```text
请只实现本次变更所归属的 OpenSpec change 中「阶段 1：UI + 隔离 mock + 页面跳转」的任务，不要开始阶段 2。

禁止接入真实接口、真实数据、权限判断或业务状态流转；mock 须与真实链路隔离（如独立目录、环境开关或占位 service），边界写进 design 或代码。

完成后更新任务状态，并说明如何本地预览。不要归档，不要开始阶段 2。
```

你在本地对照设计图验收 UI 与页面跳转。不通过则让 AI 继续修阶段 1，通过后再回复 **「UI 验收通过」**，发送第二轮：

```text
UI 验收通过。请实现本次变更所归属的 OpenSpec change 中「阶段 2：真实接口、数据、权限、状态流转和错误处理」的任务。

完成后更新任务状态，并说明实际改动、已经执行的检查、尚未验证的内容和剩余风险。不要归档。
```

检查实际改动是否符合更新后的 proposal 和 specs。所有任务完成后，进入验证。

## 5. 验证

让 AI 核对实现是否满足 change，并提供真实的测试、构建和检查结果。验证未通过时，先修复并重新验证，不要进入归档。

在 AI Chat 中发送：

```text
请验证本次变更所归属的 OpenSpec change。

请逐项对照更新后的 proposal、specs、design、tasks 和变化记录，运行与本次改动及其直接影响相关的测试、构建或检查。列出实际执行的检查、结果、未覆盖项和剩余风险；存在失败时不要宣称完成，也不要归档。
```

只有验证通过、未覆盖项和剩余风险都已确认，才能进入归档。

## 6. 归档

验收完成后，让 AI 同步规格并归档 change，为本次需求变更收尾。

在 AI Chat 中发送：

```text
请归档本次变更所归属的 OpenSpec change。

归档前确认验证结果有效，并按项目约定同步需要更新的规格。若验证无效、同步失败或无法归档，请明确说明原因，不要伪装完成；成功后告诉我归档位置和规格更新结果。
```

确认 AI 返回了真实的归档位置和规格更新结果，需求变更流程结束。

## 未启用 OpenSpec 时

把 change 换成对应的 issue、PR 或任务卡：同样先定位原记录、更新正文为当前口径、追加变化记录（不覆盖历史）、同步验收，再改代码；涉及 UI 设计图时仍按两阶段拆分任务，阶段 1 人工验收 UI 后再做阶段 2。各工具 fallback 见 [`docs/规范/可选工具协作.md`](./docs/规范/可选工具协作.md)。`AGENTS.md` 底线不打折。
