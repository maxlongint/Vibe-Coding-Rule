# 快速开始

> 核心规范是 [`AGENTS.md`](./AGENTS.md)。本页帮你选落地深度；仓库是什么、怎么复制见 [`README.md`](./README.md)。

## 场景一：只用 AGENTS.md（最小落地）

**适合**：个人项目、小团队、或只想约束 AI 协作底线。

1. 复制 [`AGENTS.md`](./AGENTS.md) 到业务项目根目录；也可用 [`README.md` §3](./README.md#3-用-ai-接入或升级-vibe-coding-rule) 的 AI 提示词自动接入。
2. 若项目已有 `AGENTS.md`，先 diff 差异，确认合并方案后再写入。

**典型用法示例**

| 改动类型 | 你要做的事 | AI 应遵守的底线 |
| --- | --- | --- |
| 修错别字 / 局部文案 | 直接描述改动 | 最小验证即可 |
| 修不改变对外行为的 bug | 说明复现与预期 | 验证相关测试或构建 |
| 改 API / 权限 / 流程 | 先在 issue 或 PR 描述写清范围与验收，再改代码 | 需求可追溯 + 完成前验证 |

需要长期项目规范或项目 skill 时，再按 [`README.md`](./README.md) 复制 `docs/` 与 `.agents/`。

## 场景二：AGENTS.md + Issue / PR（轻量需求治理）

**适合**：有基本评审流程、但暂不引入 OpenSpec 的团队。

1. 复制 `AGENTS.md`（同场景一）。
2. 会改变外部可观察行为时，先在 issue 或 PR 描述中写清：
   - 变更范围
   - 验收标准
   - 验证方式
   - 未验证项与剩余风险
3. 需求变更时：更新正文为当前口径，**追加**变更记录，不覆盖历史。

OpenSpec 的 change 目录可等价替换为 issue / PR / 任务卡，流程细节见 [`新需求开发.md`](./新需求开发.md) 与 [`需求变更.md`](./需求变更.md) 文末「未启用 OpenSpec 时」。

## 场景三：全套工具链（OpenSpec + CodeGraph + Superpowers）

**适合**：需要规格化需求、影响分析与执行纪律的中大型团队。

1. 完成场景一的 `AGENTS.md` 复制；按需复制 [`docs/README.md`](./docs/README.md) 与 [`docs/规范/代码设计原则.md`](./docs/规范/代码设计原则.md)，以及 [`.agents/README.md`](./.agents/README.md)。
2. 在**业务项目**按顺序安装 **OpenSpec → CodeGraph → Superpowers**（OpenSpec / CodeGraph 还需初始化；提示词与是否重启见 [`README.md` §4](./README.md#4-可选工具链通过-ai-安装)）。
3. 按流程文档操作：
   - 新需求 → [`新需求开发.md`](./新需求开发.md)
   - 需求变更 → [`需求变更.md`](./需求变更.md)
   - 三种命令写法 → [`流程公共约定.md`](./流程公共约定.md)

分工记忆口诀：**AGENTS 定底线 · OpenSpec 管需求 · CodeGraph 管影响 · Superpowers 管方法**。

未装齐时按 [`流程公共约定.md`](./流程公共约定.md) fallback 到 issue/PR、人工读源码与手动验证（`AGENTS.md` 底线不打折）。

| 工具 | 角色 | 未安装时的 fallback |
| --- | --- | --- |
| **OpenSpec** | 需求与验收（`openspec/changes/`） | issue、PR、任务卡 |
| **CodeGraph** | 影响分析（MCP） | AI 人工通读源码与测试 |
| **Superpowers** | 澄清、拆任务、TDD、验证纪律 | 手动澄清与自行核对证据 |

---

## 选型建议

```text
只想管 AI 别乱提交、别伪造成功     → 场景一
有评审但不想上工具链               → 场景二
要规格、验收、变更可追溯闭环         → 场景三
```

无论选哪种场景，[`AGENTS.md`](./AGENTS.md) 六条底线均适用。
