# 项目规范文档（docs/）

本目录存放**长期有效**的项目规范：技术栈约定、架构说明、跨需求业务规则、环境配置说明，以及按需复制的通用协作模板。

## 首要原则：先判断，再收录

文件被放入 `docs/`，不代表它天然适合作为长期项目规范，也不应因此自动收录到本文件的索引中。`docs/` 不对文件写入做硬性限制，但应通过索引维护提醒用户保持内容边界。

AI 在新增、移动文档或更新索引前，应先判断内容是否**长期有效、跨需求适用且已经稳定确认**。如果内容更像单次进行中需求、临时记录、任务计划或尚未确认的方案，应在更新索引前提醒用户，说明不适合收录的原因、建议使用的载体，并询问是否仍要将其作为长期项目规范收录。用户确认前，不应擅自加入索引，也不得擅自移动或删除文件。

收录时还应检查文档内容是否与其声明的用途一致。每份文档应围绕单一、稳定的职责组织内容，不应混入属于其他类型文档的操作步骤、临时过程或无关说明。例如，框架说明应记录采用的框架、版本范围、支持环境、关键依赖与约束；安装命令、部署步骤和排障操作如需长期维护，应拆分到环境配置说明或运行手册，并在索引中分别注明适用场景。

## 作为 AI 项目上下文

`docs/` 是 AI 按任务相关性读取的长期项目知识库，不是每次交互都要全量加载的固定上下文。

涉及本项目的分析、设计、实现、排查或评审时，AI 应先读取本文件，再根据下方索引中的“适用场景”和“读取要求”选择与当前任务直接相关的文档。与项目无关的一般问答无需读取本目录；索引未指向、与当前任务无关的文档也不应为了扩充上下文而加载。

确认内容适合作为项目规范后，新增、移动或删除文档时应同步维护索引、适用场景和读取要求。发现链接失效、文档不可读或规范之间存在冲突时，应显式说明，不得猜测缺失内容。

## 与进行中需求载体的区别

| 载体 | 路径 | 放什么 | 何时用 |
| --- | --- | --- | --- |
| **项目规范** | `docs/`（本目录） | 相对稳定、跨需求有效的约定 | 团队日常开发查阅 |
| **进行中需求** | `openspec/changes/<change-id>/` | 进行中的 proposal、specs、design、tasks、接口/API 契约变化 | 启用 OpenSpec 时 |
| **进行中需求（轻量）** | issue / PR 描述 / 任务卡 | 同上，无 OpenSpec 时的等价载体 | 未启用 OpenSpec 时 |

不要把「某次需求的 proposal / tasks」写进 `docs/`；也不要把「长期架构约定」只写在某个 change 里而不沉淀到 `docs/`（确需沉淀时，验收后合并进本目录）。

## 建议组织方式

按项目实际需要分子目录，例如：

```text
docs/
├── README.md           # 本文件：docs 目录说明与索引
├── experience/         # 项目结束、阶段复盘、问题沉淀后的经验包
├── 架构/               # 架构说明、模块关系、关键设计决策
└── 规范/               # 编码、协作等团队规范
    ├── 代码设计原则.md
    └── 可选工具协作.md # OpenSpec / CodeGraph / Superpowers（按需复制）
```

## 经验包

`docs/experience/` 用于存放项目结束、阶段复盘或问题沉淀后形成的经验包 Markdown。经验包不是 skill，不放入 `.agents/skills/`；如后续需要被 AI 按固定流程调用，再提炼为项目级 skill。

新增经验包时，应同步在本节或下方索引中添加入口，便于后续查找和引用。
