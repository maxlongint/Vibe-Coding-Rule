# 项目规范文档（docs/）

本目录是按索引读取的长期项目知识库，存放长期有效、跨需求适用且已经稳定确认的项目规范，例如技术栈、架构、业务规则、环境约束和条件化协作流程。

## 收录边界

单次需求、临时计划、尚未确认的方案以及进行中的 proposal、tasks 不属于项目规范。AI 每次 Chat 都需要知道、在读取本索引前必须生效，或用于判断是否读取本目录和是否允许继续的规则也不属于本目录，必须写入项目根目录 `AGENTS.md`。新增或移动文档前，应确认其职责单一、内容长期有效；不符合条件时使用正确载体，不擅自收录、移动或删除。

安装、部署和排障步骤如需长期维护，应与框架说明、架构规则等稳定知识分开，分别注明适用场景。

## 读取方式

由项目根目录 `AGENTS.md` 判断当前任务是否需要项目特有知识或条件化流程。只有任务确实需要，或 `AGENTS.md` 明确触发时，才读取本文件，再按索引选择与任务直接相关的文档；不得在每个 Chat 中默认加载本目录。与任务无关、索引未指向的文档不为扩充上下文而加载。

新增、移动、改名或删除规范时，同步维护索引。链接失效、内容不可读或规范冲突时显式说明，不猜测缺失内容。

## 与进行中需求载体的区别

| 载体 | 放什么 | 使用方式 |
| --- | --- | --- |
| `docs/` | 相对稳定、跨需求有效的项目规范 | 按本索引选择性读取 |
| OpenSpec change（唯一） | 用户已明确确认的新增需求或需求变更，以及该需求后续的目标、范围、验收、设计、任务和变化记录 | 新增需求由用户在口径确认后主动授权创建并通过冲突检查；需求变更或已有需求由用户确认继续原 change / 新建关联 change，新建分支同样由创建请求授权生成 ID 和检查冲突；同一需求只对应一个 change |

长期架构约定不应只留在进行中需求载体；确需沉淀时，在验收后合并进本目录。

## 按需组织

仅在存在实际内容时创建子目录，不预建空目录。常见目录包括：

- `架构/`：模块关系和关键设计决策；
- `规范/`：编码与协作规范；
- `experience/`：项目结束、阶段复盘或问题沉淀后形成的经验包 Markdown。

经验包不是 skill；只有后续确需由 AI 按固定流程调用时，才提炼到 `.agents/skills/`。
