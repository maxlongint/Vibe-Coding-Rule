# 需求治理与文档结构

> **适用范围：** 需要可追溯需求记录的项目。未启用 OpenSpec 或未采用 Vibe Coding 工具链时，仍须遵守 `AGENTS.md` 速查中的需求治理底线；本文提供目录结构、治理模式与变化处理细节。
>
> **权威关系：** 需求「何时必须记录」的强制底线以 `AGENTS.md` 速查为准；本文展开目录、流程与模板，不弱化强制规则。

## 默认目录结构

示意，不预建空目录或空文件：

```text
docs/
  README.md                 # 项目入口：环境、命令、文档链接
  规范/                     # 有长期规范时再建
  案例/                     # 用户确认复用价值后再建
openspec/                   # 安装 OpenSpec 并 init 后生成
  project.md
  changes/
    <change-name>/          # 有真实需求时再建 change
      proposal.md
      design.md
      tasks.md
      specs/<domain>/spec.md
  specs/<domain>/spec.md    # 归档后的规格
```

## 文档落点规则

- 目录结构是默认建议；项目已有明确约定时，应在 `docs/规范/` 中声明覆盖原因和适用范围，但只能覆盖目录结构、命名方式等可适配规则；**不修改** `AGENTS.md`。
- 不为未发生的需求、未启用的规范、未确认复用价值的案例预建空目录或空文件。
- `docs/README.md` 只做项目入口，说明项目环境、技术栈、框架、常用命令和文档入口；不承载详细规范或具体业务需求。
- `docs/规范/` 放**项目特有**的长期规则（如 UI、API、业务约定、工作流等）；具体文件名和数量由项目自行维护。新增或调整时只改 `docs/规范/` 内文件，**不修改** `AGENTS.md`。
- `AGENTS.md` 第 1–4 节（工程守则、注释、编码、提交与发布）是跨项目强制规则，**只写在 `AGENTS.md`**，不下沉到 `docs/规范/`。
- 项目可在 `docs/规范/` 提供 `规范总览.md` 或 `index.md` 作为索引；无索引时，AI 按文件名识别明显相关的规范正文。
- `docs/案例/` 只在用户明确确认某个案例具有培训和复用价值时再建立；AI 不得主动创建案例目录。

## 真实业务需求与硬触发条件

真实业务需求是指会改变用户能力、业务流程、接口契约、权限规则、数据模型、状态流转或验收标准的变更。需求治理的底线是可追溯：必须能确认需求来源、当前状态、关键验收标准、变化记录和剩余风险。

**硬触发条件：** 只要本次修改会改变外部可观察行为、接口契约、数据结构、权限规则、状态流转、验收标准或用户操作路径，就必须先有可追溯需求记录。记录载体由项目已声明的治理方式决定；**未安装或未启用 OpenSpec 时，不得因缺少 OpenSpec 而跳过记录**，应使用外部需求系统、issue、任务卡或轻量记录等已确认载体。

## 三种治理模式

- **OpenSpec 需求治理：** 项目已安装并初始化 OpenSpec 时，中等及以上真实业务需求**默认**使用 OpenSpec change、spec 和 tasks 承载；`AGENTS.md` 只规定何时必须使用、如何验证以及如何处理冲突。
- **外部需求系统治理：** 使用需求管理系统、任务系统、issue 系统等；仍应在 OpenSpec change（若启用）、任务记录、变更说明、PR 描述、提交说明或交付说明中记录外部链接、编号或用户确认来源。
- **轻量记录治理：** 适用于原型、一次性验证、小范围内部工具等；可用任务记录、任务卡、issue、变更说明、PR 描述或交付说明承载变更范围和验收标准，但仍必须可追溯。轻量记录不替代中等和重大真实业务需求在**已启用 OpenSpec 时**的 OpenSpec 记录；未启用 OpenSpec 时，轻量记录仅适用于规则允许的小范围场景，中等及以上需求须使用外部需求系统或经用户确认的其他可追溯载体。

轻量记录最小模板见 [模板/轻量记录模板.md](./模板/轻量记录模板.md)。PR 描述字段见仓库 `.github/pull_request_template.md`（若已复制到项目）。

## 治理方式未声明时

- 只读分析、复现、影响判断和临时验证可以继续。
- 一旦准备正式修改会影响行为、接口、数据、权限、流程或验收标准的内容，应先建立或更新可追溯需求记录；若项目已存在 `openspec/`，优先使用 OpenSpec change；若尚未初始化 OpenSpec，应先向用户确认记录载体（初始化 OpenSpec、外部需求系统或轻量记录），**不得默认跳过记录或自建未约定的需求目录**。
- 用户选择外部或轻量记录方式时按选择执行。
- 用户无法及时确认、但任务小且需要继续推进时，可先用任务记录、变更说明、PR 描述、提交说明或交付说明作为轻量记录载体，并说明原因和剩余风险；后续若范围升级，应补建正式需求记录（含 OpenSpec change 或外部系统事项，视项目治理方式而定）。
- 不应仅因项目未声明治理方式，就默认创建自定义需求目录；**已启用 OpenSpec 时**，真实业务需求默认以 OpenSpec change 为主记录载体；**未启用 OpenSpec 时**，经用户确认可采用外部需求系统或轻量记录（见上文「三种治理模式」）。

## OpenSpec 目录规则

> 本节仅在项目已安装并初始化 OpenSpec（存在 `openspec/`）时适用。

- OpenSpec 是**已启用 OpenSpec 的项目**中，中等和重大需求变化的主记录载体；不再维护与之重复的自定义完整需求目录。
- 项目已使用 OpenSpec 时，需求、设计、任务和规格统一写入 `openspec/changes/<change-name>/`；不得再把 `brainstorming`、`writing-plans` 的产出单独落到 `docs/superpowers/specs/` 或 `docs/superpowers/plans/` 作为主记录。
- OpenSpec 的 spec 应描述外部可观察行为和验收标准，不应只描述内部实现。
- OpenSpec 的 tasks 应包含实现步骤、验证方式和完成状态。
- 小修、纯文案、局部呈现、内部测试补充通常不需要创建 OpenSpec change，但必须在任务记录、交付说明或 PR 描述中记录变更范围和验证结果。
- 纯缺陷修复、测试补充、内部重构、依赖修补、格式调整或文案微调，通常不创建 OpenSpec change；若改变既有业务规则、已发布行为规格、验收标准或对外契约，仍按下文「需求变化」处理。

## 需求变化与归档

需求变化必须先确认归属和记录载体，再记录变化、判断影响、更新文档或任务记录以及实现计划。

### 标准动作

```text
1. 确认变化归属：原需求变化、新需求、缺陷修复、轻量调整，或外部系统已有事项。
2. 确认记录载体：OpenSpec change，或外部需求记录、任务记录、变更说明、PR 描述、交付说明等。
3. 判断变化等级：小变化、中等变化、重大变化。
4. 记录变化：OpenSpec 治理默认更新 change 下的 proposal、design、tasks 和 specs；外部或轻量治理应在对应记录中说明变更范围、取舍原因和验收变化。
5. 更新 OpenSpec change 下的 proposal.md、design.md、tasks.md、spec.md 等相关文档；若使用外部需求系统，应同步保留外部链接、编号或确认来源。
6. 必要时更新影响分析；有代码分析工具（如 CodeGraph、IDE 引用、语言服务器等）时优先使用（见 [AI协作与工具链.md](./AI协作与工具链.md)）；没有工具则记录人工影响判断。
7. 更新实现计划。
8. 更新测试和验收标准。
9. 根据变化等级重新确认，可由用户确认、负责人评审，或在记录中写明 AI/开发者自检结论和剩余风险。
10. 再继续实现。
```

### 变化等级

- **小变化：** 非关键文案、非流程性呈现调整、错误提示文案等。更新任务记录和验收标准，并重新运行相关测试。
- **中等变化：** 例如邮箱登录从大小写敏感改为不敏感、增加第三方 OAuth 账号绑定、新增输入校验规则。需更新设计记录、相关规格、任务记录，重新做必要影响分析和相关验证。
- **重大变化：** 例如认证协议、登录策略、权限模型、数据模型变化。应暂停实现，更新目标、设计和规格记录，重新做影响分析、方案评审、任务拆分，并在必要时新建 OpenSpec change。

### 升级规则

- 小变化若影响操作路径、权限可见性、状态流转、验收标准、用户对操作结果的预期、数据提交意图、法律或合规含义，应按中等或重大变化处理。
- 目标仍相同、只是范围或规则变化，且仍围绕同一能力时，继续使用原 OpenSpec change。
- 目标变成另一项系统能力、原需求已完成或归档、新需求影响完全不同模块，或需要独立评审、发布和回滚时，新建 OpenSpec change。

### 需求归档

- 需求状态默认使用 `进行中`、`已完成`、`已暂停`、`已废弃`、`已替代`；项目可以声明等效状态，但必须表达相同含义。
- 需求完成、暂停、废弃或被替代时，必须在 OpenSpec change 或外部需求系统中标记状态、日期和原因。
- 已完成需求应记录最终验收结果；未完成或废弃需求应记录停止原因、剩余风险和后续方向。
- 是否移动到归档目录由项目约定决定；没有归档目录时保持原目录不动，通过文档状态标记归档。
- 归档触发条件是需求已完成且通过验收，或用户/负责人明确要求归档。归档前应确认不再需要追加实现任务。
- 归档操作可由用户发起，AI 也可在交付总结中建议，但需用户确认后执行。
- 归档后原则上不继续追加实现任务；如需恢复或扩展，应先判断是恢复原 OpenSpec change 还是新建 change。目标和验收仍有效且范围相近时，可恢复原 change 并追加变更说明；目标已变或影响范围显著变化时，应新建 OpenSpec change。
