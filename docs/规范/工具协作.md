# 工具协作（OpenSpec · CodeGraph · Superpowers）

> 是否进入需求或工具流程、是否已取得用户授权，以项目根目录 [`AGENTS.md`](../../AGENTS.md) 为准。本文仅在 `AGENTS.md` 明确路由后按需读取，说明三项工具被触发后的职责、组合方式、需求载体、失败处理与 UI 两阶段任务重组；读取本文本身不授予创建、选定或更新 change 的权限。

## 1. 默认安装假定与中断

进入相关工具流程后，项目规则默认所需工具已经成功安装：

| 工具 | 默认假定 |
| --- | --- |
| OpenSpec | 已安装并完成当前 AI 工具侧初始化，可创建与操作 change |
| CodeGraph | 已安装并完成本仓库索引与当前 AI 工具侧接入，可查询符号与影响 |
| Superpowers | 已安装到当前 AI 工具，相关 skill 可加载并调用 |

只有当前任务确实需要某项工具时才检查其可用性，不在每个 Chat 开始时预检全部工具。执行过程中一旦需要使用某项工具，却发现其未安装、未初始化、命令/MCP/skill 不存在或当前会话无法调用，必须立即中断相关流程，明确告知用户**哪一个工具未安装或不可用**，并说明已完成项与阻塞点。不得假装可用，不得静默改用其他工具或轻量载体继续，也不得擅自安装；不依赖该工具的任务不受阻塞。

安装与初始化入口见项目根目录 [`README.md`](../../README.md) 文末「工具安装」；实际命令以当前官方说明为准。用户完成安装或修复并确认可用后，再从中断点继续。

归档 change、提交、发布、写入生产数据及其他外部副作用仍须分别遵守用户指令和 `AGENTS.md`，不因“工具已安装”而自动获得授权。

## 2. 职责与顺序

| 工具 | 负责 | 不负责 |
| --- | --- | --- |
| OpenSpec | 承载需求、范围、验收、设计、任务和变化记录 | 不根据源码猜需求 |
| CodeGraph | 提供符号、调用链、数据流、直接影响和测试面 | 不定义需求事实，不替代当前源码与测试证据 |
| Superpowers | 提供澄清、拆解、TDD、调试、验证和收尾方法 | 不扩大范围，不创建平行口径，不提供提交发布权限 |

协作检查点：

1. **确认用户授权与唯一载体**：只有用户明确确认新增需求、需求变更或指定已有需求的后续工作后，才进入 OpenSpec 流程。新增需求先完成澄清并取得用户对目标、范围和验收口径的确认；用户主动要求创建后，AI 生成 `change-id` 并只读检查现有 change。发现同名 change，或发现已有 change 可能承载该需求时，停止并列出冲突项、状态和关系，不得改用其他 ID 或创建；没有冲突时直接使用生成的 ID 创建，无需再次确认 ID。需求变更或已有需求存在一个或多个候选 change 时，仍须展示全部候选及关系，由用户确认继续原 change / 新建关联 change，不得自动选定。选择新建关联 change 时，先确认新口径；用户主动要求创建后，AI 生成 ID 并执行同等冲突检查，无需再次确认 ID。issue、PR、任务卡、设计图与外部文档只作输入来源，确认后的口径必须写入当前唯一 change，不得以其替代载体，也不得因 OpenSpec 不可用而改用它们继续。
2. **再调用工具**：按该 OpenSpec change 工作；需要代码事实时按需使用 CodeGraph；需要澄清、拆解、TDD、调试或验证方法时使用 Superpowers。CodeGraph 为按需查询，不要求在每一步提示词中点名。
3. **核对事实与证据**：工具输出须与当前源码、测试或实际命令结果核对。
4. **回填同一载体**：确认后的需求口径、设计、任务、影响分析、验证结果、未验证项和剩余风险写入该 OpenSpec change。

**Superpowers 与载体优先级（防抢戏）**：skill 不能替代 `AGENTS.md` 规定的用户确认，也不提供创建、选定或更新 change 的授权。用户确认进入 OpenSpec 流程后，OpenSpec change 优先于任何 skill 的默认落盘路径、目录结构与提交流程。可加载 brainstorming、writing-plans、TDD 等方法 skill，但：

- 用户尚未明确确认需求意图时，澄清结论只留在对话。新增需求或新建关联 change 未取得明确创建授权并通过冲突检查，或继续现有 change 时未确认载体前，不得创建、选定或更新 change，也不得形成最终设计、计划或开始实现。完成对应授权后必须先写入当前唯一 change，再继续定稿设计、计划或实现，不得带着仅存在于对话或 skill 默认路径中的结论切换 Chat。
- 确认后的设计与计划只回填该 change；禁止另建 `docs/superpowers/specs/`、`docs/superpowers/plans/` 或其他平行口径，禁止因 skill 默认行为自动提交。
- 用户明确要求跳过某类方法 skill（如 brainstorming、writing-plans）时，从其要求，不得强行套完整方法链。

## 3. 需求归属与变更判断

- AI 识别到内容可能构成新需求或需求修改时，先按 `AGENTS.md` 说明判断依据和可能影响，请求用户确认是否作为正式需求处理；用户确认需求意图前不得查找 OpenSpec 候选或进入本流程。
- 新增需求在用户确认需求意图后继续澄清；目标、范围和验收口径确认且用户主动要求创建时，AI 生成 ID 并只读检查冲突。发现同名或可能承载该需求的现有 change 时停止，没有冲突才创建。
- 需求变更或已有需求在用户确认需求意图后，才能只读核对相关 OpenSpec change 及直接相关历史记录中的来源、当前口径、任务状态和验收；随后说明目标、影响、未知项和全部候选 change，请求用户确认继续原 change / 新建关联 change。新建分支在口径确认后由用户主动要求创建，AI 生成 ID 并检查冲突。
- 新增需求或新建关联 change 未完成创建授权与冲突检查，或继续现有 change 时未确认载体前，不得写入 change、修改代码或形成冲突口径；完成后先写入当前唯一 change，再实施。

## 4. 单一事实来源与跨 Chat

新增需求或新建关联 change 按明确创建授权通过冲突检查并成功创建，或需求变更及已有需求确认继续使用的现有载体后，同一进行中需求只对应一个当前有效的 OpenSpec change。用户消息、issue、PR、任务卡、设计图和外部文档可以作为输入来源，但确认后的目标、范围、验收、设计、任务和变化记录必须回填该 change。

切换 AI Chat 或发生上下文压缩后，不依赖对话摘要自动继续。用户已明确当前 `change-id` 时，重新读取其 `proposal`、`specs`、`design` 和 `tasks`；需求变更还要读取变化记录。用户未明确当前 change 时，只读查找并展示候选，取得确认后才能恢复；change 缺失、不可读、存在多个候选或互相冲突时停止并说明。

Superpowers 产生的设计和计划也回填该 OpenSpec change，细则见 §2「Superpowers 与载体优先级」。若 skill 默认要求创建 `docs/superpowers/specs/`、`docs/superpowers/plans/` 或提交设计稿，以该 change 和提交权限为准，不另建平行文档，不自动提交。

## 5. UI 设计输入与任务重组

仅在任务明确采用规定标题“阶段 1：UI、隔离 mock 与页面跳转；阶段 2：真实接口、数据、权限、状态流转与错误处理”（须与本文及工作流提示词用字一致，含顿号与“与”），或已有 §5.5 所列重组标记时使用本节；未进入该场景时**不要读取或执行**本节及以下各小节。本节不扩大需求范围。

工作流文档与任务载体中的阶段标题必须使用上述规定原文；不得改用 “UI + …”、斜杠缩写或其他近义写法，以免触发条件与机械核对失效。

基线、设计、映射和最终任务写入对应 OpenSpec change。

阶段 1 不接入真实接口、真实数据、权限判断或业务状态流转；mock 必须与真实链路隔离，并在设计或实现中写明边界。人工 UI 验收通过后，阶段 2 才接入真实业务链路。两个阶段分别记录可独立判定的验收标准。

### 5.1 权威设计来源

- “UI 设计图来源”必须是当前可读取的权威来源集合，或明确沿用 `design.md` 已记录且重新验证可用的来源集合。
- 多个来源要记录各自作用和冲突优先级。来源为空、仍是占位符、不可读或互相冲突但无优先级时，修改设计和任务前停止并请求确认。
- 不自行把仓库目录、旧设计稿、缓存图片或历史附件加入权威集合。
- UI 输入只补充已确认范围内的视觉、布局、文案和交互状态；涉及新增业务能力、数据、接口、权限或流程时，先更新唯一需求载体。
- 设计记录应包含权威来源、页面清单、mock 隔离方式和阶段边界；写入对应 OpenSpec change 的 `design`。

### 5.2 重组基线

- 首次为需求建立任务清单时，初始需求、设计和任务只依据已确认的文字需求；先在对应 OpenSpec change 中落盘原始任务，再读取 UI 输入并补充设计。对应初始 `proposal`、`specs`、`design` 和 `tasks`。
- 首次重组已有任务时，读取 UI 输入前先固化非 checkbox“任务重组基线”，记录原任务准确数量、编号、摘要和状态。
- 已有有效基线和映射时不得覆盖原基线；先新增非 checkbox“本轮 UI 更新前任务快照”，记录当前最终任务的准确数量、编号、摘要和状态。旧载体的基线或映射不完整时，先按本节修复规则处理。
- 文字需求与 UI 输入无法可靠分离时停止，请求提供独立需求口径，不能先消费设计再反推基线。

### 5.3 任务与映射

- 需求口径未变化时，以现有任务为基线增量调整；阶段划分不得删除、合并、概括或弱化原任务、验收、验证、依赖和状态。
- 将每个原任务拆成可核对的原子约束，包括功能、字段、权限、状态、错误处理和验收条件；逐项分配给一个或多个最终任务。不得用“等”“相关内容”“按设计”“字段持久化”等概括表述替代已列明的约束。
- 任务载体必须逐项记录“原任务及其原子约束 → 更新后任务编号、阶段与状态”映射。无法归类、与设计冲突或超范围的原项先保留并列为待决问题。
- 每个基线任务须互斥标为“保留 / 拆分 / 替代 / 取消”。取消或替代必须有已确认的需求变更或变化记录依据；取消项保留非 checkbox 历史且可以没有更新后编号，替代项必须映射到实际任务。
- 未拆分任务保留原编号且只出现一次。跨阶段任务拆为 `<原编号>-S1` 和 `<原编号>-S2`，裸原编号不再作为 checkbox。
- 每个最终任务只有一条行首、非缩进 checkbox，且有唯一、可独立验收的交付物。影响完成判定的“或”“可选”“视情况”“待确认”等内容须移入未决问题；如果未决内容影响当前任务的完成标准，应依据已确认需求确定唯一临时策略，否则停止并请求确认。
- 对语义重叠部分，能力或组件任务负责具体行为、状态、mock 和错误反馈，页面任务负责组装、路由、联调和视觉验收，并写明依赖。无人承接的页面独有交互仍归页面任务；不得为套用职责模板凭空创建能力任务。

### 5.4 机械核对

重组或修复后扫描整个任务载体，确认：

- checkbox 只出现在规定的阶段任务清单，不在基线、映射、目录、未决问题、核对记录、缩进位置或代码块中出现；
- 基线数量等于映射原任务行数，原编号集合一致且各出现一次；
- 每个映射目标恰好对应一条合法 checkbox，每条合法 checkbox 都能追溯到原任务或明确的设计新增任务；
- 每项原子约束都有最终任务承接，不存在概括替代、无人承接或互相冲突；
- 合法 checkbox 总数等于各阶段任务数之和，每个最终编号频次为 1；
- 核对记录列出语义重叠、职责、依赖和各自完成标准；没有重叠时也记录检查结果。

任一设计源、数量、追溯、格式或职责检查失败时，不得声明“无遗漏”，也不得开始实现。

### 5.5 后续实现与验证触发

后续实现或验证时，只要设计或任务中出现任一 UI 重组标记，就重新执行本节的设计源、基线、映射、checkbox 和职责核对。标记包括：“UI 设计图来源”（兼容旧标题“UI 设计来源”）、规定的 UI 两阶段标题、任务重组基线、任务重组映射、设计图新增任务目录，以及 `-S1` / `-S2` 任务编号。

普通任务使用的泛化“阶段 1 / 阶段 2”标题不单独构成标记。只有上述标记全部不存在时，才能按普通任务流程跳过本节检查。

### 5.6 旧结构修复

修复错误重组的旧任务载体时，先记录所有现存 checkbox 的章节、编号、摘要和状态。只有原基线与映射互相一致，并能由需求来源、修改前版本或版本历史印证时，才保留其语义和原状态并修正结构；证据不足时停止并请求确认，不从当前重复清单反推原任务。

基线、映射、设计新增任务目录和其他非阶段区中的 checkbox 应转换为非 checkbox 记录。与阶段任务或映射目标重复的 checkbox 属于结构性副本，不得重新计入基线或作为新增原任务。

修复后仍须通过本节全部机械核对，才能继续实现或验证。
