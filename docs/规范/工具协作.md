# 工具协作（OpenSpec · CodeGraph · Superpowers）

> 强制底线见项目根目录 [`AGENTS.md`](../../AGENTS.md)。本文存在、被索引或包含调用方式，不代表工具已经被采用、可用或获得操作授权。

## 1. 采用、可用与授权

三个状态必须分开判断：

为兼容现有入口文案，其他文档中的“启用某工具”只表示本文的“采用”，不表示工具已经可用或获得操作授权。

| 状态 | 含义 | 不代表 |
| --- | --- | --- |
| **采用** | 用户、适用的上级指令或当前任务已经选择使用该工具 | 工具已经安装、初始化或能在当前会话调用 |
| **可用** | 所需命令、MCP、skill、项目初始化或索引已就绪并能实际调用 | 当前任务已经采用该工具，或允许产生外部副作用 |
| **授权** | 用户或运行环境允许执行某项安装、写入、归档、提交等操作 | 授权范围外的其他操作也被允许 |

工具只在以下条件下视为已采用：

| 工具 | 采用条件 |
| --- | --- |
| OpenSpec | 用户明确选择，或当前任务已经确认采用 |
| CodeGraph | 用户明确要求、适用的运行环境或项目指令要求，或当前任务已经确认采用 |
| Superpowers | 用户、运行环境或当前任务明确要求加载相应 skill |

文档、目录、配置、命令或索引存在只能证明可能可用，不能自行触发采用。工具被采用后再检查可用性；需要安装、初始化、重启或其他副作用时，仍须单独确认授权。

已采用的工具不可用时，应说明失败工具、失败层级、已完成项和可选处理方式。安装或初始化入口见项目根目录 [`README.md`](../../README.md)，实际命令以当前官方说明为准。未经用户确认，不自动安装，也不静默改用其他工具或载体；用户选择替代方案后，按新选择继续并说明能力差异。

## 2. 职责与顺序

| 工具 | 负责 | 不负责 |
| --- | --- | --- |
| OpenSpec | 采用后承载需求、范围、验收、设计、任务和变化记录 | 不根据源码猜需求，不因目录存在而自动采用 |
| CodeGraph | 采用后提供符号、调用链、数据流、直接影响和测试面 | 不定义需求事实，不替代当前源码与测试证据 |
| Superpowers | 采用后提供澄清、拆解、TDD、调试、验证和收尾方法 | 不扩大范围，不创建平行口径，不提供提交发布权限 |

协作检查点：

1. **先选唯一载体**：使用当前任务、issue、PR、任务卡、用户指定文档或采用 OpenSpec 时的 change；存在多个候选时先确认。当前任务只适合无需跨 Chat 的轻量工作。
2. **再按需调用工具**：OpenSpec 已采用时按 change 工作；CodeGraph 已采用且需要代码事实时查询；Superpowers 已采用时按 skill 触发规则组织方法。
3. **核对事实与证据**：工具输出须与当前源码、测试或实际命令结果核对。未采用 CodeGraph 时使用项目既有搜索和源码阅读方式，并说明未覆盖范围。
4. **回填同一载体**：确认后的需求口径、设计、任务、影响分析、验证结果、未验证项和剩余风险进入已选择的唯一载体。

## 3. 需求归属与变更判断

- 收到新需求或需求修改时，先在已选载体及直接相关历史记录中核对来源、当前口径、任务状态和验收；只有 OpenSpec 已采用时才扫描 change。
- 请求可能改变已有需求、验收、用户路径、接口、权限、状态或任务时，先说明证据、目标、影响与未知项，再确认更新原记录还是新建关联记录。
- 确认归属前不修改需求载体或代码，不创建冲突口径；确认后先更新唯一载体，再实施。
- 工具采用不改变权限边界。归档 change、提交、发布及其他外部副作用仍分别遵守用户指令和 `AGENTS.md`。

## 4. 单一事实来源与跨 Chat

同一进行中需求只选择一个当前有效载体。用户消息、设计图和外部文档可以作为输入来源，但确认后的目标、范围、验收、设计、任务和变化记录必须回填该载体。

切换 AI Chat 或发生上下文压缩后，不依赖对话摘要继续，应重新读取唯一载体。原载体仅为当前任务且后续 Chat 无法重新读取时，切换前先迁移到 issue、PR、任务卡、用户指定文档或采用 OpenSpec 时的 change。OpenSpec 已采用时，根据 `change-id` 重新读取当前的 `proposal`、`specs`、`design` 和 `tasks`；需求变更还要读取变化记录。载体缺失、不可读或互相冲突时停止并说明。

Superpowers 产生的设计和计划也回填唯一载体。若 skill 默认要求创建 `docs/superpowers/specs/`、`docs/superpowers/plans/` 或提交设计稿，以本项目选定载体和提交权限为准，不另建平行文档，不自动提交。

## 5. 场景扩展

任务采用 UI 设计输入和两阶段任务重组时，按需读取 [`UI任务重组.md`](UI任务重组.md)。未进入该场景时不加载其详细规则。

## 6. 调用与失败处理

| 类型 | 调用方式 |
| --- | --- |
| OpenSpec AI 命令或 skill | 使用当前 AI 工具初始化后实际生成的形式 |
| OpenSpec CLI | `openspec <子命令>` |
| CodeGraph MCP | `codegraph_explore` |
| CodeGraph CLI | `codegraph explore "问题"` |
| Superpowers skill | 当前 AI 工具支持的 skill 调用方式 |

斜杠命令、`$skill-name` 等形式只在当前工具实际提供时使用，不能把其他 AI 工具或版本的写法当作通用接口。命令、MCP 或 skill 不存在，索引不可读，或当前会话尚未加载新能力时，真实报告失败和可选处理方式，不得把安装成功、文件写入成功或旧输出当作能力已可用。

需要重启 AI 工具时，说明已完成项、剩余步骤和原因后停止相关工具流程。AI 不能替用户重启，也不得在新能力尚未加载时假装继续；用户明确选择其他工具、载体或项目既有方法后，方可按新选择继续。
