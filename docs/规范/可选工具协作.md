# 可选工具协作（OpenSpec · CodeGraph · Superpowers）

> 强制底线见项目根目录 [`AGENTS.md`](../../AGENTS.md)。本文是 OpenSpec、CodeGraph、Superpowers 的详细流程与限制的唯一来源；复制或存在本文，不等于已经安装、初始化或授权使用这些工具。

三项工具均为可选增强。未启用时直接按 `AGENTS.md` 和项目既有方式工作，不安装工具、不补建配套目录，也不降低留痕、影响分析和验证要求。

## 1. 启用前提与权限门禁

- **工具可用性不等于操作授权。** 安装成功、目录存在、命令可执行，只能证明能力可用，不能替代用户授权。
- **OpenSpec 仅显式启用。** 只有用户明确要求使用 OpenSpec、调用当前 AI 工具生成的等价 OpenSpec 命令或 skill，或明确要求操作某个 OpenSpec change 时，才进入对应流程。仓库已安装 OpenSpec、存在 `openspec/`、改动需要留痕、普通「新增 / 修改 / 修复」指令或 skill 建议使用，都不构成授权。
- **CodeGraph 按仓库状态触发。** 仅当根目录存在 `.codegraph/`，且任务需要定位代码、理解调用链或分析影响时，先用 CodeGraph；没有该目录就跳过，不自行安装、初始化或建立索引。
- **Superpowers 只提供方法。** skill 可以约束澄清、计划、TDD 和验证节奏，但不能扩大任务范围、触发 OpenSpec、创建提交或覆盖用户与项目规则。
- 用户已明确选择某项工具但当前环境不可用时，必须说明真实失败、已完成项和剩余步骤；需要安装、初始化、重启或改用 fallback 时，由用户选择，不得伪装能力已加载。

## 2. 职责边界

| 工具 | 负责 | 产出落点 | 不负责 |
| --- | --- | --- | --- |
| **OpenSpec** | 需求、范围、验收、设计与任务状态 | `openspec/changes/<change-id>/` | 不根据源码猜需求，不因安装完成而自动启用 |
| **CodeGraph** | 定位符号、调用链和直接影响范围 | 当前分析、需求载体的 `design` / `tasks` 或验证记录 | 不定义需求事实，不替代源码与测试证据 |
| **Superpowers** | 澄清、拆解、TDD、调试、验证和收尾方法 | 回填当前唯一需求载体 | 不另建平行口径，不提供提交、发布或 OpenSpec 授权 |

记忆口诀：**AGENTS 定底线 · OpenSpec 管需求 · CodeGraph 管影响 · Superpowers 管方法**。

## 3. 推荐协作流程

1. **选择需求载体**：只读分析无需留痕；用户已明确选择 OpenSpec 时使用对应 change，否则使用当前任务或用户指定的 issue、PR、任务卡等轻量载体。
2. **理解现状与影响**：先读取需求证据；存在 `.codegraph/` 时用 CodeGraph 定位相关符号、调用方、数据流和测试面，否则人工阅读源码与测试并记录未覆盖范围。
3. **澄清口径**：按需使用 `$brainstorming` 等方法确认目标、范围、非目标、验收标准和待决问题；结论直接回填唯一需求载体。
4. **记录与计划**：OpenSpec 已启用时更新 `proposal` / `specs` / `design` / `tasks`；未启用时在轻量载体中写清来源、范围、验收与验证方式、未验证项和剩余风险。
5. **实现**：按任务推进；可用 `$test-driven-development` 采用先失败测试、再最小实现、再验证的节奏。任何工具默认行为都不能绕过目标变更、非目标不变式和提交权限。
6. **验证**：对照需求载体运行与改动匹配的测试、构建、lint、类型检查或手工验收；列出真实结果、未覆盖项和风险。
7. **归档或收尾**：只有用户明确要求归档指定 OpenSpec change 时才执行 `/opsx:archive` 或等价操作；轻量流程则更新对应 issue / PR / 任务记录。

## 4. 单一事实来源与组合使用

同一需求只能有一份当前有效口径。工具产出是方法或证据，必须回填该载体，不能另存一份可能分叉的设计或计划。

切换 AI Chat 或发生上下文压缩时，落盘的需求载体仍是完整事实来源，不能依赖对话摘要继续操作。对已有 OpenSpec change 执行补充设计、拆分任务、实现或验证前，必须根据 `change-id` 重新读取当前的 `proposal`、`specs`、`design` 和 `tasks`；需求变更还要读取变化记录。读取失败或文档之间不一致时应停止并说明，不能凭压缩后的上下文补全。

使用 UI 设计图补充 change 前，提示词中的“UI 设计图来源”必须替换为明确且当前可读取的权威来源集合，或者明确写成“沿用 `design.md` 已记录的来源集合”。来源集合可以只有一个来源，也可以包含页面稿、流程原型和视觉资源等多个互补来源；有多个来源时必须写清各自作用，以及内容冲突时的权威顺序。若仍是占位符、为空、不可读取，或多个来源互相冲突但没有优先级，必须在修改 `design` 和 `tasks.md` 前停止，列出缺失信息或候选来源并请求确认；不得自行把仓库目录、旧设计稿、缓存图片或历史附件加入权威集合。沿用已有集合时，须重新读取并确认其中每个来源仍可用；最终在 `design` 中记录准确位置、作用、优先级与采用依据。

任务重组基线必须先于本轮 UI 设计输入形成。新建 change 时，初次创建 `proposal`、`specs`、`design` 和 `tasks` 只能依据已确认的文字需求口径；即使需求来源或当前消息同时包含 UI 设计图，也不得在初始任务中解释设计图或加入设计衍生任务，应先列出暂未读取的 UI 输入并落盘原始任务，再通过 UI 补充步骤读取设计。首次重组已有 change 时，须先在不读取本轮 UI 来源的情况下，从当前 `tasks.md` 固化非 checkbox“任务重组基线”并核对数量；若已有有效基线和映射，则不得覆盖原基线，而应先新增非 checkbox“本轮 UI 更新前任务快照”，记录当前最终任务的编号、摘要和状态。已有 UI 重组标记但基线或映射不完整时，须先按旧结构修复规则处理。只有基线或更新前快照落盘并核对后，才能读取本轮 UI 来源。若文字需求与 UI 设计内容混在一起且无法可靠分离，必须停止并请求提供独立的需求口径，不能先消费设计再事后反推基线。

需求口径未变化、仅补充设计或重新编排 `tasks.md` 时，应以现有任务为基线做增量调整。阶段划分只是任务组织方式，不得删除、合并、概括或弱化现有任务、验收标准、验证项、依赖关系和完成状态；“原任务 → 更新后任务编号、阶段与状态”的逐项映射必须写入 `tasks.md`，不能只留在 `design` 或对话回复中。无法归入新阶段、与设计输入冲突或超出原需求范围时，先保留原项并说明待决问题，不得通过缩短任务清单消除差异。已确认的需求变更明确取消或替代旧任务时，可以将对应任务标为“已取消”或“已替代”，但须记录变更来源和新旧映射，不得静默删除历史。

OpenSpec 把整个 `tasks.md` 中的 checkbox 行都视为可跟踪任务，不会根据章节标题或“不计入”等文字排除。重组后的文件只能在“阶段 1”和“阶段 2”任务清单中出现 checkbox；“任务重组基线”“任务重组映射”、父项、设计图新增任务目录、未解决问题和核对记录都必须使用非 checkbox 表格或文本，且不得在代码块中保留 checkbox 语法。阶段任务清单中的所有可跟踪任务——未拆分原任务、拆分子任务、验证任务和新增任务——都必须使用行首、非缩进的 `- [ ] <编号> ...` 或 `- [x] <编号> ...`；缩进 checkbox 属于格式错误，必须先修正。

未拆分的原任务继续使用原编号，并只在所属阶段出现一次；跨阶段原任务改为非 checkbox 父项或只保留在映射表中，实际工作统一拆为 `<原编号>-S1` 与 `<原编号>-S2` checkbox 子任务，裸原编号不再作为 checkbox，避免父项和子项重复计数或阶段 1 提前完成原任务。设计图新增任务可以另建非 checkbox 目录用于说明来源和归属，但每个新增任务只能在所属阶段保留一个实际 checkbox。整个文件中每个最终任务编号必须恰好对应一条 checkbox 行；映射表可以引用该编号，但基线、目录和其他非阶段区不得再为它创建 checkbox。

编号唯一不等于工作内容不重复。重组后每个 checkbox 还必须有唯一、可独立判定完成的交付物；发现两个任务描述同一页面、组件或交互时，须在任务文本或映射说明中明确职责与依赖。仅对重叠部分，能力或组件任务负责具体行为、状态、mock 适配和错误反馈，页面任务负责组装这些能力、路由接入、页面级联调和视觉验收，不得重复承诺组件实现。没有其他任务承接的页面独有交互、状态或错误处理仍由页面任务负责，并须写成其唯一交付物；不得为了套用职责模板而凭空创建能力任务。不能仅为消除重叠而合并原任务或让一个最终 checkbox 代替多个原任务；若无法划清边界，保留原任务追溯并列为待决问题，不能带着职责重叠开始实现。

修改任务清单前，必须先在落盘载体中固化“任务重组基线”：原 checkbox 任务准确数量，以及每项的编号、摘要和原状态；基线只能使用非 checkbox 表格或文本，原状态写成“未完成 / 已完成”等普通字段，不能复制 `- [ ]` 或 `- [x]` 语法。后续不能用映射表反推或替代这份基线。

任务重组完成后须扫描整个文件中所有 checkbox 语法，不得因章节、缩进或代码块而排除。机械核对必须确认：基线数量等于映射表原任务行数；两边原编号集合完全一致且每个原编号只出现一次；映射出的每个更新后编号在阶段任务清单中恰好对应一条行首、非缩进 checkbox；阶段任务清单中的每个 checkbox 都能追溯到一个原任务或明确列入新增任务；阶段之外、缩进位置和代码块中的 checkbox 数量均为零；全文件合法 checkbox 总数等于阶段 1 与阶段 2 的任务数之和；从全文件所有合法 checkbox 行提取的每个最终任务编号频次均为 1。还须逐对检查相同页面、组件、交互和验收结果的任务，并在 `tasks.md` 的“核对记录”中列出发现的语义重叠及其“能力实现 → 页面集成”职责、依赖和各自完成标准；不存在重叠时也要明确落盘记录检查结果。任一数量、追溯、格式、设计源或职责边界检查不一致时，不得声明“无遗漏”，也不得开始实现。

后续实现或验证时，只要 `design` 或 `tasks.md` 出现任一 UI 重组标记——“UI 设计图来源”（兼容旧标题“UI 设计来源”）、规定的 UI 两阶段标题、任务重组基线、任务重组映射、设计图新增任务目录，或 `-S1` / `-S2` 任务编号——就必须执行完整的设计源、基线、映射、checkbox 和语义职责核对。普通任务使用的泛化“阶段 1 / 阶段 2”标题不单独视为 UI 重组标记。只有所有 UI 重组标记都不存在时，才能按普通 tasks 流程跳过检查；不得把只完成一部分的旧文件当成无 UI change。

修复已经错误重组的旧 `tasks.md` 时，先记录所有现存 checkbox 的章节、编号、摘要和状态，不得直接把这些 checkbox 重新当作原任务建立基线。若已有“任务重组基线”和“任务重组映射”彼此一致，并能与 `proposal`、`specs`、修改前版本或版本历史相互印证，则保留其语义和原状态，把基线及新增任务目录中的 checkbox 转为非 checkbox 记录；非阶段区与阶段任务或映射目标重复的 checkbox 属于待转换的结构性副本，不算新增原任务。若基线或映射缺失、互相矛盾，且无法从修改前版本、版本历史或已确认需求载体可靠重建，则停止修复，列出冲突和缺失证据并请求确认，不能从当前重复清单反推基线。修复后仍须通过上述全部机械核对，才能开始实现。

已确认的需求变更取消或替代旧任务时，映射须把每个基线任务明确标为“保留 / 拆分 / 替代 / 取消”之一且四类互斥。“取消”项只保留非 checkbox 历史与映射，允许没有更新后编号；“替代”项必须映射到实际存在的 checkbox。机械核对应分别统计四类，并确认其并集完整覆盖基线，不能要求已取消任务留下永久未完成 checkbox。

| 使用组合 | 唯一需求载体 | 处理方式 |
| --- | --- | --- |
| 未启用 OpenSpec | 当前任务、issue、PR、任务卡或用户指定文档 | 澄清、设计、计划和验证结果都更新到该载体 |
| 仅 OpenSpec | 对应 change | 需求与验收进 `proposal` / `specs`，方案和任务进 `design` / `tasks` |
| 仅 Superpowers | 已选定的轻量载体 | skill 产出直接回填，不额外创建 `docs/superpowers/` 平行文档 |
| OpenSpec + Superpowers | 对应 change | `$brainstorming` 回填 `proposal` / `specs`，计划回填 `tasks`，必要设计回填 `design` |
| 任意组合 + CodeGraph | 上述既有载体 | CodeGraph 只补充代码事实、影响分析和未覆盖范围，不改变需求口径 |

若 Superpowers 默认要求生成 `docs/superpowers/specs/`、`docs/superpowers/plans/`、提交设计稿或转交后续 skill，以项目选定的唯一载体和 `AGENTS.md` 权限为准；不得同时维护两份口径。

## 5. 调用方式与占位符

| 类型 | 写法 | 使用位置 | 示例 |
| --- | --- | --- | --- |
| AI skill（Superpowers） | `$技能名` | AI 对话框 | `$brainstorming`、`$test-driven-development` |
| AI 命令或 skill（OpenSpec） | 当前 AI 工具生成的等价命令或 skill | AI 对话框 | Claude Code 中可为 `/opsx:propose add-sso-login`，Cursor 中可为 `/opsx-propose add-sso-login`；部分工具以 skill 形式提供 |
| OpenSpec CLI | `openspec 子命令` | 项目根目录终端 | `openspec validate add-sso-login` |
| CodeGraph MCP | `codegraph_explore` | 支持 MCP 的 AI 工具 | 查询符号、文件或影响范围 |
| CodeGraph CLI | `codegraph explore "问题"` | 项目根目录终端 | `codegraph explore "login 调用链"` |

- `<change-id>` 是占位符，应替换为简短明确的小写英文名并用短横线分隔，例如 `add-sso-login`；目录为 `openspec/changes/<change-id>/`。
- `/opsx:*` 只是本文的通用示例记法，不代表所有 AI 工具都使用冒号形式。实际调用以当前 AI 工具生成的等价 OpenSpec 命令或 skill 为准；具体命令集合取决于当前 OpenSpec 版本和 profile。基础工作流通常包含 `propose`、`apply`、`sync`、`archive`；`explore`、`new`、`continue`、`ff`、`verify` 等扩展能力不可用时，退回当前版本支持的命令，或直接按 change 文档完成等价操作。
- 命令或 MCP 不存在时必须显式反馈，不得把文件写入成功当成命令、技能或 MCP 已加载。

## 6. 各工具的限制

### OpenSpec

- 未获显式授权时，不创建、更新、应用、同步或归档 change；留痕改用当前任务或用户指定载体。
- 用户明确要求操作指定 change 后，只执行用户要求的动作，并只处理该 change 和其直接需要同步的规格，不顺手处理其他 change。
- 用户已选择 OpenSpec 但环境不可用时，说明失败并请求用户选择安装 / 初始化或改用轻量载体。
- OpenSpec 文档不能覆盖用户已确认的需求事实；发现冲突时先中性说明再询问。

### CodeGraph

- 根目录没有 `.codegraph/` 时跳过，不把“建立索引”当成普通影响分析的默认前置步骤。
- `.codegraph/` 存在时，在 grep / find 或逐文件通读之前优先使用 `codegraph_explore`；MCP 不可用时可用 `codegraph explore` CLI。
- CodeGraph 输出属于代码事实。关键结论仍要与当前源码、测试或实际命令结果一致，不能据此定义验收标准。
- 工具不可用或索引无法覆盖时，人工阅读相关源码与测试，并记录未覆盖范围；不得假装完成影响分析。

### Superpowers

- skill 的触发规则只决定采用什么方法，不授予安装依赖、编辑额外文件、commit、push、merge 或发布权限。
- skill 不能自行决定启用 OpenSpec；需要 OpenSpec 时仍须满足显式授权门禁。
- 澄清、设计和计划必须回填唯一需求载体；默认文档路径与项目单一事实来源冲突时，以项目规则为准。
- TDD 是推荐实现方法；未安装 Superpowers 时按项目技术栈完成等价验证，`AGENTS.md` 的完成前验证底线不变。

项目自建 skill 放在 `.agents/skills/<skill-name>/SKILL.md`，并在 `.agents/README.md` 建索引；Superpowers 是安装在 AI 工具侧的方法库，不作为项目自建 skill 复制进仓库。

## 7. 工具不可用时的 fallback

| 工具 | Fallback |
| --- | --- |
| OpenSpec | 用当前任务、issue、PR、任务卡或轻量记录承载需求与验收，至少写清范围、验证方式、未验证项与风险 |
| CodeGraph | 人工阅读相关源码、调用方和测试，使用项目已有搜索工具，并记录未覆盖范围 |
| Superpowers | 手动完成澄清、拆任务、测试优先节奏、调试与完成前证据核对 |

用户未选择某项工具时，工具缺失不应阻塞任务，应采用等价流程继续并如实说明能力差异。用户已明确选择工具但当前不可用时，暂停相关流程，请用户选择安装 / 初始化或改用 fallback，不得自行替换载体。
