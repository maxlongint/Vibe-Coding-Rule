# 代码设计原则

本文档定义项目通用的代码设计原则，适用于前端、后端、全栈、脚本和自动化任务。它用于指导实现取舍，不替代具体技术栈、框架或业务规则文档。与更具体的项目文档、需求记录或 `AGENTS.md` 底线冲突时，优先遵循更具体或更高优先级的约束。

## 总原则

代码设计优先服务于业务正确性、清晰边界和可维护性。抽象、复用和拆分都应基于真实复杂度，而不是基于假想需求或形式上的整洁。

实施时遵循以下取舍顺序：

1. 先保证行为正确、错误显式、契约清楚。
2. 在满足正确性的前提下，选择简单直接的实现。
3. 当重复逻辑代表同一知识、同一业务规则或稳定复用能力时，再抽象为公共模块。
4. 不为了未来可能用到而提前引入配置层、扩展点、基类、框架或复杂目录结构。

## KISS：保持简单

优先选择容易理解、容易验证、容易维护的方案。除非复杂性来自真实业务需求，否则不要引入额外的抽象层、间接调用、配置入口或通用框架。

适用要求：

- 能用清晰函数表达的，不引入复杂类层级。
- 能用现有项目模式解决的，不新建平行架构。
- 能局部修改完成的，不扩大到无关模块重构。
- 复杂实现必须能说明它解决了什么真实问题。

## YAGNI：不做未被需要的设计

不要为了未来可能出现的需求提前设计。扩展点、配置项、兼容层、通用模块和插件化能力，只有在需求真实存在或已有稳定复用场景时才引入。

适用要求：

- 不为一次性需求建立通用框架。
- 不为假想的多租户、多语言、多渠道、多端形态提前改造。
- 不预留没有调用方的参数、分支、接口或配置。
- 如果未来需求出现时再改成本更低，应优先保持当前实现简单。

## DRY：不要重复知识

DRY 关注的是知识和规则的重复，不是代码长得像就必须合并。相同业务规则、权限判断、校验逻辑、数据转换和协议适配应集中维护，避免多处散落导致口径不一致。

适用要求：

- 同一业务规则出现两处及以上时，应优先评估是否沉淀为公共函数、服务、组件或模块；只有语义一致、变化原因一致且抽象后更清楚时才抽象。
- 相同校验和错误语义应共享，不得在不同入口各写一套。
- 语义不同但结构相似的代码可以保留独立实现，避免强行合并造成隐式耦合。
- 抽象后如果调用方需要大量参数、布尔开关或特殊分支，说明抽象边界需要重新判断。

## 关注点分离

不同职责应保持边界清楚。展示、状态、业务规则、数据访问、外部集成、权限校验、错误处理等关注点不应无序混在同一个函数、组件或模块中。

适用要求：

- 前端避免把复杂业务规则、请求适配和 UI 细节全部堆在组件中。
- 后端避免把参数解析、权限判断、业务规则、数据访问和外部调用全部堆在路由或控制器中。
- 脚本和自动化任务也应区分输入解析、核心处理、输出呈现和错误处理。
- 跨层调用应通过清晰接口完成，不直接访问其他层的内部细节。

## 高内聚、低耦合

模块内部应围绕同一职责组织，模块之间通过明确接口协作。相关逻辑放在一起，不相关模块不要互相知道过多实现细节。

适用要求：

- 一个模块应有清楚的责任边界和稳定的对外入口。
- 修改一个模块内部实现时，不应迫使无关调用方同步修改。
- 公共模块不应依赖具体页面、具体接口、具体业务场景的临时细节。
- 当文件持续膨胀、职责混杂或修改时总牵连无关逻辑，应考虑拆分边界。

## 最小惊讶原则

代码的命名、接口、默认行为、错误语义和副作用应符合项目既有习惯和调用者直觉。不要隐藏重要行为，也不要让名称和实际效果相互误导。

适用要求：

- 命名应表达真实职责，不使用过度宽泛或误导性的名称。
- 函数和组件不应在名称之外暗含重要副作用。
- 默认值只能表达业务上明确的默认行为，不得掩盖缺失字段、接口失败或权限失败。
- 新增接口和模块应尽量沿用项目已有调用方式、错误风格和目录习惯。

## 抽象与复用判断清单

满足以下条件时，优先考虑抽象或沉淀公共能力：

- 同一业务规则、校验逻辑、权限判断或数据转换已经重复出现。
- 多个调用方的语义一致，未来需要保持同一口径演进。
- 抽象后能减少维护成本，并让调用方更清楚。
- 抽象边界稳定，不依赖某个临时页面、临时接口或一次性流程。
- 可以用清晰名称和少量参数表达能力，不需要大量开关控制行为。

出现以下情况时，不应急于抽象：

- 只是代码形状相似，但业务语义不同。
- 只有一个调用方，且未来复用场景不明确。
- 抽象后参数更多、分支更多、调用更难懂。
- 抽象需要改变无关模块、接口契约或既有行为。
- 抽象只是为了“看起来更通用”，没有降低真实复杂度。

## 前端适用补充

前端实现应优先保持组件职责清晰、状态来源明确、交互行为可预测。

- 能由已有数据推导的状态，不重复存储。
- 复杂业务逻辑优先沉淀到项目既有的 hooks、services、utils、model 或等价层。
- 可复用组件应抽象稳定的交互和展示模式，不强行合并语义不同的 UI。
- 加载、空状态、错误状态、权限状态应有一致处理方式。

## 后端适用补充

后端实现应优先保持领域规则集中、接口契约稳定、错误处理显式。

- 路由、控制器或项目既有等价入口负责协议接入，不承载大量核心业务规则。
- 核心业务规则应集中在 service、use case、domain 或项目既有等价层中。
- 数据访问、外部服务、消息队列、第三方 SDK 等依赖应有清晰边界。
- 关键写操作应关注幂等性、事务边界和失败反馈。
