# 流程公共约定（OpenSpec · Superpowers · CodeGraph）

> 强制底线见 [`AGENTS.md`](./AGENTS.md)。本文只说明三种调用方式与占位符约定，供 [`新需求开发.md`](./新需求开发.md) 与 [`需求变更.md`](./需求变更.md) 共用。

## 三种调用方式

文中出现的命令分三类，调用位置和写法都不一样：

| 类型 | 写法 | 在哪里用 | 例子 |
| --- | --- | --- | --- |
| **AI 技能**（Superpowers） | `$技能名` | AI 对话框里直接打 | `$brainstorming`、`$test-driven-development` |
| **AI 斜杠命令**（OpenSpec） | `/opsx:命令 参数` | AI 对话框里直接打 | `/opsx:propose add-sso-login` |
| **终端 CLI**（OpenSpec） | `openspec 子命令` | 项目根目录的终端里 | `openspec validate add-sso-login` |

## 占位符与命令可用性

- 尖括号 `<change-id>` 是**占位符**，要替换成真实值。用小写加短横线，能一眼看懂做什么，例如 `add-sso-login`、`update-password-policy`。目录名与占位符一致，即 `openspec/changes/<change-id>/`。
- **基础工作流**（`/opsx:propose`、`/opsx:apply`、`/opsx:sync`、`/opsx:archive`）：`openspec init` 之后就能用。
- **扩展工作流**（`/opsx:explore`、`/opsx:new`、`/opsx:continue`、`/opsx:ff`、`/opsx:verify`）：需要先配 profile 再跑一次 `openspec update` 才有；没有就退回基础命令，或直接让 AI 读写 `openspec/changes/<change-id>/` 下的文档。

## Superpowers 与单一事实来源

- Superpowers 只贡献**方法**（澄清、拆任务、验证纪律等），不另产平行文档。
- `$brainstorming` 的澄清结论回填到 OpenSpec change 的 `proposal` / `specs`；`$writing-plans` 的拆解结果落进 `tasks.md`。
- 具体约束见 `AGENTS.md` 文末「可选增强」；不必在每次提示词里重复叮嘱。

## TDD 的定位

- **强制底线**（`AGENTS.md`）：完成前必须验证，并说明结果与剩余风险。
- **推荐方法**（启用 Superpowers 时）：`$test-driven-development` 可作为实现阶段的推荐节奏——先写失败测试、再写实现、再核对证据；未启用 Superpowers 时，按项目惯例验证即可，底线不打折。

## 项目 skill 与第三方 skill

| 类型 | 路径 | 说明 |
| --- | --- | --- |
| 项目自建 skill | `.agents/skills/<skill-name>/` | 入口 `SKILL.md`，索引见 `.agents/README.md` |
| 第三方 skill（skills CLI） | `.agents/skills/`（项目级安装） | `npx skills add <pkg>`，安装后更新 `.agents/README.md` |
| Superpowers | 用户目录（非本项目路径） | 执行方法库，按官方说明安装；产出回填需求载体，见 `AGENTS.md` |

## 未安装可选工具时

| 工具 | Fallback |
| --- | --- |
| OpenSpec | 用 issue、任务卡、PR 描述或轻量记录承载需求与验收；至少写清变更范围、验证方式、剩余风险 |
| CodeGraph | 人工通读相关源码与测试，并记录未覆盖范围 |
| Superpowers | 手动完成澄清、按任务推进、完成前自行核对验证证据 |

`AGENTS.md` 六条底线在上述任何组合下均不打折。
