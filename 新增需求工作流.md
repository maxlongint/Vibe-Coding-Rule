# 新增需求工作流

> **范围**：仅本规范仓工作流教程，**不**随 Vibe Coding Rule 接入复制到业务项目。使用时在本仓库查阅或打开对应页面。默认使用 OpenSpec 主流程。强制底线见 [`AGENTS.md`](./AGENTS.md)，工具默认假定与协作规则见 [`docs/规范/工具协作.md`](./docs/规范/工具协作.md)。各步骤可以在同一个或不同的 AI Chat 中完成；操作已有 change 时，须在提示词中提供明确的 change-id。

本流程只在用户明确确认要把内容作为正式新增需求处理后使用。AI 识别到潜在新需求时必须先说明依据和影响并请求需求意图确认；确认前不得检查 OpenSpec change 或自动进入本流程。用户确认需求意图后先澄清目标、范围和验收口径；用户主动要求创建前，不得创建或更新 change，也不得形成最终设计、计划或开始实现。

开始前按默认假定检查 OpenSpec 是否可用（本流程以 OpenSpec change 为唯一载体，故先检 OpenSpec）。若发现未安装、未初始化或无法调用，立即中断并明确告知是 OpenSpec，说明已完成项与阻塞点；不自动安装，也不改用 issue/PR/任务卡或其他轻量载体继续。后续步骤若需要 CodeGraph 或 Superpowers 却不可用，同样中断并指明该工具；不因「流程开头未预检」而视为可缺省。

新需求不在澄清前扫描或选择现有 change。需求口径确认且用户主动要求创建后，AI 根据已确认口径生成 `change-id`，再只读检查现有 change；发现同名 change，或发现已有 change 可能承载该需求时，立即停止并列出冲突项、状态和关系，不得改用其他 ID 或创建 change。没有冲突时直接创建。

新需求只走五个步骤：

```text
澄清需求 → 创建 change 并起草 proposal → 实现 → 验证 → 归档
```

创建 change 时，由 AI 根据已确认需求生成简短、明确的小写英文 `change-id`，使用短横线分隔，例如 `add-dark-mode`；无需用户单独确认该 ID。创建前必须完成冲突检查；创建成功后，将 AI 返回的最终 ID 填入后续提示词中的 `<change-id>` 占位符，确保切换 AI Chat 后仍能定位同一个 change。

**带 UI 设计图时**采用条件式两阶段实现：五步主流程不变，在第 2 步规划时拆好任务，第 3 步分两轮执行（先 UI + mock，**人工验收 UI** 后再接真实逻辑）。

## 1. 澄清需求

先让 AI 使用 Superpowers 的 `brainstorming` 方法判断当前是既有项目还是空项目，再澄清业务需求与关键技术决策。只有目标、范围、验收标准和关键技术决策都已明确，才能进入第 2 步。这一步是需求口径尚未确认时的临时澄清，不创建 change，也不实现；不得在此阶段形成最终设计或实施计划。

在 AI Chat 中发送：

```text
我想实现：<需求描述>。

请使用 Superpowers 的 `brainstorming` 方法澄清新增需求。本步只澄清，不要创建或更新 OpenSpec change、proposal、specs、design、tasks，不要形成最终设计或实施计划。

先判断当前工作目录：既有项目则核实并默认沿用现有技术基线（语言、框架、数据等），仅确认本需求的变更项；空项目则提出相应的 MVP 技术基线选项及取舍。

不要自动创建 proposal。
```

与 AI 对话，直到你确认需求已经清楚；确认后由你主动进入下一步，将结论写入 OpenSpec change。创建成功前不切换 Chat，不开始设计、计划或实现。若需求已在 issue、PR 或项目文档中写清，可跳过本步，直接进入第 2 步并注明来源。

## 2. 创建 change 并起草 proposal

让 AI 根据已确认的需求生成 `change-id` 并检查冲突；没有冲突时直接创建 OpenSpec change，形成后续实现与验收的唯一依据。

在 AI Chat 中发送：

```text
需求口径已确认，请根据以下来源创建 OpenSpec change。

需求来源：<上轮对话结论 / 文档路径 / issue·PR 链接 / 其他说明>

如果需求来源或本消息包含 UI 设计图，本步仅按已确认的文字需求口径创建初始任务，不读取或解释设计图，也不加入设计衍生任务；列出暂未读取的 UI 输入。若文字需求与设计图无法可靠分离，请停止并说明。先形成原始 tasks，设计图留到后续 UI 补充步骤处理。

请读取并对照需求来源，根据已确认的口径生成简短、明确、使用小写英文和短横线分隔的 change-id，并只读检查现有 change。

如果存在同名 change，或发现已有 change 可能承载本需求，请停止并列出冲突项、状态和关系；不要改用其他 ID，也不要创建 change。

确认无冲突后，使用生成的 change-id 创建 change，并起草相互一致的 proposal、specs、design 和 tasks。

成功后返回最终 change-id 和创建内容；先不要实现。
```

**有 UI 设计图时**，再发送一条补充提示词：

```text
本需求有 UI 设计图，请补充更新本需求 change 的 design 和 tasks。

Change ID：<change-id>
UI 设计图来源：<实际本地路径 / Figma·蓝湖等链接 / 附件说明及作用 / 沿用当前 change 已记录的来源集合>

以上字段必须替换为实际且可读取的来源，不得保留占位符。先不要读取 UI 来源；先读取该 change 的 proposal、specs、design 和当前 tasks，完成设计源、基线、快照与旧结构修复门禁；未通过则停止。

门禁通过后，读取 UI 来源并更新 design 和 tasks，形成两个阶段、任务映射、未决问题及核对记录。完成后返回核对结果，先不要实现。

```

检查 AI 返回的 `change-id`、冲突检查结果、目标、范围、验收标准和任务拆分是否准确，并确认 AI 已读取你填写的需求来源。有 UI 设计图时，再确认 AI 已读取设计图位置，且 `tasks.md` 已按两阶段拆分。确认无误后，再进入实现。

## 3. 实现

让 AI 按 change 中的任务清单完成开发，并同步任务状态。

**无 UI 设计图**，在 AI Chat 中发送：

```text
请实现本需求的 OpenSpec change。

Change ID：<change-id>

请先读取并对照该 change 的 proposal、specs、design 和 tasks，再按 tasks 逐项实现。

完成后更新任务状态，并说明实际改动、已经执行的检查、尚未验证的内容和剩余风险。不要归档。
```

**有 UI 设计图**——分两轮执行；阶段 1 由 AI 实现，**UI 由你人工验收**，通过后再发阶段 2。

第一轮，在 AI Chat 中发送：

```text
请只实现本需求的 OpenSpec change 中「阶段 1：UI、隔离 mock 与页面跳转」的任务，不要开始阶段 2。

Change ID：<change-id>

请先读取该 change 的 proposal、specs、design 和 tasks，完成设计源、基线、映射、checkbox 和职责核对。

核对通过后，逐项实现阶段 1 的全部未完成任务，不得删除、合并或改写阶段 2 及其他未完成任务。

遵守 UI 两阶段实现的阶段边界和 mock 隔离要求。

完成后仅更新实际完成的阶段 1 任务状态，列出完成项、未完成项及仍待阶段 2 处理的全部任务，并说明如何本地预览。不要归档或开始阶段 2。
```

你在本地对照设计图验收 UI 与页面跳转。不通过则让 AI 继续修阶段 1，通过后再回复 **「UI 验收通过」**，发送第二轮：

```text
UI 验收通过。请实现本需求的 OpenSpec change 中「阶段 2：真实接口、数据、权限、状态流转与错误处理」的任务。

Change ID：<change-id>

请先读取该 change 的 proposal、specs、design 和 tasks，完成设计源、基线、映射、checkbox 和职责核对。

核对通过后，逐项实现阶段 2 的全部未完成实现任务。

完成后根据“任务重组映射”检查整个 tasks，不得遗留未实现或未说明的需求任务。计划在下一步执行的验证类任务可保持“待验证”，但须完整列出，且此时不得宣称整个 change 已完成或可归档。更新实际完成的任务状态，说明完整任务对照结果、实际改动、已执行的检查、待验证项和未覆盖风险；存在实现遗漏时不得宣称实现完成。不要归档。
```

检查实际改动是否符合 proposal 和 specs。所有任务完成后，进入验证。

## 4. 验证

让 AI 核对实现是否满足 change，并提供真实的测试、构建和检查结果。验证未通过时，先修复并重新验证，不要进入归档。

在 AI Chat 中发送：

```text
请验证本需求的 OpenSpec change。

Change ID：<change-id>

请读取并逐项对照该 change 的 proposal、specs、design 和 tasks。

按顺序验证：
1. 若存在 UI 重组标记，完成设计源、基线、映射、checkbox 和职责核对。
2. 建立“需求与验收标准 → tasks 任务编号 → 实现与验证证据”的逐项对应，确认全部实现任务均已处理。
3. 执行并更新标为“待验证”的任务，运行与本次改动及其直接影响相关的测试、构建或检查。

列出实际执行的检查及其结果、遗漏项、未覆盖项和剩余风险；若存在适用检查不一致、任务遗漏、仍无结论的待验证项或检查失败，不要宣称完成或归档。
```

只有验证通过、未覆盖项和剩余风险都已确认，才能进入归档。

## 5. 归档

验收完成后，让 AI 同步规格并归档 change，为本次新需求收尾。

在 AI Chat 中发送：

```text
请归档本需求的 OpenSpec change。

Change ID：<change-id>

请按项目规范确认验证结果有效，同步需要更新的规格并归档。成功后告诉我归档位置和规格更新结果。
```

确认 AI 返回了真实的归档位置和规格更新结果，新需求流程结束。
