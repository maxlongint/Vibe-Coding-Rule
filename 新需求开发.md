# 新需求开发

> 这是一份面向开发人员的新需求操作指引。仅在用户已明确选择 OpenSpec 后使用以下主流程；未选择时见文末「未启用 OpenSpec 时」。强制底线见 [`AGENTS.md`](./AGENTS.md)，工具门禁见 [`docs/规范/可选工具协作.md`](./docs/规范/可选工具协作.md)。各步骤可以在同一个或不同的 AI Chat 中完成；操作已有 change 时，须在提示词中提供明确的 change-id。

开始前确认项目已经完成 OpenSpec 初始化，且 change 目录结构可用。具体如何读取、创建、实现、验证和归档 change，由 AI 根据当前环境与项目约定处理，提示词只需说明目标、输入、边界和期望结果。若 OpenSpec 整体未安装或未初始化，按工具协作指南暂停，请用户选择安装 / 初始化或改用轻量载体。

新需求只走五个步骤：

```text
澄清需求 → 创建 change 并起草 proposal → 实现 → 验证 → 归档
```

创建 change 时，由 AI 根据需求内容生成简短、明确的小写英文 `change-id`，使用短横线分隔，并检查是否与现有 change 重名，例如 `add-dark-mode`。AI 创建完成后应返回最终使用的 ID；执行后续步骤时，将该 ID 填入提示词中的 `<change-id>` 占位符，确保切换 AI Chat 后仍能定位同一个 change。

**带 UI 设计图时**采用条件式两阶段实现：五步主流程不变，在第 2 步规划时拆好任务，第 3 步分两轮执行（先 UI + mock，**人工验收 UI** 后再接真实逻辑）。

## 1. 澄清需求

先让 AI 了解项目现状，帮你把模糊想法整理成明确的目标、范围和验收标准。这一步只探索，不创建 change，也不实现。

在 AI Chat 中发送：

```text
我想实现：<需求描述>。

请先了解与本需求直接相关的项目现状，再帮我澄清目标、范围、非目标、验收标准和待决问题。

这一步只做需求探索，先不要创建 OpenSpec change，也不要实现。
```

与 AI 对话，直到你确认需求已经清楚，再进入下一步。若需求已在 issue、PR 或项目文档中写清，可跳过本步，直接进入第 2 步并注明来源。

## 2. 创建 change 并起草 proposal

让 AI 根据已确认的需求（来自上轮澄清、外部文档或任务记录）写入 OpenSpec change，形成后续实现与验收的唯一依据。

在 AI Chat 中发送：

```text
请根据以下需求来源创建 OpenSpec change。

需求来源：<上轮对话结论 / 文档路径 / issue·PR 链接 / 其他说明>

如果需求来源或本消息同时包含 UI 设计图，本步只根据已确认的文字需求口径创建初始任务，不读取或解释设计图，也不加入设计衍生任务；请列出暂未读取的 UI 输入。若文字需求与设计图无法可靠分离，请停止并说明。先形成原始 tasks，设计图在后面的 UI 补充步骤处理。

请先读取并对照需求来源，根据需求内容自动生成简短、明确的小写英文 change-id，使用短横线分隔，并确认不与现有 change 重名。然后创建 change，起草 proposal、specs、design 和 tasks，使目标、范围、非目标、验收标准、方案和任务保持一致。

如果需求来源不可读、OpenSpec 未初始化或创建失败，请明确说明原因，不要伪装完成。成功后返回最终使用的 change-id，并说明创建了哪些内容；先不要实现。
```

**有 UI 设计图时**，再发送一条补充提示词：

```text
本需求有 UI 设计图，请补充并更新本需求 change 的 design 与 tasks.md。

Change ID：<change-id>

UI 设计图来源：<实际本地路径 / Figma·蓝湖等链接 / 附件说明及作用 / 沿用 design.md 已记录的来源集合>
上面的字段必须替换为明确且可读取的权威来源集合，或明确写明沿用 design.md 已记录的集合；允许多个互补来源，但要写清各自作用和冲突时的优先级。仍是占位符、为空、不可读取或存在未定优先级的冲突来源时，请停止并说明，不要自行选择仓库目录或旧设计稿，也不要修改 design 和 tasks.md。

先不要读取上面的 UI 来源。请先读取该 change 的 proposal、specs、design 和当前 tasks：首次重组时，立即把当前原任务固化为非 checkbox“任务重组基线”并核对数量；已有有效基线和映射时保留原基线，改为新增非 checkbox“本轮 UI 更新前任务快照”，记录当前最终任务的编号、摘要和状态；已有重组标记但基线或映射不完整时先停止并修复。确认基线或快照已落盘后，才读取 UI 来源并按项目 `docs/规范/可选工具协作.md` 中的 OpenSpec 任务重组规则处理：

1. 在 design 中补充设计图来源、页面清单和 mock 隔离方式。
2. 以现有 tasks 为基线，将任务编排为阶段 1「UI + 隔离 mock + 页面跳转」和阶段 2「真实接口、数据、权限、状态流转和错误处理」，并分别写清验收标准。
3. 不得遗漏、合并或弱化原任务及验收内容；跨阶段任务拆为对应的 S1 / S2 checkbox 子任务。每个 checkbox 必须有唯一交付物；对重叠部分，能力/组件任务负责具体行为、mock 和错误反馈，页面任务负责组装、路由、页面联调与视觉验收，并写明依赖。没有其他任务承接的页面独有交互仍归页面任务，不能为套用模板而凭空拆任务。
4. 保留已落盘的“任务重组基线”或本轮更新前快照，并将“任务重组映射”、设计图新增任务和未解决问题写入 tasks.md；这些追溯区只能使用非 checkbox 表格或文本，实际任务只在所属阶段保留 checkbox。
5. 整个 tasks.md 只能在阶段 1、阶段 2 出现 checkbox，每个最终任务编号在全文件恰好对应一条 checkbox 行；按全文件核对数量、编号、映射、checkbox 完整性和语义重叠，发现不一致或职责边界不清时先修正，不要宣称无遗漏。

完成后返回核对结果。先不要实现。

```

检查目标、范围、验收标准和任务拆分是否准确；确认 AI 已读取你填写的需求来源。有 UI 设计图时，再确认 AI 已读取设计图位置，且 `tasks.md` 已按两阶段拆分。确认无误后，再进入实现。

## 3. 实现

让 AI 按 change 中的任务清单完成开发，并同步任务状态。

**无 UI 设计图**，在 AI Chat 中发送：

```text
请实现本需求的 OpenSpec change。

Change ID：<change-id>

请先读取并对照该 change 的 proposal、specs、design 和 tasks，再按 tasks 逐项实现，不要改变未要求调整的行为、接口、数据、权限和流程。

完成后更新任务状态，并说明实际改动、已经执行的检查、尚未验证的内容和剩余风险。不要归档。
```

**有 UI 设计图**——分两轮执行；阶段 1 由 AI 实现，**UI 由你人工验收**，通过后再发阶段 2。

第一轮，在 AI Chat 中发送：

```text
请只实现本需求的 OpenSpec change 中「阶段 1：UI + 隔离 mock + 页面跳转」的任务，不要开始阶段 2。

Change ID：<change-id>

请先根据 change-id 重新读取并逐项对照该 change 的 proposal、specs、design、tasks，以及 tasks 中的“任务重组基线”和“任务重组映射”；以当前落盘内容为准，不依赖之前的对话或摘要。

执行前先检查：design 已记录明确且可读取的 UI 权威来源集合，多来源的作用及发生内容冲突时的优先级清楚；checkbox 只能存在于阶段 1、阶段 2；每个最终任务编号恰好对应一条 checkbox 行；相似任务已对重叠部分明确“能力/组件实现 → 页面组装/集成验收”的职责与依赖，没有重复交付物，页面独有交互也有明确责任任务。不符合时按详细规则修复，不得把非阶段区的结构性副本重新计入基线；设计源、基线、映射或职责边界缺失或冲突且无法从已确认需求载体重建时，停止并说明，不要开始实现。

非 checkbox 父项和映射行只用于追溯。逐项实现阶段 1 的全部 checkbox，包括未拆分原任务、`-S1` 子任务和明确列入新增任务的 checkbox；不得删除、合并或改写阶段 2 及其他未完成任务。

禁止接入真实接口、真实数据、权限判断或业务状态流转；mock 须与真实链路隔离（如独立目录、环境开关或占位 service），边界写进 design 或代码。

完成后只更新实际完成的阶段 1 任务状态，列出完成项、未完成项和仍待阶段 2 处理的全部任务，并说明如何本地预览。不要归档，不要开始阶段 2。
```

你在本地对照设计图验收 UI 与页面跳转。不通过则让 AI 继续修阶段 1，通过后再回复 **「UI 验收通过」**，发送第二轮：

```text
UI 验收通过。请实现本需求的 OpenSpec change 中「阶段 2：真实接口、数据、权限、状态流转和错误处理」的任务。

Change ID：<change-id>

请先根据 change-id 重新读取并逐项对照该 change 的 proposal、specs、design、tasks，以及 tasks 中的“任务重组基线”和“任务重组映射”；以当前落盘内容为准，不依赖之前的对话或摘要。

执行前先检查：design 已记录明确且可读取的 UI 权威来源集合，多来源的作用及发生内容冲突时的优先级清楚；checkbox 只能存在于阶段 1、阶段 2；每个最终任务编号恰好对应一条 checkbox 行；相似任务已对重叠部分明确“能力/组件实现 → 页面组装/集成验收”的职责与依赖，没有重复交付物，页面独有交互也有明确责任任务。不符合时按详细规则修复，不得把非阶段区的结构性副本重新计入基线；设计源、基线、映射或职责边界缺失或冲突且无法从已确认需求载体重建时，停止并说明，不要开始实现。

非 checkbox 父项和映射行只用于追溯。逐项实现阶段 2 的全部未完成实现 checkbox，包括未拆分原任务、`-S2` 子任务和明确列入新增任务的 checkbox；不要只按阶段标题概括实现。

完成后根据“任务重组映射”检查整个 tasks.md，不得遗留未实现或未说明的需求任务。按计划应在下一步执行的验证类任务可以保持“待验证”，但必须完整列出，且此时不得宣称整个 change 已完成或可归档。更新实际完成的任务状态，并说明完整任务对照结果、实际改动、已经执行的检查、待验证项和尚未覆盖的风险；存在实现遗漏时不得宣称实现完成。不要归档。
```

检查实际改动是否符合 proposal 和 specs。所有任务完成后，进入验证。

## 4. 验证

让 AI 核对实现是否满足 change，并提供真实的测试、构建和检查结果。验证未通过时，先修复并重新验证，不要进入归档。

在 AI Chat 中发送：

```text
请验证本需求的 OpenSpec change。

Change ID：<change-id>

请根据 change-id 重新读取并逐项对照 proposal、specs、design、tasks，以及 tasks 中的“任务重组基线”和“任务重组映射”；以当前落盘内容为准，不依赖之前的对话或摘要。

按顺序验证：
1. 若 design 或 tasks 中出现任一 UI 重组标记（“UI 设计图来源”或旧标题“UI 设计来源”、规定的 UI 两阶段标题、任务重组基线、任务重组映射、设计图新增任务目录、`-S1` / `-S2` 编号），执行完整核对：比较基线与映射的数量和原编号集合；确认权威来源集合可读取，多来源的作用及发生冲突时的优先级明确，映射编号存在、阶段外 checkbox 为零、每个最终编号恰好对应一条 checkbox 行；确认重叠任务的职责与依赖清楚、没有重复交付物，页面独有交互有责任任务。上述标记全部不存在时才跳过本项；普通任务使用的泛化“阶段 1 / 阶段 2”标题不单独视为 UI 重组标记。
2. 建立“需求与验收标准 → tasks 任务编号 → 实现与验证证据”的逐项对应；采用两阶段重组时同时纳入基线和映射，确认两个阶段的实现任务均已处理；否则确认普通 tasks 中的实现任务均已处理。
3. 执行并更新标为“待验证”的任务，运行与本次改动及其直接影响相关的测试、构建或检查。

列出实际执行的检查、结果、遗漏项、未覆盖项和剩余风险；存在适用检查不一致、任务遗漏、仍无结论的待验证项或检查失败时，不要宣称完成，也不要归档。
```

只有验证通过、未覆盖项和剩余风险都已确认，才能进入归档。

## 5. 归档

验收完成后，让 AI 同步规格并归档 change，为本次新需求收尾。

在 AI Chat 中发送：

```text
请归档本需求的 OpenSpec change。

Change ID：<change-id>

归档前确认验证结果有效，并按项目约定同步需要更新的规格。若验证无效、同步失败或无法归档，请明确说明原因，不要伪装完成；成功后告诉我归档位置和规格更新结果。
```

确认 AI 返回了真实的归档位置和规格更新结果，新需求流程结束。

## 未启用 OpenSpec 时

把 change 换成对应的 issue、PR 或任务卡：同样先澄清需求、写清范围与验收，再改代码；有 UI 设计图时仍按两阶段拆分任务，阶段 1 人工验收 UI 后再做阶段 2。各工具 fallback 见 [`docs/规范/可选工具协作.md`](./docs/规范/可选工具协作.md)。`AGENTS.md` 底线不打折。
