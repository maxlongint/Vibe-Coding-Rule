# 新需求开发（OpenSpec 主 · Superpowers 辅）

> 强制底线见 [`AGENTS.md`](./AGENTS.md)。本文讲开发人员怎么操作、执行什么命令、产出什么。
>
> 分工：`AGENTS.md` 定底线 · OpenSpec 管需求/规格/任务/验收 · CodeGraph 管影响分析 · Superpowers 管执行纪律（澄清、TDD、验证）。
>
> **文档只有一处事实来源——OpenSpec 的 change 目录。** 三种命令写法、占位符、TDD 定位与工具 fallback 见 [`流程公共约定.md`](./流程公共约定.md)。

## 操作步骤

### 第 1 步　判断要不要建 change

判断标准就一句：**这次改动会不会改变"外部能观察到的行为"**——会，就建 change；不会，通常不必。"外部可观察"指接口、数据结构、权限、流程、状态流转、验收标准、用户对结果的预期这些别人能感知到的东西。

**要建 change 的场景：**

- 新增能力：接 SSO 单点登录、加一个导出报表接口。
- 改规则：密码最短长度从 6 位提到 12 位、订单状态新增"已退款"。
- 改契约：某 API 返回结构变了、分页默认条数从 20 改成 50。
- 改权限：某功能从"仅管理员可见"放开到"所有登录用户"。

**不必建 change 的场景：**

- 改错别字、调整提示文案（且不改变操作含义）。
- 内部变量/函数重命名、补注释、纯代码整理。
- 修一个不改变对外行为的 bug（比如修空指针，但输入输出和原设计一致）。

拿不准时，按"会改变外部行为"从严处理，先建 change。这一步只是判断，不执行任何命令。

### 第 2 步　澄清需求

需求已经很清楚可以跳过；只要目标、范围、验收标准有一处含糊，就先让 AI 帮你问清楚，别急着写代码。在对话框里触发澄清技能：

```text
$brainstorming
帮我澄清这个需求：目标是什么、范围到哪、哪些明确不做、验收标准怎么定。一次问一个问题，先不要写代码。
```

> 这里只把 `$brainstorming` 当澄清对话用——不要它写设计稿、提交，也不要转去 `$writing-plans`；澄清结论留到第 3 步写进 `proposal` / `specs`。（change 目录此时还没创建，所以结论先留在对话里，第 3 步再落地。）
>
> 不必在提示词里叮嘱"别写独立设计稿"——`AGENTS.md` 文末已约束 Superpowers 产出回填到主需求载体，自动生效。

**产出**：一份澄清结论——目标 / 非目标 / 验收标准 / 待决问题，作为下一步写 proposal 的输入（不落地成独立文档）。

### 第 3 步　Proposal——定「做什么」

让 AI 创建 change 并起草 proposal 和 specs，你来审核目标、范围、验收标准对不对。下面用 `<change-id>` 表示实际 change id，例如 `add-sso-login`。在对话框里发：

```text
/opsx:propose <change-id>
```

> 扩展工作流用 `/opsx:new <change-id>`；都不可用时直接说：「这个项目用 OpenSpec，请在 `openspec/changes/<change-id>/` 下创建 change，先写 proposal 和 specs，不要实现。」

**产出**：

- `openspec/changes/<change-id>/proposal.md`——背景、目标、非目标、用户影响、验收标准、风险。
- `openspec/changes/<change-id>/specs/<domain>/spec.md`——外部可观察行为和验收场景（写"系统应该表现成什么样"，不写实现细节）。
- `openspec/changes/<change-id>/tasks.md`——初始任务清单与验证方向。

### 第 4 步　创建隔离工作区——worktree + 新分支

新需求进入开发前，**默认让 AI 在独立 worktree 里工作**，并为这个 change 建独立分支。worktree 不是替代分支，而是"新目录 + 新分支"：主工作区保持不动，AI 的改动留在对应需求目录里。

命名保持一致：

```text
change id: <change-id>
branch:    codex/<change-id>
worktree:  .worktrees/<change-id>
```

在对话框里发：

```text
为 change `<change-id>` 准备隔离开发工作区。

请在当前项目中创建并进入 `.worktrees/<change-id>`，使用分支 `codex/<change-id>`。创建前必须确认 `.worktrees/` 已被 Git 忽略，并确认当前工作区没有会混入本需求的未提交无关改动；任一条件不满足就停止并报告原因。后续影响分析、实现、验证都在该 worktree 中完成。不要 commit、push、merge 或发布。

完成后告诉我 worktree 路径、分支名、忽略检查结果、当前工作区状态，以及你接下来会在哪个目录继续。
```

> 如果项目不允许新增本地 `.worktrees/` 目录，可以把 worktree 建到项目外部固定目录，但仍要保持"一个需求一个 worktree + 一个分支"。
>
> 纯只读分析不需要建分支或 worktree；错别字、极小文档修补等低风险小改，可以只建传统分支或直接在当前分支处理，但不能和未提交的无关改动混在一起。

**产出**：AI 返回 worktree 路径、分支名、忽略检查结果、当前工作区状态，以及后续操作会在哪个目录继续。

### 第 5 步　影响分析——先搞清楚「会动到哪」

定方案前先把范围摸清楚，**别跳过这步直接拍方案**。让 AI 在第 4 步创建的 worktree 中，用 CodeGraph 把调用链、依赖、受影响范围理清楚。

```text
在 `<change-id>` 对应的 worktree 中完成本需求影响分析，先不要实现。

请找出相关功能区域、受影响模块、关键数据流、调用链、回归测试范围、未覆盖项和剩余风险。优先使用 CodeGraph；如果 CodeGraph 不可用，就阅读相关源码与测试，并说明哪些范围没有覆盖到。请明确区分目标变更与非目标不变式，其他既有接口、数据结构、权限、错误处理、布局和兼容性默认保持不变。

如发现需要补充技术方案、数据流、安全策略或回滚方案，只提出回填 `design.md` / `tasks.md` 的建议，不要直接改代码。
```

**产出**：影响分析结论——受影响模块、依赖关系、回归范围、未覆盖项。这份结论是下一步写 design / tasks 的依据。

### 第 6 步　补方案、校准 tasks（按需）

`/opsx:propose` 已经会创建初始 `tasks.md`，所以这一步不是为了"再建任务"。只有当影响分析发现方案、数据流、风险或回归范围需要补充时，才让 AI 把结论回填到 `design.md` 和现有 `tasks.md`。下面两条**二选一**，区别只是节奏：

```text
/opsx:ff <change-id>        # 快进：一口气补方案并校准 tasks，适合需求清楚、想快点过
/opsx:continue <change-id>  # 分步：一次推进一个阶段、停下来给你审核，适合复杂需求逐步把关
```

> 扩展命令不可用：直接让 AI 根据影响分析更新 `design.md` 和 `tasks.md`。如需配合 `$writing-plans` 重新拆任务，结果也要落进 OpenSpec 的 `tasks.md`，不要另存独立计划文件。

**产出**：

- `design.md`（按需）——技术方案、数据流、安全策略、回滚方案。
- 更新后的 `tasks.md`——补齐受影响模块、回归范围、验证方式、未覆盖项。

### 第 6.5 步　UI / 交互设计确认（按需）

这一步不是所有需求都要做；只有当需求会改变用户可见界面、交互路径、视觉风格、布局、文案含义、信息层级、表单、状态反馈或其他用户体验时才需要。纯后端接口、内部逻辑、脚本、重构、配置或无界面影响的改动可以跳过，但要在 `tasks.md` 或回复里说明"无 UI / 交互影响"。

涉及 UI / 交互时，进入实现前必须先确认设计输入。设计依据优先写进 `design.md`；`tasks.md` 只同步需要执行的实现任务、验收点和验证项，不承载完整 UI 方案。

- 已有设计源：Figma、Stitch、截图、原型、线框图、现有页面参照、品牌/组件规范等，先把设计来源和约束写进 `design.md`，再把对应实现任务、验收点和验证项同步到 `tasks.md`。
- 没有设计源：不要让 AI 直接自由发挥；先让 AI 给出低保真 UI 方案、关键页面结构、交互状态、组件清单、文案含义和验收点，等你确认后再拆任务或实现。
- 只改局部体验：也要说明非目标不变式，例如相邻布局、既有组件尺寸、权限、数据结构、错误处理、兼容性等默认保持不变。

在对话框里按实际情况二选一发。

**已有设计源**（例如设计图、截图、原型、Figma 导出、Stitch 产物等）：

```text
这个 change 涉及用户界面 / 交互体验。进入实现前，先确认 UI 设计输入。

设计源路径：`<ui-source-path>`（例如 `assets/ui/`，请替换为实际路径）。

请先只读分析设计源，不要修改设计资产，也不要写代码。

请把设计来源、关键页面结构、状态、组件、视觉/交互约束回填到 `design.md`，并把对应实现任务、验收点和验证项同步到 `tasks.md`。

更新完先让我审核，审核通过前不要实现。不要把未经确认的视觉或交互自由发挥写进代码。
```

**暂无设计源**（需求需要 UI / 交互，但还没有设计图或原型）：

```text
这个 change 涉及用户界面 / 交互体验，但目前没有设计源。进入实现前，先确认 UI 设计输入。

请先给出低保真 UI 方案或关键界面说明，覆盖关键页面结构、状态、组件、文案含义、交互约束、目标变更与非目标不变式。

先不要写代码，也不要直接更新 `design.md` / `tasks.md`。等我确认 UI 方案后，再把确认后的设计依据回填到 `design.md`，并把对应实现任务、验收点和验证项同步到 `tasks.md`。

审核通过前不要实现。不要把未经确认的视觉或交互自由发挥写进代码。
```

**产出**：若适用，`design.md` 里的 UI / 交互设计依据、关键页面/状态/组件/文案/相邻行为不变式，以及 `tasks.md` 里的实现任务、验收点和验证项；若跳过，说明无 UI / 交互影响及跳过原因。

### 第 7 步　实现

让 AI 按 `tasks.md` 逐项推进，每完成一项就更新状态，你逐项审核。不在生产链路用 mock、默认值、空对象伪造成功。若第 6.5 步适用，必须先确认 UI / 交互设计输入，再进入实现。

```text
/opsx:apply <change-id>
```

启用 Superpowers 时，**推荐**用 TDD 技能作为实现节奏（非 `AGENTS.md` 强制底线；底线仍是完成前验证并说明结果）：

```text
$test-driven-development
请按 tasks.md 实现 `<change-id>`，每个任务先写失败测试，跑红，再写实现跑绿，最后我审核。
```

未启用 Superpowers 时，按项目惯例实现，并满足 `AGENTS.md` 验证要求。

**产出**：代码 + `tasks.md` 里每项的完成状态、验证结果、剩余风险。

### 第 8 步　验证

跑校验和测试并确认真实结果，不把"没验证"说成"已通过"；验不了要写清原因和剩余风险。

```text
openspec validate <change-id>   # 终端：校验 change 文档结构是否完整合法
/opsx:verify <change-id>        # 扩展工作流；不可用就省略这条
```

再加上项目自己的测试/构建/lint，例如 `npm test`、`pytest`、`make test`，按项目实际命令来。最后用验证技能核对证据：

```text
$verification-before-completion
帮我核对 `<change-id>` 的验证证据：实际跑了哪些命令、结果如何、还有什么没覆盖。
```

**产出**：验证证据——实际执行的命令、结果、未覆盖项、剩余风险。

### 第 9 步　集成回主工作区

验证通过后，**不要把 worktree 里的文件手工复制回主目录**。worktree 不是临时副本，它对应的是需求分支；正确做法是把需求分支通过本地 merge 或 PR 合回主分支。合并、推送、创建 PR、删除分支或删除 worktree 都属于提交/发布相关动作，必须由你明确选择后再执行。

在对话框里发：

```text
`<change-id>` 已在 worktree 中完成实现和验证。请准备把需求分支集成回主工作区。

请先复核当前分支、worktree 路径、验证证据和待提交改动；如果还有未提交改动，先列出改动摘要和建议提交信息，等我确认后再提交。不要自动 commit、push、merge、删除分支或删除 worktree。

确认验证证据有效后，请给我这 4 个选项让我选择：

1. 本地合并回主分支，并在主工作区重新验证
2. 推送需求分支并创建 PR
3. 保留当前分支和 worktree，稍后我自己处理
4. 放弃本次工作

我选择后，你再按对应流程执行；如果选择本地合并，合并后必须在主工作区重新运行相关验证，再清理 worktree 和已合并分支。放弃工作前必须再次让我明确确认。
```

> 主分支名称按项目实际情况确认，通常是 `main`、`master` 或 `develop`。本地合并完成后，主目录才能看到新需求代码；PR 流程则由远端平台合并，主目录需要再拉取最新主分支。

**产出**：集成方式、目标主分支、提交/PR 信息（如有）、主工作区验证结果、worktree/分支清理状态、剩余风险。

### 第 10 步　归档

只有在第 9 步已经集成完成，并且主工作区或已合并 PR 的目标分支验收通过后，才把 delta spec 合并进主 spec 并归档。若选择保留分支，change 仍是进行中或暂停；若放弃本次工作，按废弃状态处理，不归档成已完成。

```text
/opsx:sync <change-id>      # 把 change 里的 spec 改动合并进主 spec（按需）
/opsx:archive <change-id>   # 归档；终端等价命令：openspec archive <change-id>
```

**产出**：已归档的 change + 状态标记（`进行中` / `已完成` / `已暂停` / `已废弃` / `已替代`）。

## 产出内容总览

```text
openspec/changes/<change-id>/
  proposal.md            # 第 3 步：背景、目标、非目标、用户影响、验收标准、风险
  specs/<domain>/spec.md # 第 3 步：外部可观察行为与验收场景（不写实现细节）
  tasks.md               # 第 3 步创建；第 6/6.5 步按影响分析与 UI/交互输入校准
  design.md              # 第 6/6.5 步按需补充：技术方案、数据流、安全策略、回滚、UI/交互设计输入
```

这些文档都由 AI 起草、你审核确认，必要时手动微调——`openspec` CLI 只管结构校验和归档，不生成正文。

## 未启用 OpenSpec 时

把 change 换成 issue、任务卡、PR 描述或一份轻量记录，承载同样的产出：需求与验收、影响分析、任务与验证。各工具 fallback 见 [`流程公共约定.md`](./流程公共约定.md)。`AGENTS.md` 底线不打折。
