# 新需求开发

> 这是一份面向开发人员的新需求操作指引。强制底线见 [`AGENTS.md`](./AGENTS.md)；所有操作都在 AI Chat 中完成。

新需求只走五个步骤：

```text
澄清需求 → 创建 change 并起草 proposal → 实现 → 验证 → 归档
```

文中的 `<change-id>` 是本次需求的唯一标识，请替换成简短、明确的小写英文名称，并用短横线分隔，例如 `add-dark-mode`。

**带 UI 设计图时**采用条件式两阶段实现：五步主流程不变，在第 2 步规划时拆好任务，第 3 步分两轮执行（先 UI + mock，**人工验收 UI** 后再接真实逻辑）。

## 1. 澄清需求

先让 AI 了解项目现状，帮你把模糊想法整理成明确的目标、范围和验收标准。这一步只探索，不创建 change，也不实现。

在 AI Chat 中发送：

```text
我想实现：<需求描述>。

请完成 OpenSpec 的需求探索（对应 `/opsx:explore`）。如果当前环境支持这个命令，可以使用；不可用时直接检查项目现状并继续，不要因此停止。

请帮我澄清需求，确认目标、范围、非目标、验收标准和待决问题。先不要创建 change，也不要实现。
```

与 AI 对话，直到你确认需求已经清楚，再进入下一步。若需求已在 issue、PR 或项目文档中写清，可跳过本步，直接进入第 2 步并注明来源。

## 2. 创建 change 并起草 proposal

让 AI 根据已确认的需求（来自上轮澄清、外部文档或任务记录）写入 OpenSpec change，形成后续实现与验收的唯一依据。

在 AI Chat 中发送：

```text
请根据以下需求来源创建 OpenSpec change：`<change-id>`。

需求来源：<上轮对话结论 / 文档路径 / issue·PR 链接 / 其他说明>

这项操作对应 `/opsx:propose <change-id>`。如果当前环境支持这个命令，可以使用；不可用时直接在 `openspec/changes/<change-id>/` 下创建 change。请先读取并对照需求来源，再起草 proposal、specs、design 和 tasks，不要因此停止。

完成后说明创建了哪些内容，先不要实现。
```

**有 UI 设计图时**，再发送一条补充提示词：

```text
本需求有 UI 设计图，请补充并更新 change `<change-id>` 的 design 与 tasks.md。

UI 设计图位置：<本地路径 / Figma·蓝湖等链接 / 附件说明>
请读取并对照设计图，在 design 中记录设计图来源、页面清单与 mock 隔离方式；在 tasks.md 中拆成两阶段，每阶段各自写清验收标准：
- 阶段 1：UI + 隔离 mock + 页面跳转
- 阶段 2：真实接口、数据、权限、状态流转和错误处理

先不要实现。
```

检查目标、范围、验收标准和任务拆分是否准确；确认 AI 已读取你填写的需求来源。有 UI 设计图时，再确认 AI 已读取设计图位置，且 `tasks.md` 已按两阶段拆分。确认无误后，再进入实现。

## 3. 实现

让 AI 按 change 中的任务清单完成开发，并同步任务状态。

**无 UI 设计图**，在 AI Chat 中发送：

```text
请实现 OpenSpec change：`<change-id>`。

这项操作对应 `/opsx:apply <change-id>`。如果当前环境支持这个命令，可以使用；不可用时直接读取该 change 的 proposal、specs、design 和 tasks，并按 tasks 逐项实现，不要因此停止。

完成后更新任务状态，并说明实际改动、已经执行的检查、尚未验证的内容和剩余风险。不要归档。
```

**有 UI 设计图**——分两轮执行；阶段 1 由 AI 实现，**UI 由你人工验收**，通过后再发阶段 2。

第一轮，在 AI Chat 中发送：

```text
请只实现 OpenSpec change `<change-id>` 的「阶段 1：UI + 隔离 mock + 页面跳转」，不要开始阶段 2。

禁止接入真实接口、真实数据、权限判断或业务状态流转；mock 须与真实链路隔离（如独立目录、环境开关或占位 service），边界写进 design 或代码。

完成后更新任务状态，并说明如何本地预览。不要归档，不要开始阶段 2。
```

你在本地对照设计图验收 UI 与页面跳转。不通过则让 AI 继续修阶段 1，通过后再回复 **「UI 验收通过」**，发送第二轮：

```text
UI 验收通过。请实现 OpenSpec change `<change-id>` 的「阶段 2：真实接口、数据、权限、状态流转和错误处理」。

完成后更新任务状态，并说明实际改动、已经执行的检查、尚未验证的内容和剩余风险。不要归档。
```

检查实际改动是否符合 proposal 和 specs。所有任务完成后，进入验证。

## 4. 验证

让 AI 核对实现是否满足 change，并提供真实的测试、构建和检查结果。验证未通过时，先修复并重新验证，不要进入归档。

在 AI Chat 中发送：

```text
请验证 OpenSpec change：`<change-id>`。

这项操作对应 `/opsx:verify <change-id>`。如果当前环境支持这个命令，可以使用；不可用时直接对照 proposal、specs 和 tasks 完成同等验证，不要跳过。

请运行与本次改动相关的测试、构建或检查，列出实际执行的命令、结果、未覆盖项和剩余风险；存在失败时不要宣称完成，也不要归档。
```

只有验证通过、未覆盖项和剩余风险都已确认，才能进入归档。

## 5. 归档

验收完成后，让 AI 同步规格并归档 change，为本次新需求收尾。

在 AI Chat 中发送：

```text
请归档 OpenSpec change：`<change-id>`。

这项操作对应 `/opsx:archive <change-id>`。如果当前环境支持这个命令，可以使用；不可用时直接按当前项目的 OpenSpec 约定同步规格并归档，不要因此停止。

归档前必须确认验证结果有效。完成后告诉我归档位置和规格更新结果。
```

确认 AI 返回了真实的归档位置和规格更新结果，新需求流程结束。

## 未启用 OpenSpec 时

把 change 换成对应的 issue、PR 或任务卡：同样先澄清需求、写清范围与验收，再改代码；有 UI 设计图时仍按两阶段拆分任务，阶段 1 人工验收 UI 后再做阶段 2。各工具 fallback 见 [`流程公共约定.md`](./流程公共约定.md)。`AGENTS.md` 底线不打折。
