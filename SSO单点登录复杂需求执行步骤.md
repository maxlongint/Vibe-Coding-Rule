# SSO 单点登录复杂需求执行步骤

本文档是一个复杂需求执行示例，用来说明：在大型既有项目中新增 SSO 单点登录能力时，团队和 AI 应如何按 `AGENTS.md` 完成需求记录、影响分析、方案设计、实现、验证、评审、发布与归档。

SSO 属于高风险、跨模块真实业务需求。它通常会改变登录入口、认证协议、用户绑定、权限映射、Session 或 token 逻辑、前端路由守卫、配置管理、审计日志和验收标准。因此，在正式修改代码前，必须先建立可追溯的需求记录。

推荐主流程：

```text
判断文档性质和需求等级
  -> 确认需求记录载体
  -> 创建 docs/需求/<change-name>/ 需求记录
  -> 影响分析
  -> 方案确认
  -> 实现计划
  -> 分阶段实现
  -> 验证
  -> 评审
  -> 发布、回滚和归档
```

## 适用前提

本示例默认项目采用仓库内需求文档治理。如果项目已经明确使用外部需求系统、issue 系统或任务系统，应在外部记录、任务记录、PR 描述、提交说明或交付说明中保留需求来源、验收标准、变化记录、验证结果和剩余风险。

如果项目没有启用 OpenSpec，也不需要为了本需求强行引入 OpenSpec。只要在 `docs/需求/<change-name>/` 中保持 `proposal.md`、`design.md`、`tasks.md`、`decisions.md`、`changelog.md` 和 `specs/` 的职责清晰，就符合 `AGENTS.md` 的需求治理底线。

如果项目已经使用 OpenSpec、ADR、RFC 或同类方法，可以把这些方法承载在 `docs/需求/<change-name>/` 内，而不是绕过项目规则另建不可追溯的位置。

## 第一步：判断文档性质和需求等级

### 目标

先判断当前工作属于哪类文档和哪类变更，避免把真实业务需求写成普通说明文档，也避免在没有记录的情况下直接编码。

### 判断结果示例

```text
文档性质：真实业务需求示例
需求类型：复杂真实业务需求
治理方式：仓库内需求文档治理
建议目录：docs/需求/add-sso-login/
变化等级：重大变化
必须先记录：是
必须评审：是
必须验证：是
建议发布方式：灰度发布，并保留回滚路径
```

### 为什么是重大变化

SSO 会影响认证协议、登录策略、权限模型和用户身份绑定。如果处理不当，可能造成：

- 原账号密码登录不可用。
- 用户错绑导致权限串号。
- Session 或 token 异常失效。
- `redirect_uri`、`state`、`nonce` 校验不严导致安全风险。
- 第三方身份 token 校验不完整。
- 上线后无法快速切回原登录路径。

### 检查点

这一步结束时，应确认：

- 已判断 SSO 是真实业务需求，不是普通技术文档。
- 已确认采用仓库内、外部系统或轻量记录中的哪一种治理载体。
- 已确认本示例采用 `docs/需求/add-sso-login/`。
- 已确认需求变化等级是重大变化。
- 已暂停直接实现，先进入需求记录。

## 第二步：创建需求记录目录

### 目标

在编码前创建独立、可追溯的需求记录，说明为什么做、怎么做、怎么验收、发生过哪些变化、还有哪些风险。

### 推荐目录

```text
docs/需求/add-sso-login/
  proposal.md
  design.md
  tasks.md
  decisions.md
  changelog.md
  specs/
    auth/
      spec.md
```

### 文件职责

```text
proposal.md   说明为什么做、目标是什么、边界是什么
design.md     说明怎么设计、影响范围、风险和方案决策
tasks.md      拆分实现任务、验证方式和完成状态
decisions.md  记录重要决策及原因
changelog.md  记录需求变化历史、状态、归档原因
spec.md       记录 auth 领域的行为规格和验收场景
```

### 给 Codex 的示例指令

```text
请按 AGENTS.md 为 SSO 单点登录创建仓库内需求记录。

要求：
1. 需求目录使用 docs/需求/add-sso-login/。
2. 创建 proposal.md、design.md、tasks.md、decisions.md、changelog.md。
3. 在 specs/auth/spec.md 中记录认证领域行为规格。
4. 记录需求来源为“用户确认新增 SSO 单点登录能力”。
5. 状态标记为“进行中”。
6. 明确第一阶段保留原账号密码登录。
7. 暂时不要写实现代码。
```

### proposal.md 应包含

- 背景：为什么接入 SSO。
- 目标：新增 SSO 登录能力。
- 非目标：第一阶段不替换原账号密码登录。
- 用户影响：用户如何进入 SSO 登录流程。
- 范围边界：本次是否包含自动建号、权限同步、部门同步。
- 关键验收标准：成功登录、失败处理、原登录回归、灰度开关。
- 风险：账号绑定、安全校验、权限映射、回滚。

### design.md 应包含

- 身份源：OIDC、SAML、企业微信、钉钉、Azure AD 或自建 IdP。
- 登录流程：发起登录、第三方跳转、callback、登录态生成。
- 用户绑定规则：邮箱、手机号、工号、外部身份 ID 或首次登录创建。
- 权限映射：SSO 用户如何映射系统角色。
- Session 机制：是否沿用现有 token。
- 安全要求：`state`、`nonce`、`redirect_uri` 白名单、token 签名和有效期校验。
- 配置开关：如何启用、禁用和灰度。
- 错误处理：哪些失败向用户展示，哪些进入审计日志。
- 回滚方案：SSO 异常时如何切回原登录。
- 影响分析：由 codegraph、人工代码索引或团队确认结果补充。

### tasks.md 应包含

```text
1. 调研现有认证流程。
2. 补充影响分析。
3. 确认第一阶段方案。
4. 增加 SSO 配置结构。
5. 增加 SSO 登录发起接口。
6. 增加 SSO callback 接口。
7. 增加第三方身份 token 校验。
8. 增加用户绑定或自动创建逻辑。
9. 接入现有 session/token 生成流程。
10. 前端登录页增加 SSO 入口。
11. 前端处理 callback 成功和失败状态。
12. 增加后端测试。
13. 增加前端测试。
14. 增加发布、灰度和回滚说明。
15. 完成评审和归档记录。
```

每个任务都应记录目标、修改范围、验证方式、完成状态和剩余风险。

### specs/auth/spec.md 示例

```text
Requirement: SSO 登录入口

Scenario: 用户从登录页发起 SSO 登录
- GIVEN 系统已启用 SSO
- WHEN 用户点击 SSO 登录按钮
- THEN 系统生成第三方身份源登录地址
- AND redirect_uri、state、nonce 均符合安全要求

Scenario: SSO 未启用时隐藏入口
- GIVEN 系统未启用 SSO
- WHEN 用户打开登录页
- THEN 登录页不展示 SSO 登录入口
- AND 原账号密码登录仍然可用

Scenario: callback 校验失败时拒绝登录
- GIVEN 用户从第三方身份源返回 callback
- WHEN state、nonce 或 token 校验失败
- THEN 系统拒绝创建登录态
- AND 用户看到安全的失败提示
- AND 审计日志记录失败原因
```

### changelog.md 示例

```text
# Changelog

## 2026-06-02

- 状态：进行中
- 来源：用户确认新增 SSO 单点登录能力
- 变化等级：重大变化
- 记录载体：docs/需求/add-sso-login/
- 说明：创建需求记录，正式实现前先完成影响分析、方案确认和验收标准。
```

### 检查点

这一步结束时，应确认：

- 需求目录独立存在。
- 需求来源、状态、边界、验收标准和剩余风险可追溯。
- 重要决策有 `decisions.md`。
- 变化历史有 `changelog.md`。
- 尚未开始实现代码。

## 第三步：做影响分析

### 目标

在方案设计和实现计划前识别受影响模块、调用链、测试范围和安全风险。项目配置了 codegraph、代码索引工具、依赖图工具或同类能力时，应优先使用；没有工具时，也必须记录人工影响判断。

### 给 Codex 的示例指令

```text
请按 AGENTS.md 对 docs/需求/add-sso-login/ 做影响分析。

重点查找：
1. 登录入口。
2. 认证服务。
3. 用户服务。
4. 权限和角色服务。
5. Session 或 token 逻辑。
6. 前端路由守卫。
7. 登录页和 callback 页。
8. 配置加载逻辑。
9. 审计日志。
10. 认证相关测试。

请输出受影响模块、关键调用链、需要修改的文件、需要回归的测试和主要风险，并把结论写入 design.md 和 tasks.md。
```

### 使用 codegraph 的示例目标

具体命令以团队选定工具为准。示例只表达操作目标：

```powershell
codegraph index .
codegraph query "auth login session token user role permission"
codegraph impact "login"
codegraph impact "createSession"
codegraph impact "authCallback"
```

如果没有 codegraph，应在 `design.md` 中写明：

```text
影响分析方式：人工代码检索
限制：未配置 codegraph 或同类依赖图工具
剩余风险：可能遗漏动态路由、运行时注入或间接调用链
```

### 影响清单示例

```text
可能需要修改：
- backend/auth/*
- backend/user/*
- backend/session/*
- frontend/pages/login/*
- frontend/router/guard/*
- config/auth.*
- tests/auth/*
```

### 风险清单示例

```text
- 原账号密码登录不能被破坏。
- 已登录用户 session 不能异常失效。
- 用户绑定错误可能造成权限串号。
- redirect_uri 配置错误可能造成安全风险。
- SSO 用户权限映射错误可能导致越权。
```

### 回写需求记录

影响分析不能只停留在聊天记录中，应回写到：

- `design.md`：影响范围、调用链、风险和缓解措施。
- `tasks.md`：需要修改的模块、测试任务和验证命令。
- `specs/auth/spec.md`：新增或调整验收场景。
- `changelog.md`：记录本次影响分析更新。

### 检查点

这一步结束时，应确认：

- 已识别前后端受影响模块。
- 已识别需要回归的测试。
- 已识别安全风险。
- 影响分析已写入需求记录。
- 未使用工具分析时，已写明人工判断限制和剩余风险。

## 第四步：确认方案和决策

### 目标

比较多个方案，选定第一阶段低风险方案，并把重要取舍写入 `decisions.md`。

### 给 Codex 的示例指令

```text
请基于 docs/需求/add-sso-login/ 和影响分析，比较 3 种 SSO 接入方案。

要求：
1. 每种方案说明实现方式、优点、缺点、风险和适用场景。
2. 给出第一阶段推荐方案。
3. 推荐方案必须适合大型既有项目，优先降低上线风险。
4. 把最终方案写入 design.md。
5. 把关键取舍写入 decisions.md。
6. 根据方案更新 tasks.md 和 specs/auth/spec.md。
7. 暂时不要写实现代码。
```

### 方案比较示例

方案 A：保留原登录，并新增 SSO 入口。

- 优点：风险低，便于灰度，原登录可作为兜底。
- 缺点：登录页和认证逻辑会同时存在两种入口。
- 适用：大型既有项目第一阶段。
- 推荐：是。

方案 B：默认跳转 SSO，原登录作为备用。

- 优点：企业内部用户体验更统一。
- 缺点：一旦 SSO 配置错误，登录入口风险更大。
- 适用：SSO 稳定后第二阶段。
- 推荐：暂不作为第一阶段。

方案 C：完全替换原登录。

- 优点：登录体系更统一。
- 缺点：上线风险最大，回滚困难。
- 适用：后续成熟阶段。
- 推荐：不建议第一阶段采用。

### decisions.md 示例

```text
# Decisions

## 2026-06-02：第一阶段保留原账号密码登录

决策：第一阶段采用“保留原登录 + 新增 SSO 登录入口 + 配置开关灰度”。

原因：
- 现有登录仍是生产兜底路径。
- SSO 涉及外部身份源，首次上线需要降低不可控风险。
- 配置开关可以支持灰度和快速回滚。

影响：
- 登录页需要同时支持两种入口。
- 后端认证模块需要区分原登录和 SSO callback。
- 验证必须覆盖原登录回归和 SSO 失败场景。
```

### 检查点

这一步结束时，应确认：

- 已比较 2 到 3 个方案。
- 已明确推荐方案。
- 已说明为什么不采用其他方案。
- `design.md`、`decisions.md`、`tasks.md` 和 `spec.md` 已同步更新。

## 第五步：编写实现计划

### 目标

把方案拆成可执行、可评审、可验证的小任务。计划必须包含测试和验收方式，不能只列开发动作。

### 给 Codex 的示例指令

```text
请基于 docs/需求/add-sso-login/ 编写 SSO 实现计划。

要求：
1. 每一步都有明确目标。
2. 每一步说明修改范围。
3. 每一步说明要新增或更新的测试。
4. 每一步说明验证命令。
5. 每一步说明风险和完成标准。
6. 不要一次性大改。
7. 计划写入 docs/需求/add-sso-login/tasks.md。
```

### 每个任务应包含

```text
任务名称：
目标：
输入：
修改范围：
测试：
验证命令：
风险：
完成标准：
状态：
```

### 推荐拆分

```text
1. 调研现有认证流程。
2. 增加 SSO 配置结构。
3. 增加 SSO 登录发起接口。
4. 增加 SSO callback 接口。
5. 增加第三方身份 token 校验。
6. 增加用户绑定或自动创建逻辑。
7. 接入现有 session/token 生成流程。
8. 前端登录页增加 SSO 入口。
9. 前端处理 callback 和错误页。
10. 增加后端测试。
11. 增加前端测试。
12. 增加发布、灰度和回滚说明。
```

### 检查点

这一步结束时，应确认：

- 计划可以逐项执行。
- 每个任务有验证方式。
- 没有把多个高风险变更塞进同一步。
- 需求记录中的验收标准和任务保持一致。

## 第六步：分阶段实现

### 目标

按计划逐步实现，并在每个高风险认证逻辑变更前补充测试。任何会改变行为、接口、权限、数据或验收标准的新变化，都要先更新需求记录，再继续实现。

### 给 Codex 的示例指令

```text
请实现 docs/需求/add-sso-login/ 中 tasks.md 的第 N 个任务。

要求：
1. 先阅读需求记录和相关代码。
2. 如果任务涉及认证、权限、状态流转或外部集成，先写或更新测试。
3. 先运行测试，确认失败原因符合目标行为。
4. 再修改实现代码。
5. 再运行相关验证。
6. 更新 tasks.md 中该任务状态和验证结果。
7. 不要修改无关文件。
```

### 实现原则

- 优先复用现有认证、用户、权限和 session 逻辑。
- 不引入新的认证体系，除非需求记录已经明确要求并说明影响。
- 不删除原账号密码登录入口。
- 不用 mock、本地缓存、旧字段或默认值伪造真实登录成功。
- 接口失败、权限失败、字段缺失、非法状态和外部资源不可访问必须显式失败。
- 用户可见关键写操作必须防止重复提交，并展示等待状态。
- 写入成功后跳转必须等待写入结果，不能提前隐藏失败状态。

### 测试重点

后端测试：

- 未开启 SSO 时，原登录正常。
- 开启 SSO 后，可以生成登录跳转地址。
- callback 校验 `state` 成功。
- callback 校验 `state` 失败时拒绝登录。
- 第三方身份 token 无效时拒绝登录。
- 已绑定用户可以登录。
- 禁用用户不能登录。
- 权限映射正确。

前端测试：

- 登录页展示或隐藏 SSO 入口符合配置。
- 点击 SSO 入口后跳转正确。
- callback 成功后进入系统。
- callback 失败后展示错误。
- 原账号密码登录仍可用。

### 检查点

每个实现任务结束时，应确认：

- 测试已新增或更新。
- 目标测试已运行。
- 原登录相关测试未被破坏。
- 变更范围符合计划。
- `tasks.md` 中对应任务状态、验证结果和剩余风险已更新。

## 第七步：处理需求变化

### 目标

在实现过程中遇到范围、规则、验收标准变化时，先判断变化等级并更新记录，再继续实现。

### 示例变化

小变化：登录按钮文案从“SSO 登录”改为“企业登录”。

- 更新 `tasks.md`。
- 更新验收标准。
- 重新运行相关前端测试或截图检查。

中等变化：用户绑定规则从邮箱匹配改为工号匹配。

- 更新 `design.md`。
- 更新 `specs/auth/spec.md`。
- 更新 `tasks.md`。
- 更新 `decisions.md` 说明取舍。
- 重新做必要影响分析。
- 重新验证绑定相关测试。

重大变化：从 OIDC 改为 SAML，或从保留原登录改为默认跳转 SSO。

- 暂停实现。
- 更新 `proposal.md`、`design.md` 和 `specs/auth/spec.md`。
- 更新 `changelog.md`。
- 重新做影响分析。
- 重新走方案评审。
- 重新拆实现计划。
- 必要时判断是否新建需求记录。

### 检查点

这一步结束时，应确认：

- 变化归属已确认。
- 记录载体已确认。
- 变化等级已判断。
- 文档、计划、测试和验收标准已同步更新。
- 重新确认后才继续实现。

## 第八步：验证

### 目标

用证据确认功能、回归、安全和构建状态。完成前不能只说“看起来可以”，必须说明实际运行了什么、结果是什么、没有覆盖什么。

### 给 Codex 的示例指令

```text
请根据 docs/需求/add-sso-login/ 的验收标准和 tasks.md，列出必须运行的验证命令，并逐项执行。

要求：
1. 如项目存在 .node-version，先通过 fnm 使用对应 Node 版本。
2. 运行后端单元测试。
3. 运行认证集成测试。
4. 运行前端测试。
5. 运行构建检查。
6. 验证原账号密码登录不受影响。
7. 验证 SSO 成功和失败场景。
8. 汇报实际运行的命令、结果、未覆盖原因和剩余风险。
```

### 验证清单

- 需求记录结构检查。
- 认证单元测试。
- 认证集成测试。
- 前端登录流程测试。
- 原账号密码登录回归。
- SSO 失败场景。
- 权限回归。
- 构建检查。
- 安全配置检查。
- 灰度开关检查。
- 真实 IdP 或测试 IdP 联调。

### 命令示例

具体命令以项目实际技术栈为准。Windows PowerShell 环境下，如果存在 `.node-version`，先使用对应 Node 版本：

```powershell
$nodeVersion = Get-Content -Path ".node-version"
fnm use $nodeVersion
npm run test
npm run build
```

如果项目没有 `.node-version`，使用当前默认 Node 环境：

```powershell
npm run test
npm run build
```

### 汇报格式

```text
已运行：
- npm run test：通过
- npm run build：通过
- 认证集成测试：通过

未运行：
- 真实企业 IdP 联调，因为测试环境未提供 IdP 配置。

剩余风险：
- 需要在测试环境完成真实 IdP callback 联调。
```

### 检查点

这一步结束时，应确认：

- 验证命令已实际运行。
- 结果明确，不把未验证项写成已通过。
- 未验证项有原因。
- 剩余风险已记录到任务记录、评审材料或交付说明中。

## 第九步：评审

### 目标

让评审者确认实现是否符合需求记录，是否安全，是否影响既有登录和权限。

### 给 Codex 的示例指令

```text
请基于当前 SSO 变更准备评审材料。

要求：
1. 汇总 docs/需求/add-sso-login/ 的目标和验收标准。
2. 汇总代码变更文件。
3. 汇总影响分析结果。
4. 汇总已运行验证命令。
5. 标出安全敏感点。
6. 标出剩余风险和需要人工确认的问题。
7. 不执行 commit、push、merge、deploy 或 release。
```

### 评审材料应包含

```text
需求来源：
需求目录：docs/需求/add-sso-login/
当前状态：
推荐方案：
影响范围：
主要变更文件：
验证结果：
安全关注点：
回滚方式：
剩余风险：
需要人工确认：
```

### 评审重点

- 是否影响原账号密码登录。
- `state`、`nonce`、`redirect_uri` 是否安全。
- 用户绑定规则是否可能错绑。
- 权限映射是否清晰。
- 错误处理是否可观测。
- 是否有可执行回滚开关。
- 验收项是否全部覆盖。
- 影响分析中识别的模块是否都处理或解释了。

### 检查点

这一步结束时，应确认：

- 评审材料完整。
- 需求记录、影响分析和测试结果都进入评审上下文。
- 安全敏感点被明确标出。
- 评审问题已记录并处理。

## 第十步：提交、发布与回滚

### 目标

降低上线风险，确保 SSO 异常时可以快速恢复原登录路径，并遵守提交与发布权限。

### 权限规则

未经用户明确要求，AI 不得执行：

- `git commit`
- `git push`
- `git merge`
- `release`
- `deploy`
- 任何会改变远端仓库、线上环境或发布状态的操作

如果用户要求提交或发布，AI 必须先汇总：

- 当前变更。
- 关联需求记录。
- 验证结果。
- 剩余风险。
- 未完成验证原因。

### 发布说明示例

```text
发布目标：
- 灰度开放 SSO 登录入口。
- 保留原账号密码登录。

配置项：
- SSO_ENABLED
- SSO_PROVIDER
- SSO_REDIRECT_URI
- SSO_ALLOWED_REDIRECT_HOSTS

灰度步骤：
1. 测试环境接入。
2. 内部管理员账号试点。
3. 部分部门灰度。
4. 全量开放。

回滚方式：
1. 关闭 SSO_ENABLED。
2. 保留原账号密码登录。
3. callback 接口拒绝新的 SSO 登录请求。
4. 保留已存在用户绑定数据，避免反复迁移。
5. 保留审计日志，便于排查已发生登录事件。
```

### 发布前检查

- 需求验收项已覆盖。
- 所有必要验证已完成。
- 原登录入口可用。
- SSO 配置可关闭。
- `redirect_uri` 白名单正确。
- 生产 IdP 配置已人工确认。
- 日志和监控已准备。
- 回滚责任人和操作路径已确认。

### 发布后检查

- SSO 登录成功率。
- SSO callback 错误率。
- 原账号密码登录成功率。
- 用户绑定失败数量。
- 权限加载失败数量。
- 登录相关告警。

### 检查点

这一步结束时，应确认：

- 发布步骤清晰。
- 回滚步骤可执行。
- 配置开关可用。
- 监控指标明确。
- 线上验证责任人明确。
- 未经授权没有执行提交或发布动作。

## 第十一步：完成和归档

### 目标

需求完成、暂停、废弃或被替代时，记录最终状态、日期、原因、验收结果和剩余风险。

### changelog.md 归档示例

```text
## 2026-06-20

- 状态：已完成
- 原因：SSO 第一阶段已完成并通过验收
- 最终验收：
  - 原账号密码登录可用
  - SSO 登录成功和失败场景已覆盖
  - 灰度开关和回滚路径已验证
- 剩余风险：
  - 后续如切换为默认 SSO，需要新一轮需求变化评审
```

如果需求暂停、废弃或被替代，也应写明停止原因、剩余风险和后续处理方向。是否移动到归档目录由项目约定决定；如果没有归档目录，保持原目录不动，通过状态标记归档。

### 检查点

这一步结束时，应确认：

- 需求状态已更新。
- 日期和原因已记录。
- 最终验收结果已记录。
- 未完成事项和剩余风险已记录。
- 归档后的需求不再直接追加实现任务；如需扩展，先判断是恢复原需求还是新建需求。

## 最终完成标准

SSO 需求只有在以下条件都满足后，才能认为可以进入合并、发布或归档：

- 已确认需求记录载体。
- `docs/需求/add-sso-login/` 已创建或等效外部记录已建立。
- 需求来源、状态、目标、边界、验收标准、变化记录和剩余风险可追溯。
- 影响分析已完成，并写入需求记录。
- 方案已评审并记录到 `design.md` 和 `decisions.md`。
- 实现计划已拆分成小任务，并记录验证方式。
- 每个实现任务都有测试或验证说明。
- 原账号密码登录仍然可用。
- SSO 成功和失败场景都已覆盖。
- 安全敏感点已评审。
- 发布和回滚方案已准备。
- Codex 汇报了实际运行的验证命令、结果、未验证原因和剩余风险。
- 未经用户明确授权，没有执行提交、推送、合并、发布或部署。
- 完成、暂停、废弃或替代时，`changelog.md` 已记录最终状态、日期和原因。

核心原则：先按 `AGENTS.md` 判断文档性质和需求治理方式，再记录需求；先做影响分析和方案确认，再拆计划；先更新需求变化记录，再改实现；最后用实际验证、评审、灰度和回滚保证复杂认证需求可控。
